## Context

当前 kflow-init LEGACY 模式通过 L1/L2/L3 三层代码逆向扫描生成产品级文档基线，但产出与变更级文档（kflow-explore 的 functional-designs/、kflow-design 的 detailed-design.md）存在系统性脱节：模块按代码目录划分而非功能菜单、内容密度低、技术设计维度缺失。`fix-init-product-docs-alignment`（2026-05-15）已对齐章节结构，但内容填充和划分逻辑的根本问题尚未解决。

## Goals / Non-Goals

**Goals:**
- 前后端项目：通过 L2.5 前端工程扫描，按真实功能菜单划分模块，使 init 产出的功能设计文档与变更级文档在结构和内容维度上对齐
- 纯后端项目：使用独立的简化模板，按设计域划分，去除不适用的 UX 密集章节
- 产品级 technical-designs 从 4 文件扩展为 6 文件，完整覆盖 detailed-design.md 的全部内容维度
- archive MERGE 逻辑适配新的产品级文档组织方式

**Non-Goals:**
- 不改变变更级文档体系（kflow-explore / kflow-design 的产出格式不变）
- 不改变 kflow-init 的主体流程（DETECT→SCAN→PROFILE→...→COMPLETE 13 步结构不变）
- 不追求 init LEGACY 产出的内容质量达到 explore 级别（草稿仍为草稿，仅提升结构和覆盖度）

## Decisions

### D1: L2.5 前端扫描层定位

**决策**: L2.5 作为 L2 和 L3 之间的独立扫描层，仅对前后端项目执行。提取前端路由配置、菜单/导航组件、页面组件中的表单字段和操作按钮。

**理由**: 后端代码的目录结构（controllers/services/models）与前端功能菜单没有对应关系。只有前端代码中的路由和菜单配置才能提供真实的功能导航结构。

**扫描目标**:
| 目标 | 来源示例 | 提取内容 |
|------|---------|---------|
| 路由配置 | `src/router/index.ts`, `src/routes.tsx`, `pages/` 目录 | 路径→页面映射、嵌套路由层级 |
| 菜单配置 | `src/config/menu.ts`, `<Menu>`, `<Sidebar>` 组件 | 一级菜单、二级菜单、权限控制 |
| 页面组件 | `src/pages/`, `src/views/` 下的 `.tsx/.vue` | 页面标题、表单字段(name/label/rules)、操作按钮 |
| 国际化文件 | `src/locales/`, `src/i18n/` | 中文标签→补充页面名/菜单名 |

**替代方案**: 
- 方案 B（放弃前端扫描，用 AI 推断菜单）→ 放弃。AI 推断不可靠，且违背"从代码中提取事实"的原则
- 方案 C（仅扫描路由，不扫描组件）→ 放弃。路由只能提供页面结构，无法提取表单字段和操作，内容密度提升有限

### D2: 产品级 functional-designs 目录结构

**决策**: 前后端项目按一级菜单创建目录，每个目录内使用 `index.md` + `part-NN.md`（条目>30时拆分），与变更级保持一致。

```
docs/designs/functional-designs/
├── {一级菜单1}/
│   ├── index.md
│   ├── part-01.md
│   └── part-02.md          ← 条目>30时出现
├── {一级菜单2}/
│   ├── index.md
│   └── part-01.md
└── ...（纯后端则为 {domain}.md 平铺文件）
```

**理由**: 
- 用一级菜单而非二级菜单（用户选择），避免目录碎片化
- 二级菜单作为 part-NN.md 内的功能点分组维度
- 目录名用一级菜单的 kebab-case，确保文件系统友好
- index.md + part-NN.md 结构与变更级对齐，archive MERGE 时无需结构转换

### D3: 纯后端简化模板

**决策**: 纯后端项目使用独立模板 `backend-domain.md`，章节结构为变更级模板的裁剪版：

| 变更级章节 | 纯后端处理 |
|-----------|-----------|
| 功能点ID/名称/优先级 | ✅ 保留 |
| 用户故事 | ✅ 保留（API 消费者视角） |
| 所属页面与菜单 | → 替换为「所属设计域」 |
| 可执行操作 | ✅ 保留（API 操作/服务调用） |
| 表单项定义 | → 替换为「接口参数定义」 |
| 业务规则 | ✅ 保留 |
| 业务流程上下文 | ✅ 保留（服务调用链视角） |
| 功能行为矩阵 | ✅ 保留 |
| 交互约束 | → 替换为「调用约束」（限流/权限/超时） |

**理由**: 纯后端项目无 UI 层，页面/菜单/表单章节无意义。但保留功能点体系（基于 API 端点划分），确保归档时与变更级 FP-ID 体系兼容。

### D4: technical-designs 6 文件体系

**决策**: 新增 `config-items.md` 和 `error-handling.md`，与 detailed-design.md 的六大部分形成完整映射：

| detailed-design.md | 产品级文件 | 说明 |
|---|---|---|
| §一 系统架构 | architecture.md | 已有，模板微调增加配置项/错误处理索引 |
| §二 设计域章节 | data-model.md + api-catalog.md | 已有 |
| §三 NFR | nfr-baseline.md | 已有 |
| §四 子变更划分 | — | 变更级临时产物，不入产品级 |
| §五 配置项设计 | **config-items.md** [新增] | 环境变量、配置项汇总 |
| §六 错误处理设计 | **error-handling.md** [新增] | 错误码、处理策略汇总 |

**理由**: 
- 子变更划分是实施阶段的临时产物，变更结束后失去意义
- 配置项和错误处理是产品持续运行的知识资产，应沉淀到产品级

### D5: init LEGACY 内容填充策略

**决策**: 按项目类型和章节的"可推断性"采用差异化填充：

| 章节 | 前后端项目 | 纯后端项目 |
|------|-----------|-----------|
| 功能点列表 | 路由+页面→推断 FP | API 端点→推断 FP |
| 用户故事 | 页面 title + 注释→推断 | 标注"待人工补充" |
| 页面/菜单 | 路由+菜单配置→**直接提取** | 替换为设计域 |
| 可执行操作 | 按钮/form→推断 | API method→推断 |
| 表单项 | 页面组件→**直接提取** | 替换为接口参数 |
| 业务规则 | 前端校验→提取 | 后端校验注解→提取 |
| 业务流程 | 路由跳转链→推断 | API 调用链→推断 |
| 行为矩阵 | 标注"待人工验证" | 标注"待人工验证" |

**关键原则**: 能提取的尽量提取，提取不了标注"待人工补充/验证"，不编造。

## Risks / Trade-offs

- [前端项目无标准路由框架] → L2.5 扫描降级：检测不到标准路由时回退到 L2 目录扫描，并提示用户手动提供功能菜单结构
- [前端代码未包含菜单配置（如纯静态页面）] → 从页面目录结构推断层级，标注"由目录结构推断，待确认"
- [已有产品文档迁移] → 首次重新执行 init 时检测旧格式 `{module}.md` 平铺文件，提示用户是否迁移到新目录结构
- [模板数量增加] → 从当前约 40 个模板增至约 44 个（+4 新增，-1 废弃），维护负担略增，但消除了两套模板体系的认知负担
