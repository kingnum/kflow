# Design: phase-review-mechanism-upgrade

## Context

当前 KFlow Skills 体系有三个设计阶段（explore/prototype/design），均采用"一次产出 + 一次性审查"模式。explore 有"两动作规则"保护上下文，design 有四视角交叉审查，但三者均缺少系统化的自我完善机制。此外，各阶段输出产物的边界定义不够精确，存在越界风险。

本次设计基于 2026-05-14 explore session 中与用户达成的全部决策。

## Goals / Non-Goals

**Goals:**
- 为 explore/prototype/design 三个阶段建立 10 轮自循环审查机制
- 明确每个阶段的 IN/OUT 任务边界
- 建立文档创建白名单约束
- 将 functional-designs/ 内容从"功能点清单"升维到"用户体验规格说明书"
- 调整审查目录结构，区分自审与交叉审查

**Non-Goals:**
- 不改变 plan/code/code-review/e2e-test/bug-fix/integration-test/archive 阶段的边界（这些已足够清晰）
- 不改变四视角审查的核心逻辑（仅调整存储结构）
- 不改变子变更划分逻辑
- 不引入新的外部依赖

## Decisions

### D1: 10 轮自循环审查强制执行

**决策**: explore/prototype/design 每个阶段 SHALL 执行完整的 10 轮自审，不允许提前终止，即使连续多轮无新问题也必须完成全部 10 轮。

**理由**: 固定轮次消除"何时收敛"的主观判断，确保每个阶段的设计经过充分打磨。10 轮的经验依据：前 3-4 轮通常发现结构性缺失，中间 3-4 轮发现细节问题，最后 2-3 轮进行边界和一致性 polish。

**替代方案**: 允许提前收敛（如连续 3 轮无新问题即终止）— 被拒绝，因为"无新问题"依赖自审者的能力，可能因审视疲劳导致假收敛。

### D2: self-reviews/ + cross-reviews/ 目录结构

**决策**: 变更级审查记录分为两个独立根目录：
- `self-reviews/` — 按阶段分子目录（explore/prototype/design），存放各阶段 10 轮自审记录
- `cross-reviews/` — 按审查批次分子目录（时间戳命名），存放四视角交叉审查报告

**理由**: `self-reviews` 和 `cross-reviews` 语义自明，一眼区分"自我检查"和"多视角交叉验证"。与现有 `review-reports/` 名称完全不同，避免混淆。

**替代方案**: `phase-reviews/` + `multi-perspective/` — 语义不如 self/cross 直白；自审记录按阶段分散存放 — 结构不统一，`prototype.pen` 是文件而非目录，无法容纳子目录。

### D3: 时间戳命名规范

**决策**: 审查记录统一以开始时间命名：
- 单文件报告: `{YYYYMMDD}-{HHMMSS}.md`（如 `20260514-093000.md`）
- 多文件报告: `{YYYYMMDD}-{HHMMSS}/` 目录（如 `20260514-150000/`）

**理由**: 自然排序即时间序，无需维护轮次编号；时间戳精确到秒，避免并发冲突；文件名自描述，无需额外元数据。

### D4: 阶段边界白名单模式

**决策**: 每个阶段的实现 SHALL 只允许创建其"输出产物"表中列出的文件。任何未在输出产物表中列出的文件均禁止创建。信息不足时：
1. 在当前阶段文档中标记"待补充"并说明具体缺少什么
2. 记录到 `docs/skill-suggestion.md` 作为流程改进建议
3. 禁止自创新文档来填补信息空缺

**理由**: 白名单模式消除"Agent 自创文档"的不确定性，确保变更目录结构始终可预测。skill-suggestion.md 作为安全阀，让未满足的需求有记录管道而不污染产出。

### D5: 功能设计内容升维

**决策**: functional-designs/ 的功能点描述从"功能点ID + 名称 + 描述 + 关联"扩展为：
- 所属页面/菜单路径
- 触发操作（按钮/链接/动作）
- 涉及表单项（字段名/类型/校验规则/默认值）
- 业务规则（前置条件/校验规则/触发条件/后置结果）
- 业务流程位置

**理由**: 为 prototype 提供精确输入，让原型设计阶段无需重新猜测页面结构和交互细节。聚焦"用户能做什么、页面什么样、业务规则是什么"。

### D6: 四视角审查保持事件驱动

**决策**: 四视角审查不引入固定轮次。每次审查（包括初次审查和分级重审）以独立时间戳目录保存。闭环逻辑不变：1轮 → 修复 → 按严重度分级重审 → 验证关闭。

**理由**: 四视角审查本质是"交叉验证"而非"自我完善"，审查成本高（4 Agent 并行），问题发现数通常在前 1-2 轮收敛。固定轮次会导致大量空转。

**替代方案**: 四视角审查也固定 5 轮 — 被拒绝，成本收益不匹配。

### D7: 自审执行方式

**决策**: 自审由当前阶段 Agent 自己执行（方案A），不启动独立审查 Agent。Agent 在完成阶段产物初稿后，进入 10 轮自审循环：每轮检查维度 → 发现问题 → 修复 → 记录报告 → 进入下一轮。

**理由**: 自审的目标是"自我完善"而非"独立验证"。Agent 最了解自己产出物的上下文，外部 Agent 在自审阶段反而需要额外信息同步。真正的独立验证留给四视角交叉审查。

### D8: 原型自审聚焦覆盖性

**决策**: prototype 阶段的 10 轮自审聚焦"覆盖性检查"——确保所有 FP 有对应原型页面、所有交互状态（加载/空/错误/边界）有对应组件。自审全部完成后才进入用户评审。

**理由**: prototype 的核心价值是"可视化功能设计"，覆盖性是第一优先级。视觉风格一致性、组件命名统一性等作为辅助维度。用户评审后置，确保用户看到的是经过自审打磨的版本。

## Risks / Trade-offs

- **[R1] 10 轮自审可能导致 Agent 上下文膨胀** → 每轮报告为独立文件，主 Agent 仅持有当前轮次上下文；历史轮次按需读取，减轻上下文压力
- **[R2] 10 轮自审可能产生大量冗余修复（后期多为微调）** → 后期轮次的审查维度可自动加权转向"边界检查"和"一致性检查"，减少对核心内容的无意义修改
- **[R3] 阶段边界白名单可能与现有 Skill 实现冲突** → 实现阶段需逐一审计现有 Skill 的文件创建行为，发现违规时更新 Skill 而非白名单
- **[R4] cross-reviews/ 多批次目录可能导致旧引用失效** → synthesis.md 维护审查批次索引，记录每批次的目录路径和审查原因
