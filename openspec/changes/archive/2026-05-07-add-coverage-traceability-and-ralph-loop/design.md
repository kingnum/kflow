## Context

当前 Skills 体系在 `core-mechanisms.md` (v1.7.0) 中定义了完整的阶段门控机制，各阶段产物（functional-designs/、detailed-design.md、api-tests/、e2e-tests/、integration-tests/）均有独立的 index.md + part-NN.md 目录化结构。但缺少跨阶段的统一覆盖追溯文件，无法在门控时机械化验证"每个功能点在各阶段产物中是否都有对应落点"。

同时，当前子代理使用局限于特定场景（kflow-design 四视角审查、kflow-code 多 Agent 并行），其他执行类阶段由主 Agent 内联执行，缺少统一的迭代优化机制。已有的 ralph-loop Skill（`.claude/skills/ralph-loop/SKILL.md`）具备自引用迭代循环能力，但未与阶段 Skill 集成。

## Goals / Non-Goals

**Goals:**
- 建立 `traceability.md` 覆盖追溯矩阵，实现 FP→Design→API Test→E2E Test→Code→Integration Test 全链路可追溯
- 在阶段门控中集成覆盖率自动验证（对应列覆盖率 = 100% 方可通过）
- 执行类阶段统一采用 ralph-loop 子代理模式，输出质量由迭代保证
- 子代理完成后主 Agent 验收，不满足要求时记录 skill-suggestion.md 并询问重跑
- 各阶段独立维护 traceability.md 对应列，不依赖跨阶段聚合

**Non-Goals:**
- 不改变创造性阶段（设计探索/原型设计/详细设计/归档/审计）的执行模式
- 不修改 ralph-loop Skill 本身的核心机制
- 不改变现有产物文件的格式和结构
- 不在 traceability.md 中存储设计内容本身（仅存储覆盖映射）
- 不自动触发阶段回退（验收不通过由用户决策）

## Decisions

### Decision 1: traceability.md 作为独立文件，各阶段独立写入

**选择**: 在 `docs/changes/{change}/traceability.md` 创建独立的覆盖追溯矩阵文件，每个阶段完成后由该阶段的执行者填写对应列。

**替代方案考虑**:
- 嵌入 `.status.md`: 会使状态文件过于臃肿，状态和覆盖是不同维度
- 嵌入各 `index.md`: 覆盖信息分散在多个文件，无法一处总览
- 自动聚合: 需要跨文件解析，增加复杂度且不必要（独立维护更可靠）

**格式设计**:

```markdown
# 覆盖追溯矩阵：{change-name}

## 覆盖总览

| 功能点ID | 功能设计(§) | 详细设计(§) | 接口测试(ID) | E2E测试(ID) | 编码实现(SC) | 集成测试(ID) |
|----------|------------|------------|-------------|------------|-------------|-------------|
| FP-001 | p01 §2.1 | des §3.2 | API-01 | S01 | user-auth | I-01 |

## 阶段覆盖统计

| 阶段 | 覆盖FP数 | 总FP数 | 覆盖率 | 状态 | 更新时间 |
|------|---------|--------|--------|------|----------|
| 功能设计 | 45 | 45 | 100% | ✅ | 2026-05-07 10:00 |
| 详细设计 | 45 | 45 | 100% | ✅ | 2026-05-07 11:00 |
| 接口测试 | 44 | 45 | 97.8% | ⚠️ | - |

## 缺口追踪

| 功能点ID | 缺失阶段 | 发现时间 | 状态 |
|----------|---------|----------|------|
| FP-023 | 接口测试 | 2026-05-07 14:00 | 待补充 |
```

### Decision 2: 执行类阶段的 ralph-loop 子代理模式

**选择**: 执行类阶段（计划/编码/审查/接口单元测试/E2E测试/集成测试/缺陷修复）统一使用 ralph-loop 子代理，创造性阶段维持主 Agent 直连。

**替代方案考虑**:
- 全部阶段使用: 设计类阶段需要创造性和上下文理解，强行 ralph-loop 会降低质量
- 完全不用 ralph-loop: 失去迭代优化带来的质量提升

**执行流程**:
```
1. 复杂度评估 → 确定轮次 (5-10)
2. 构建阶段专属提示词 + 完成承诺 COMPLETED
3. 启动 ralph-loop 子代理 (/ralph-loop)
4. 子代理迭代执行，直到输出 COMPLETED
5. 主 Agent 验收:
   - 产物文件是否齐全
   - traceability.md 对应列覆盖率 = 100%
   - 无 TODO/TBD 占位符
6. 验收结果:
   ├── 通过 → 更新状态 → 释放门控
   └── 不通过 → 记录 skill-suggestion.md → AskUserQuestion 重跑
```

### Decision 3: 复杂度评估公式

**选择**: 基于可量化指标计算复杂度，映射到迭代轮次。

```
复杂度分 = 功能点数 × 1 + 接口数 × 1.5 + 场景数 × 2

低复杂度 (< 20 分) → 5 轮
中复杂度 (20-50 分) → 7 轮
高复杂度 (> 50 分) → 10 轮
```

**说明**: 场景数权重最高（E2E/集成测试场景复杂度高于单接口），接口数次之，功能点基准权重为 1。

### Decision 4: 完成承诺格式

**选择**: 统一使用 `COMPLETED` 作为 ralph-loop 完成承诺。

**替代方案考虑**:
- 每阶段不同承诺 (e.g., `COVERAGE_100_PLAN`): 过于复杂，ralph-loop 只需要简单的信号
- 无承诺机制: 无法自动化判断子代理完成

`COMPLETED` 统一简洁，子代理在验证产物合格后输出即可。

### Decision 5: traceability.md 生命周期管理

**选择**:
- **创建**: `kflow-design` 阶段创建 traceability.md，基于 functional-designs/index.md 中的 FP 清单初始化空白矩阵
- **填写**: 各执行阶段完成后，由该阶段的子代理（通过 ralph-loop）填写对应列
- **验证**: 阶段门控时读取对应列，检查覆盖率 = 100%
- **归档**: 归档前最终检查所有列覆盖率 = 100%

## Risks / Trade-offs

- **[风险] traceability.md 手动维护可能导致遗漏** → 通过门控自动检查覆盖率，覆盖率 < 100% 时阻塞，强制追补
- **[风险] ralph-loop 可能陷入无限循环** → `--max-iterations` 参数兜底，达到上限后主 Agent 强制验收
- **[风险] 复杂度评估不准确导致轮次不足或过多** → 首轮实践后可在 skill-suggestion.md 中反馈调整，后续迭代优化公式
- **[风险] 创造性阶段不使用子代理可能产出质量波动** → 已有四视角审查机制保障设计质量，本次不改变该模式
- **[取舍] traceability.md 独立维护 vs 自动聚合** → 选择独立维护，牺牲部分自动化换取可靠性和可控性
