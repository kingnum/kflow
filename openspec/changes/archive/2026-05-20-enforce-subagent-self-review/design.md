## Context

当前三阶段（explore/prototype/design）自审由阶段 Agent 自身执行（见 `phase-self-review` spec §"自审由当前阶段 Agent 执行"）。prototype 阶段的 VERIFY 步骤已验证"子代理串行审查"模式有效（5 轮导航 + 5 轮 Playwright），但 SELFREV 步骤仍由主 Agent 执行，与 VERIFY 模式不一致。design 阶段仍使用"分工制"（轮次分组维度），与其他两阶段的"重复制"不一致。

`core-mechanisms.md` §15 将设计类阶段归类为"主 Agent 直连"，本变更在该框架内针对自审环节开放子代理执行。

## Goals / Non-Goals

**Goals:**
- 三阶段 SELFREV 统一改为子代理串行执行，每轮独立 Agent(subagent)
- 子代理审查范围一致性：每轮子代理执行相同的全维度检查（重复制）
- design 阶段从分工制改为重复制，与 explore/prototype 统一
- 子代理边审边修，发现问题直接修复产物文件
- 最小变更原则：仅改造 SELFREV 步骤，不动其他流程

**Non-Goals:**
- 不改变自审维度定义（探索/原型/设计各四维度不变）
- 不改变 10 轮强制执行规则
- 不改变自审报告命名和内容规范
- 不改变自审后流转（explore→完成、prototype→用户评审、design→四视角交叉审查）
- 不改变 VERIFY 步骤的现有子代理机制
- 不新增功能

## Decisions

### Decision 1: 子代理执行 vs 主 Agent 执行

**选择**: 子代理执行

**理由**:
- 子代理拥有独立上下文，不受"作者盲点"影响——主 Agent 撰写产物后上下文已被内容填充，自审时难以切换视角
- prototype VERIFY 已验证子代理审查模式有效
- 每轮子代理独立启动，上下文干净，等价于"每次重新审查"

**替代方案**: 维持主 Agent 自审（拒绝）——作者盲点问题无解

### Decision 2: 串行 vs 并行

**选择**: 串行（顺序执行）

**理由**:
- 并行子代理在相同产物上审查，会高概率发现相同问题（浪费 Token 和去重成本）
- 串行模式中：Round N 发现问题 → 主 Agent 读取 → 修复 → Round N+1 在修复后的产物上审查 → 发现新问题 → 自然收敛
- 串行保证每轮有增量价值，并行无法保证

**替代方案**: 并行子代理（拒绝）——重复发现率高，浪费资源

### Decision 3: 重复制 vs 分工制（design 阶段）

**选择**: 重复制（每轮全维度）

**理由**:
- 分工制（R1-3 结构性、R4-7 细节、R8-10 边界）的假设是"早期轮次发现的问题在后期轮次仍会关注"，但如果后期不检查结构性维度，早期遗漏的结构性问题可能永远不会被发现
- 重复制自然收敛：后期轮次问题越来越少，是因为产物真正在变好，而不是因为检查范围变窄
- 三阶段统一为重复制降低心智负担

**替代方案**: 维持 design 分工制（拒绝）——与其他阶段不一致，且存在后期遗漏早期维度问题的风险

### Decision 4: 边审边修 vs 仅出报告

**选择**: 边审边修（子代理发现问题直接修复产物文件）

**理由**:
- 子代理拥有完整 Skill 上下文，能够正确修复
- 减少轮次间"报告→主 Agent 修复"的延迟
- 子代理修复后产物即为下一轮审查的输入，加速收敛

**约束**: 子代理修复应保守——只修复确认的问题，不做重构或额外改进（非必要不增加新功能）

## Additional Impact: SELFREV vs VERIFY 区分表

`core-mechanisms.md` §16.4 中「与自审机制的区别」表当前以"执行者"作为 SELFREV 和 VERIFY 的主要区分维度：

| 维度 | 当前 | 变更后 |
|------|------|--------|
| SELFREV 执行者 | 当前阶段 Agent 自身 | **子代理 (Agent subagent)** |
| VERIFY 执行者 | 独立子代理 | 独立子代理（不变） |

变更后两者执行者统一为子代理，区分维度从"执行者"转为"目的 + 范围"：
- SELFREV: 维度级审查（4 维度），10 轮，边审边修
- VERIFY: 具体检查项（5 项目），5 轮，出报告

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| Token 成本增加：10 轮 × 子代理，每轮需要独立上下文 | 子代理上下文仅包含审查所需内容（产物文件 + 维度规则），不加载完整对话历史 |
| 子代理修复质量：子代理修复后可能引入新问题 | 下一轮子代理会重新审查，形成自纠正循环 |
| 收敛速度：如果子代理修复引入回归，收敛可能变慢 | 10 轮下限保证至少 10 轮审查，即使前期有回归也会在后期被发现 |
| design 从分工制改为重复制后维度覆盖面变化 | 每轮全维度检查消除了"后期遗漏早期维度"的风险，实际上更安全 |
