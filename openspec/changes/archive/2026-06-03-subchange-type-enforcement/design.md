# Design: 子变更类型严格二分执行机制

## Context

当前系统中子变更类型严格二分（后端/前端）的规则已在 `04-gates-and-transitions.md §5.3` 中定义，但存在以下缺口：

1. **源头无标记**：explore 阶段的功能点拆分不区分前后端，FP 粒度过粗时一个 FP 自然跨越两层
2. **划分无校验**：design 阶段子变更划分纯靠人工判断类型，无自动一致性校验
3. **共享无归属**：错误码、配置项、共享 DTO 等跨层关切无明确的子变更归属规则
4. **事后无检测**：verify D3 只检查输入源是否存在，不检查输出是否越界；code-review 按子变更类型选择检查项，后端SC 不做前端代码质量检查

## Goals / Non-Goals

**Goals:**
- 在 explore 阶段就为每个 FP 标记前后端类型，从源头消除混合可能性
- 在 design 阶段自动校验子变更类型一致性，混合则阻塞
- 为跨层共享关切提供明确的默认归属规则
- 在 verify 和 code-review 阶段增加越界检测作为兜底

**Non-Goals:**
- 不改变子变更划分的整体流程和原则（仍由 design 阶段基于完整设计认知划分）
- 不增加第三种子变更类型（保持后端/前端严格二分）
- 不改变项目类型判定逻辑（前后端项目 vs 纯后端项目）

## Decisions

### Decision 1: FP 类型标记在 explore 阶段完成

**选择**：在 explore 的 SPLIT 步骤中为每个 FP 标记类型，而非推迟到 design 阶段。

**理由**：
- explore 阶段 FP 拆分时已经明确了每个 FP 的用户故事、页面/操作/业务规则——这些信息足以判断前后端归属
- 推迟到 design 阶段会失去源头约束——design 阶段接收的是已拆分但未分类的 FP，无法在接收时校验
- 与原子级拆分的自然互补："无法归类"成为判定 FP 粒度是否合适的新标准

**替代方案考虑**：
- 在 design 阶段分类：FP 已经流转到 design，此时再发现混合已经晚了，design 文档可能已按混合结构写好了
- 不在 FP 层面标记，仅加强 design 校验：可以挡住划分时的混合，但挡不住 FP 本身粒度过粗的问题

### Decision 2: 共享 DTO/类型定义不归属任何子变更，由变更级管理

**选择**：共享 DTO/interface 类型定义由变更级 agent 在「集成对接」步骤统一产出为共享类型文件，写入 `changes/{change}/shared-types/` 目录，前后端子变更各自引用而不定义。

**理由**：
- 共享类型是前后端的契约，归属任一子变更都会造成另一侧的隐式依赖
- 独立于子变更管理，可以在设计完成后、子变更编码前先行产出
- `shared-types/` 目录作为变更级产物，由 design 阶段产出后锁定，变更管理清晰

**替代方案考虑**：
- 归属后端子变更：前端子变更需要引用后端子变更的类型定义，引入子变更间源码依赖
- 创建独立的「契约子变更」：增加子变更数量但内容单薄（通常只有类型文件），性价比低

### Decision 3: 共享关切归属按"定义侧归后端，展示侧归前端"原则

**选择**：

| 共享关切 | 归属方 | 理由 |
|---------|--------|------|
| 错误码定义 + 后端处理逻辑 | 后端子变更 | 错误码是服务端契约 |
| 错误码→用户提示文案 | 前端子变更 | 展示逻辑属于前端域 |
| 配置项定义 | 后端子变更 | 后端定义，前端通过 API 获取 |
| Schema 变更 | 后端子变更 | 数据层属于后端域 |
| Schema→UI 影响 | 前端子变更（关联标注） | 前端域内响应式处理 |

**理由**：这个原则与现有系统架构一致——后端定义数据和规则，前端消费和展示。避免引入中性"共享"类型破坏二分体系。

### Decision 4: 越界检测采用 grep 模式匹配而非 AST 分析

**选择**：verify D3.2 和 code-review 越界检测使用 grep + 文件扩展名检测，而非 AST 解析。

**理由**：
- 与现有 code-review 的原型对账机制（grep 硬编码颜色值/组件名/路由）保持一致
- 跨语言项目（后端 Java/Go/Python + 前端 React/Vue）无法用统一 AST
- 越界检测的精度要求不高——检测到 .tsx 文件存在于后端SC 的代码目录就足以触发提示

**替代方案考虑**：
- AST 分析：更精确但需要语言特定工具，跨项目类型不可行
- 仅依赖人工审查：失去自动化兜底的价值

### Decision 5: 越界检测严重度设为 🟡 警告，不阻塞流程

**选择**：跨层越界检测结果标记为 🟡 警告级别，不阻塞阶段流转。

**理由**：
- 避免误报阻塞正常流程（如工具函数文件、类型定义文件可能被错误分类）
- 警告足以引起人工注意和确认，确认后可继续
- 与现有 verify 严重度体系一致（输入源缺失→🔴阻塞，占位符→🟡警告，越界检测→🟡警告）

## Risks / Trade-offs

- **FP 类型标记增加 explore 负担**：每个 FP 需要额外判断前后端归属。→ 缓解：类型判定标准明确（API/数据/规则=后端，页面/组件/样式=前端），且"无法归类=需要继续拆"的规则实际降低了后续返工风险。
- **旧版功能设计文档不兼容**：已有变更的 functional-designs/index.md 缺少类型列。→ 缓解：verify D2 对缺少类型列的旧文档标记 🟡 警告而非 🔴 阻塞，允许平滑过渡。
- **grep 越界检测可能误报**：前端项目中的 .ts 文件可能被误判为后端代码。→ 缓解：🟡 警告级别 + 明确的误报排除指引（类型定义文件、工具函数文件）。
- **共享类型目录增加变更级管理复杂度**：`shared-types/` 作为新产物需要纳入门控检查。→ 缓解：仅在需要时创建（有跨子变更共享类型时才产出），作为条件产物（🔶）而非必须产物（✅）。
