## Context

DevFlow Skills 体系现有设计文档包含核心运行机制和 8 个 Skill 规格，但部分阶段设计不够详细，缺少实际实现所需的具体流程、输出格式和门控规则。同时，不同项目类型（前后端项目 vs 纯后端项目）的开发流程存在差异，现有设计未区分处理。

**当前状态**：
- docs/designs/core-mechanisms.md 定义了目录结构和状态文件格式
- docs/designs/skills/ 下有 8 个 Skill 规格，但部分缺少详细流程
- 功能点上限为 20，无子变更上限
- 测试阶段命名和流程不够清晰
- 缺少归档阶段、流程指引阶段设计

**约束**：
- 保持文档驱动、阶段门控的核心设计原则
- 遵循现有目录结构规范
- 参考 superpowers、gstack、openspec 等现有 Skills 设计模式

## Goals / Non-Goals

**Goals:**
- 完善 10 个 Skills 的详细规格设计，覆盖全开发流程
- 区分前后端项目和纯后端项目的开发流程差异
- 定义清晰的门控规则和阶段流转机制
- 提供可执行的自动化流程设计
- 增加必要的索引和记录文件机制

**Non-Goals:**
- 不涉及 Skills 的代码实现
- 不修改现有 kflow-skills 体系
- 不改变文档驱动开发的核心架构

## Decisions

### 1. 功能点上限调整：20 → 10

**决策**：将子变更功能点上限从 20 调整为 10。

**理由**：
- 功能点级任务拆分采用全展开形式，每功能点 7-9 个 checkbox
- 20 个功能点会产生 140-180 个 checkbox，过于繁重
- 10 个功能点约 70-90 个 checkbox，粒度适中

**替代方案**：
- 方案 A：功能点级分组简化（分组显示）——用户选择全展开
- 方案 B：保持 20 上限，简化任务拆分 ——可能影响追踪精度

### 2. 子变更上限：最大 20 个

**决策**：一个变更最多 20 个子变更，超过时拆分为多个变更。

**理由**：
- 防止变更过于庞大，管理复杂度过高
- 20 个子变更足够覆盖大部分功能需求
- 超过 20 个子变更暗示需求粒度过粗，应按业务划分

### 3. 项目类型区分机制

**决策**：在设计探索阶段自动检测项目类型，区分前后端项目和纯后端项目。

**检测规则**：
- 检测前端工程标识：package.json 前端框架依赖、.vue/.tsx/.jsx 文件、前端构建配置
- 存在 → 前后端项目，走 7 阶段流程
- 不存在 → 纯后端项目，走 5 阶段流程（跳过原型设计、浏览器自动化测试）

**理由**：
- 纯后端项目无需原型设计和浏览器自动化测试
- 接口单元测试作为纯后端项目的验收标准
- 减少不必要的阶段执行，提升效率

### 4. 测试阶段命名与定位

**决策**：重命名为 `devflow-e2e-qa`（浏览器自动化功能测试），仅前后端项目启用。

**规则**：
- 核心是使用浏览器进行功能测试，不使用 curl 等接口调用
- 前置数据缺失时通过浏览器操作录入
- 无法自动解决的问题咨询用户
- 每轮完整执行所有用例，异常视为失败

**理由**：
- 明确测试方式为浏览器自动化，避免混淆
- 与接口单元测试区分定位

### 5. 归档条件简化

**决策**：归档条件统一为"主变更 .status.md 中所有阶段标记完成"。

**理由**：
- 状态文件已记录所有阶段和子变更进度
- 简化门控判断，避免复杂的多条件检查
- 前后端项目和纯后端项目使用同一归档入口

### 6. 流程指引意图识别

**决策**：使用关键词映射识别用户意图，自动引导对应阶段。

**关键词映射**：
- 新功能/开始/创建 → devflow-explore
- 继续/下一步 → 继续活跃变更
- 测试/QA/验收 → devflow-e2e-qa
- 修复/Bug → devflow-fix
- 状态/进度 → devflow-status
- 归档/完成 → devflow-archive

**理由**：
- 降低用户使用门槛，无需记忆所有 Skill 名称
- 自动检测活跃变更数量，智能选择或让用户选择

## Risks / Trade-offs

### [风险] 功能点上限调整导致子变更数量增加

→ **缓解**：设置子变更上限 20，超过时提醒拆分变更

### [风险] 项目类型检测可能误判

→ **缓解**：检测结果显示给用户确认，允许手动修正

### [风险] 浏览器自动化测试前置数据录入场景多样

→ **缓解**：根据测试用例文档前置条件动态执行，无法解决时咨询用户

### [风险] 流程指引意图识别不完整

→ **缓解**：skill-suggestion.md 记录反馈，后续补充关键词

### [权衡] 纯后端项目验收仅依赖接口单元测试

→ **说明**：简化流程，但可能遗漏集成场景测试。可通过后续补充接口集成测试阶段解决
