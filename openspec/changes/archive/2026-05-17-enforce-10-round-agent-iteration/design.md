## Context

§15 当前设计允许 Agent 迭代子代理自行判断完成时机——"Agent 在所有产物合格后自然返回验收报告"。这导致 Agent 可能一轮通过即返回，执行质量无保障。§16 为创造性阶段建立了强制10轮自审机制，执行类阶段应采用对等的强制标准。

两类阶段的10轮机制并行互补：

| 维度 | §15 执行类阶段（本设计） | §16 创造性阶段（已有） |
|------|------------------------|----------------------|
| 执行者 | Agent 迭代子代理 | 主 Agent 直连 |
| 记录方式 | .status.md 轮次计数器（轻量） | self-reviews/{phase}/{ts}.md（详细报告） |
| 轮次来源 | 固定10轮，统一标准 | 固定10轮，分阶段维度 |

## Goals / Non-Goals

**Goals:**
- 所有执行类阶段通过 Agent 迭代子代理跑满10轮
- 轮次计数器记录在 .status.md 中，主 Agent 验收时检查
- 10轮为硬下限，即使提前满足产物条件也必须继续

**Non-Goals:**
- 不改变创造性阶段的10轮自审机制（§16 维持不变）
- 不改变缺陷修复3次尝试上限（上限和下限独立共存）
- 不为执行类阶段增加详细轮次报告（不需要 self-reviews/ 记录）

## Decisions

### Decision 1: 计数器放在 `## 基本信息` 区

在 .status.md 的 `## 基本信息` 区新增一行 `**执行轮次**: {N} / 10`，与 `**当前阶段**`、`**变更类型**` 等同级。阶段切换时重置为 `1 / 10`。变更级和子变更级 .status.md 统一采用此格式。

**理由**: 计数器跟随当前阶段生命周期，阶段级别的元数据应归属基本信息区。变更级集成测试阶段同样通过 Agent 迭代执行，需要对等追踪。不需要每阶段独立计数（阶段表中已有"备注"列可以记录轮次进度）。

### Decision 2: INIT 步骤由主 Agent 在 DISPATCH 前执行

§15.3 流程新增步骤 1.5（INIT）：主 Agent 在启动 Agent 子代理前，将 .status.md 中 `执行轮次` 写入 `1 / 10`。

**理由**: 计数器初始化和 Agent 调度由同一执行者（主 Agent）完成，避免 Agent prompt 中额外增加初始化逻辑。Agent 子代理只需负责递增，简化 prompt 复杂度。

### Decision 3: Agent prompt 内嵌轮次迭代指令

Agent prompt 增加：「每完成一轮迭代，更新 .status.md 执行轮次计数器为当前轮次号。必须完成全部 10 轮迭代后才可返回验收报告，禁止在第 10 轮前返回。」

**理由**: 轮次递增是 Agent 子代理的内部执行行为，由 Agent 自身完成。主 Agent 在 ACCEPT 步骤检查结果。

### Decision 4: 10轮与3次上限独立共存

- 10轮：Agent 迭代子代理的整体迭代轮次下限（外层循环）
- 3次上限：同一测试用例修复尝试的上限（内层缺陷修复循环）
- 两机制独立运作，不互相替代或冲突

**场景**: 第5轮时某测试用例3次修复尝试均失败 → 升级用户决策（保留/跳过/回退），但 Agent 子代理仍需完成剩余的5-10轮（处理其他可修复问题）。极端情况下所有用例在第1轮全部达到3次上限 → 升级用户，Agent 子代理后续2-10轮递增计数器并标注"所有问题已升级用户决策，无可修复项"，仍须递增至10/10后方可返回。

**理由**: 下限保障迭代深度、上限防止死循环，各司其职。上限触达不应成为提前终止轮次的理由（但缺乏可执行工作时可以——见 Decision 5）。

### Decision 5: 无实际工作可执行时允许记录并跳过

当 Agent 在所有10轮中已无实际工作可做（如所有测试通过、所有问题已修复、前轮无新发现），剩余轮次标记为"无新发现"并通过记录完成——但不跳过轮次计数。计数器仍递增到 10/10。

**理由**: 避免"为了凑轮次而制造问题"。10轮是下限检查——确保至少完成10轮才停止，但每轮内容由实际进展决定。非创造性阶段（如编码已完成所有FP实现）的后续轮次主要是验证和边界检查。

## Risks / Trade-offs

- **效率 vs 质量**: 10轮强制增加执行开销（预估低复杂度任务可能4-5轮即达标但须跑满10轮）。缓解：低复杂度任务后续轮次为轻量验证，不强制每轮都有产出。
- **与3次上限的层级混淆**: 缺陷修复阶段两种机制并存，Agent 可能混淆层级。缓解：prompt 中明确区分"轮次迭代"与"修复尝试"，Agent 在轮次报告中分别记录。
- **用户感知**: 用户可能在低复杂度变更中感受到"拖沓"。缓解：复杂度评估仍是 prompt 中的节奏指引，Agent 可根据不同轮次阶段调整每轮深度（前6轮重点执行、后4轮验证和边界检查）。
