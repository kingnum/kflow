## Context

DevFlow Skills 体系 v1.4.0 已定义 10 个 Skill 的完整设计规格，核心机制（阶段门控、子变更划分、审查闭环、根因分类路由）已稳定。本次 v1.5.0 升级聚焦于**补齐闭环缺口**和**规范化命名/文档管理**，不改变核心流程架构。

当前状态：
- `kflow-archive` 仅做文件移动 + 索引更新，未将设计知识沉淀到产品级
- `kflow-bug-fix` 仅处理子变更级测试失败，集成测试失败缺少标准化修复流程
- 缺少对 DevFlow Skills 使用情况的评估机制
- 缺少项目启动时的环境能力发现和工具链推荐机制
- `design-explore.md` / `design.md` 命名区分度不足

## Goals / Non-Goals

**Goals:**
1. 新增 `kflow-audit`：归档前自动审计 + 手动调用，问题归属阶段回退
2. 新增 `kflow-init`：环境能力发现（MCP + Skills），输出工具链方案
3. 扩展 `kflow-bug-fix`：单 Skill 支持子变更级和变更级两级缺陷修复
4. 增强 `kflow-archive`：归档时合并功能设计 + 详细设计到产品级
5. 统一文件命名：`functional-design.md` + `detailed-design.md`
6. 建立大文档拆分策略：产品级多文件 + 变更级条件拆分

**Non-Goals:**
- 原型设计（.pen）合并到产品级——pencil 文件需手动打开编辑，暂不可行
- 弱模型能力评估——暂缓，后续实验数据支撑
- 已有 Skill 的核心流程变更——仅扩展，不重构

## Decisions

### 决策 1：`kflow-audit` 作为归档门控而非独立阶段

**选择**：将 audit 嵌入 `kflow-archive` 的门控检查中，而非在阶段流程中新增"审计阶段"。

**理由**：
- 审计本身不产生设计产物，与"设计探索→原型→详细设计"等产出驱动的阶段不同质
- 归档前自动触发满足覆盖需求，手动调用满足灵活需求
- 单独阶段会增加流程复杂度，且用户几乎不会单独"进入审计阶段"

**替代方案**：审计作为归档后的独立步骤——被否决，因为审计不通过时归档已经完成，回退成本更高。

### 决策 2：`kflow-bug-fix` 单 Skill + 内部上下文检测

**选择**：一个 Skill 文件，执行时第一步检测调用上下文（子变更级 vs 变更级），然后路由到对应流程。

**理由**：
- 根因分类引擎（决策树 + 多 Agent）两级共享，避免分散维护后 drift
- 回退门控规则同源，差异仅在于回退范围（单子变更 vs 联动）
- 用户无需记忆两个不同的触发词
- Skill 文件约 400 行，仍在可维护范围内；两级差异用对比表呈现

**替代方案**：拆分为 `kflow-bug-fix` 和 `kflow-bug-fix-integration` 两个 Skill——被否决，因为共享内容重复维护成本高，不一致风险大。

**上下文检测逻辑**：
```
1. 读取活跃变更 .status.md → 当前阶段
   - "集成测试" + 存在 test-reports/integration/round-{n}.md → 变更级
   - "接口单元测试" / "E2E测试" → 子变更级
2. 无法判断时：AskUserQuestion 询问用户
```

### 决策 3：归档合并按设计域组织产品级文档

**选择**：`docs/designs/domains/{domain}.md` 作为合并目标，每个文件同时承载功能设计和技术设计（分上下两章）。

**理由**：
- 产品级文档按设计域划分是最自然的信息架构——阅读者按域查阅，而非按变更批次
- 功能设计 + 技术设计合并在一个领域文件内，形成"这个域长什么样（功能）+ 怎么实现的（技术）"的完整视图
- 变更批次信息通过"来源变更"标注追溯，不丢失溯源能力

**冲突检测**：当新归档变更涉及已有设计域时，默认为"替换更新"策略（新设计覆盖旧设计），保留旧版本链接。若检测到结构性冲突则提示人工裁决。

### 决策 4：文件命名采用"功能-详细"双层体系

**选择**：`functional-design.md`（设计探索阶段输出）+ `detailed-design.md`（详细设计阶段输出）。

**理由**：
- "functional" vs "detailed" 形成自然的范围递进——先定义做什么，再设计怎么做
- 两个文件都以 "design" 结尾，明确归属设计大类
- 消除了 "explore"（探索）的模糊感——"探索"听起来不像正式产出

### 决策 5：产品级文档从一开始即多文件拆分

**选择**：产品级采用 `index.md`（索引入口）+ `domains/*.md`（设计域）+ 全景文档（architecture / data-model / api-catalog / nfr-baseline），从一开始就拆分。

**理由**：
- 产品级文档必然持续增长（每次归档合并），单文件模式在 3+ 次迭代后不可持续
- 索引入口保持轻量，提供全局导航
- 全景文档仅在跨域变更时更新（如新增数据实体追加到 data-model.md）

**变更级文档拆分触发条件**：功能点 ≤30 单文件（`detailed-design.md` 内部分章节），>30 拆分为 `detailed-design/{domain}.md` + 索引入口。

### 决策 6：工具推荐优先级

**选择**：
- 原型设计阶段：pencil MCP 为唯一首选
- E2E 测试阶段：`/playwright-cli` skill 为首选，playwright MCP 为备选
- `kflow-browser-rules` 从推荐列表中移除

**理由**：pencil MCP 与 DevFlow 流程原生集成（.pen 文件输出门控），`/playwright-cli` skill 提供更高层的浏览器自动化封装。

## Risks / Trade-offs

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 文件重命名导致现有变更目录引用断裂 | 高 | 中 | 兼容期处理：Skill 读取时同时检查新旧文件名，提示迁移 |
| `kflow-bug-fix` 上下文检测误判 | 低 | 中 | 检测逻辑基于 `.status.md` 当前阶段，确定性高；无法判断时询问用户 |
| 归档合并时设计域匹配置冲突 | 中 | 低 | 默认替换更新 + 保留旧版本链接；结构性冲突时人工裁决 |
| 产品级文档拆分过细导致阅读碎片化 | 低 | 低 | 索引入口提供全局导航；领域文件内聚功能+技术，减少跨文件查阅 |
| `kflow-init` 无法感知所有 MCP | 中 | 低 | 分两层感知：运行时（已连接 MCP）+ 配置（settings.json）+ 内置知识库补充推荐 |
| 单 Skill 两级 bug-fix 文件过长 | 低 | 低 | 共享逻辑只写一次，两级差异用对比表，预估 400-500 行仍在可维护范围 |
