## Why

现有 DevFlow Skills 体系（v1.3.0）经过全面验证，发现 13 项改进空间，核心问题集中在三类：

1. **线性流程假设不适应现实开发**：当前设计假设探索→设计→计划→编码→测试→归档是单向不可逆的。实际开发中，编码阶段发现设计问题、需求中途变更、测试暴露设计缺陷等场景频繁发生，缺少回退机制导致"将错就错"式返工。

2. **设计阶段信息不足时被迫做出不可逆决策**：当前在探索阶段（信息最少时）进行子变更划分，详细设计阶段才有可能发现划分不合理，但 design-explore.md 已标记"完成"，回退陷入困境。

3. **质量保障断层**：设计文档有四视角审查，但代码实现无审查；子变更有独立测试，但跨子变更接口契约无人验证；非功能需求（性能、安全合规）在设计阶段无定义点，后期必然因性能/安全/合规问题返工。

## What Changes

### 架构调整：设计层级重新划分

- **详细设计从子变更级提升到变更级**：所有功能点的统一详细设计在变更级完成，子变更划分在详细设计之后执行
- **子变更职责简化为执行单元**：子变更仅负责计划→编码→测试→缺陷修复，不再承担设计职责
- 审查从"每子变更一次"变为"变更级一次"，减少审查次数同时提高全局视角
- design-explore.md 增加修订记录和版本标注

### 流程机制增强

- **新增 `⚠️ 需修订` 状态 + 阶段回退门控**：明确定义回退条件、回退后产物处理规则、依赖子变更联动规则
- **新增代码审查子阶段**：编码阶段完成后、子变更完成前，增加安全+质量两视角并行代码审查
- **新增集成测试阶段**：所有子变更编码完成后、归档前，验证跨子变更接口契约
- **子变更阶段统一拆分**：将模糊的"测试"阶段拆分为"接口单元测试"和"E2E测试"两个独立阶段
- **缺陷修复增加根因分类路由**：区分实现错误/测试错误/设计错误，设计错误触发阶段回退

### 设计质量增强

- **详细设计文档增加 NFR 章节**：性能需求、安全需求、可用性需求、可维护性需求
- **计划阶段增加 DoD 验收标准**：每功能点明确定义验收标准（Happy Path + Error Path + Edge Case + Quality）
- **审查闭环验证规则**：高严重度问题强制重新审查，中严重度原视角重审，低严重度抽查

### 运维与一致性

- **新增数据库迁移管理**：migrations/ 目录，迁移文件命名规范，执行与回滚记录
- **多环境配置**：service-guide.md 区分 dev/test/staging/prod 四套环境
- **跨变更冲突检测**：kflow-guide 在开始新变更时检测与活跃变更的文件冲突
- **条件产物引用规范化**：所有 Skill 文档中输入/输出产物用 ✅/🔶/⏭️ 明确标注项目类型差异

## Capabilities

### New Capabilities

- `change-rollback`: 需求变更管理与阶段回退机制，包含 ⚠️ 需修订 状态、回退门控规则、产物联动处理
- `code-review-phase`: 编码阶段后的代码审查子阶段，安全+质量两视角并行审查
- `integration-testing`: 变更级集成测试阶段，验证跨子变更接口契约
- `nfr-requirements`: 详细设计中的非功能需求定义，性能/安全/可用性/可维护性
- `database-migration-management`: 数据库迁移管理，版本化迁移脚本与执行记录
- `multi-environment-config`: 多环境服务配置，区分 dev/test/staging/prod
- `cross-change-conflict-detection`: 跨活跃变更的文件冲突检测与预防
- `defect-root-cause-classification`: 缺陷根因三分法分类（实现/测试/设计错误）与路由
- `definition-of-done`: 任务验收标准（DoD），每功能点 4 维验收
- `review-closed-loop`: 审查问题闭环验证，按严重度分级重审

### Modified Capabilities

- `devflow-explore`: 输出从"功能点+子变更划分"变为"功能点清单+需求澄清"，不再做子变更划分
- `devflow-design`: 从子变更级提升到变更级，统一设计所有功能点，子变更划分后置
- `devflow-plan`: 输入从子变更 design.md 变为变更级 design.md 相关章节，新增 DoD 验收标准
- `devflow-code`: 新增代码审查子阶段、数据库迁移管理步骤
- `devflow-e2e-qa`: 评分维度按项目类型区分，条件产物引用规范化
- `devflow-fix`: 新增根因分类机制，设计错误路由到阶段回退
- `core-mechanisms`: 阶段状态体系扩展（⚠️ 需修订），子变更阶段拆分，回退门控规则，目录结构调整

## Impact

- 影响所有设计文档：docs/designs/ 下 11 个现有文件需修改
- 新增能力规格：10 个 capability specs
- 目录结构调整：设计产物从 subchanges/*/ 移至变更级，新增 migrations/、integration-tests.md
- Skills 数量不变（仍为 10 个），但职责边界显著变化
- 子变更目录从 7 文件/目录简化为 2 文件/目录（tasks.md + test-reports/）
