## ADDED Requirements

### Requirement: 代码审查子阶段

系统 SHALL 在编码阶段的 TDD 循环全部完成后、子变更标记完成前，插入代码审查子阶段。

#### Scenario: 代码审查执行时机
- **WHEN** 子变更所有功能点的 TDD 循环完成（全 Green + 已提交）
- **THEN** 系统启动代码审查子阶段
- **AND** 审查通过后子变更编码阶段标记完成

#### Scenario: 代码审查不通过
- **WHEN** 代码审查发现高严重度问题 ≥ 1 或中严重度问题 ≥ 3
- **THEN** 系统标记编码阶段为 ❌ 阻塞
- **AND** 输出代码审查报告，列出待修复问题
- **AND** 修复完成后重新审查

### Requirement: 代码审查双视角并行

系统 SHALL 使用两个并行 Agent 执行代码审查，视角分别为安全+规范和质量+性能。

#### Scenario: 安全+规范视角审查
- **WHEN** 代码审查启动
- **THEN** Agent 1 检查 SQL注入、XSS、CSRF、敏感信息泄露、编码规范、依赖安全漏洞
- **AND** 输出安全+规范审查报告

#### Scenario: 质量+性能视角审查
- **WHEN** 代码审查启动
- **THEN** Agent 2 检查 N+1查询、内存泄漏风险、错误处理完整性、代码复杂度
- **AND** 输出质量+性能审查报告

### Requirement: 代码审查报告格式

系统 SHALL 按标准格式输出代码审查报告。

#### Scenario: 审查报告内容
- **WHEN** 代码审查完成
- **THEN** 报告包含审查时间、修改文件数、代码变更行数
- **AND** 问题清单含严重程度、类别、文件位置、描述、修复状态
- **AND** 包含门控结论（通过/阻塞）
