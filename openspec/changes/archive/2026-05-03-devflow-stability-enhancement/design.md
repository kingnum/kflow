## Context

DevFlow Skills 体系 v1.3.0 现有设计经过全面验证，识别出 13 项改进空间。核心架构问题在于：设计阶段信息不足时被迫做出不可逆的拆分决策（探索阶段就划分子变更），以及线性流程无法处理实际开发中的回溯需求。

**当前状态**：
- 10 个 Skills，2 种项目类型（前后端 7 阶段，纯后端 5 阶段）
- 详细设计在子变更级执行，四视角审查每子变更一次
- 子变更划分在设计探索阶段完成（信息最少的时候）
- 阶段流转为单向，无回退机制

**约束**：
- 保持文档驱动、阶段门控的核心设计原则
- 遵循现有 kebab-case 命名规范和目录结构约定
- 不增加 Skill 数量（维持 10 个）
- 渐进式流程机制保留（产品级/功能需求级/功能缺陷级自动调整）

## Goals / Non-Goals

**Goals:**
- 将详细设计提升到变更级，子变更划分后置到设计完成之后，消除"先划分后设计"的矛盾
- 建立完整的阶段回退机制，支持需求变更和设计错误的合规回溯
- 填补代码审查、集成测试两个质量保障断层
- 体系化非功能需求（NFR）、验收标准（DoD）、审查闭环
- 增加数据库迁移、多环境配置、跨变更冲突检测等运维支撑能力

**Non-Goals:**
- 不涉及 Skills 的代码实现
- 不修改现有 kflow-skills（旧体系）的任何内容
- 不改变文档驱动开发的核心架构
- 不引入新的外部依赖或工具

## Decisions

### 1. 设计层级重新划分：设计在变更级，执行在子变更级

**决策**：将 `kflow-design`（详细设计）从子变更级提升到变更级。设计探索完成后直接进入详细设计（变更级），完成统一设计后再划分子变更。子变更不再承担设计职责，仅负责计划→编码→测试的执行工作。

**新阶段归属**：

| 阶段 | 归属层级 | 变化 |
|------|---------|------|
| 设计探索 | 变更级 | 输出从"功能点+子变更划分"变为"功能点清单+需求澄清" |
| 原型设计 | 变更级 | 不变 |
| 详细设计 | **变更级** (原为子变更级) | 统一设计所有功能点，输出单一 design.md |
| 计划 | 子变更级 | 输入从子变更 design.md 变为变更级 design.md 相关章节 |
| 编码 | 子变更级 | 新增代码审查子阶段、数据库迁移管理 |
| 接口单元测试 | 子变更级 (新增) | 从模糊的"测试"阶段拆分出来 |
| E2E测试 | 子变更级 | 从模糊的"测试"阶段拆分出来 |
| 集成测试 | **变更级 (新增)** | 所有子变更完成后验证跨子变更接口契约 |
| 归档 | 变更级 | 增加集成测试门控条件 |

**子变更划分时机**：详细设计完成 → 审查通过 → 基于完整设计认知划分

**理由**：
- 消除"信息不足时被迫做不可逆决策"的根本矛盾
- 架构决策在变更级一次做出，全局一致（避免 3 个子变更选 3 种技术方案）
- 审查从 N 次（每子变更）减少为 1 次（变更级），审查者能看到完整架构
- 子变更职责简化为执行单元，每个子变更目录从 7 文件简化为 2 文件

**替代方案**：
- 方案 A：保持当前架构，增加设计修订机制修补 → 治标不治本，复杂度会持续上升
- 方案 B：完全取消子变更概念 → 粒度太粗，无法管理大型变更

### 2. 目录结构调整

**决策**：调整目录结构以反映设计层级变化。

```
docs/changes/{change}/
├── .status.md                     # 变更总状态
├── tasks.md                       # 变更总任务清单
├── design-explore.md              # 设计探索输出 (不含子变更划分)
├── design.md                      # 统一详细设计 (NEW: 从 subchanges/*/ 提升)
├── api-tests.md                   # 统一接口测试用例 (NEW)
├── e2e-tests.md                   # 统一E2E测试用例 (NEW, 仅前后端)
├── integration-tests.md           # 集成测试用例 (NEW)
├── prototype.pen                  # 原型设计 (可选)
├── architecture.md                # 架构设计 → 合并到 design.md
├── review-reports/                # 四视角审查 (变更级, 一次)
│   ├── business-review.md
│   ├── technical-review.md
│   ├── security-review.md
│   ├── quality-review.md
│   └── synthesis.md
├── migrations/                    # 数据库迁移 (NEW)
│   ├── 001_user-auth_create_users.sql
│   └── migration-log.md
├── test-reports/                  # 变更级测试报告
│   └── integration/               # 集成测试报告 (NEW)
│       ├── round-1.md
│       └── summary.md
├── subchanges/
│   └── {subchange}/
│       ├── .status.md             # 子变更状态
│       ├── tasks.md               # 子变更任务清单 (含DoD)
│       └── test-reports/          # 子变更测试报告
│           ├── api/
│           │   ├── round-1.md
│           │   └── summary.md
│           ├── e2e/
│           │   ├── round-1.md
│           │   └── summary.md
│           └── fix-reports/
│               └── fix-{timestamp}.md
└── checkpoints/                   # 中断恢复 (不变)
```

**关键变化**：
- 设计产物 (design.md, api-tests.md, e2e-tests.md) 从 `subchanges/{subchange}/` 移至变更级
- 新增 `integrations-tests.md` 和 `migrations/`
- 子变更目录从 7 文件简化为 2 文件（tasks.md + test-reports/）
- review-reports/ 保持在变更级（原来就在，仅确认）

### 3. 阶段回退机制

**决策**：在现有 5 种状态值（✅ 🔄 ⏳ ⏭️ ❌）基础上新增 `⚠️ 需修订`，定义完整的回退门控规则。

**新增状态值**：

| 状态值 | 含义 | 触发条件 |
|--------|------|---------|
| ⚠️ 需修订 | 阶段产物因需求变更或设计错误需修订 | 需求变更、设计错误根因、审查发现结构性问题 |

**回退门控规则**：

```
阶段回退:

触发回退时:
1. 当前阶段状态: 🔄 → ❌ 阻塞
2. 目标回退阶段状态: ✅ → ⚠️ 需修订
3. 目标阶段之后的所有阶段状态: 重置为 ⏳ 待开始
4. 受影响产物的状态: 标记为"待修订"（不删除，保留参考价值）

依赖子变更联动:
├── 仅当前子变更受影响 → 单独回退
├── 接口契约变更 → 依赖该接口的子变更联动回退
└── 架构级变更 → 所有子变更联动回退

回退完成时:
1. 修订阶段重新执行完毕: ⚠️ 需修订 → ✅ 完成
2. 后续阶段逐个重新执行
```

**design-explore.md 修订规则**：

| 修订时机 | 允许 | 修订范围 | 影响 |
|---------|------|---------|------|
| 详细设计阶段 | ✅ | 功能点拆分、技术方案 | 更新版本号+修订记录 |
| 第一个子变更编码 | ✅ (走需求变更流程) | 该子变更相关功能点 | 回退设计阶段 |
| 后续子变更编码 | ⚠️ 有限 | 仅该子变更 | 不影响其他 |
| 集成测试阶段 | ❌ | - | 必须创建新变更 |

### 4. 代码审查子阶段

**决策**：在编码阶段 TDD 循环全部完成后、子变更标记完成前，插入代码审查子阶段。采用两视角并行审查（轻量于设计阶段的四视角审查）。

```
编码阶段流程 (修订):
  门控检查 → 服务指引 → TDD循环(每功能点) → 全功能点完成
      → 代码审查(NEW) → 子变更完成 → 下一子变更
```

**审查视角**：

| Agent | 审查内容 | 门控规则 |
|-------|---------|---------|
| Agent 1 (安全+规范) | SQL注入/XSS/CSRF、敏感信息泄露、编码规范、依赖安全漏洞 | 高严重度 ≥1 → ❌ 阻塞 |
| Agent 2 (质量+性能) | N+1查询、内存泄漏、错误处理完整性、代码复杂度 | 高严重度 ≥1 或 中 ≥3 → ❌ 阻塞 |

**审查报告位置**：`subchanges/{subchange}/test-reports/review/` 或合并到编码阶段的产出中。

### 5. 集成测试阶段

**决策**：在所有子变更编码和 E2E 测试完成后，归档前，增加变更级集成测试。

**测试用例来源**：基于各子变更 design.md 中"与其他子变更的接口"章节自动生成。

**执行位置**：变更级，非子变更级。

**门控**：集成测试通过是归档的必要条件（前后端和纯后端项目均需执行）。

**纯后端项目**：集成测试聚焦于 API 间调用链和数据一致性验证，不涉及浏览器。

### 6. 子变更阶段统一拆分

**决策**：将子变更级模糊的"测试"阶段拆分为两个独立阶段。

```
当前:  详细设计 → 计划 → 编码 → 测试(模糊)
修订:  详细设计 → 计划 → 编码 → 接口单元测试 → E2E测试
```

子变更 `.status.md` 阶段表格更新为 5 列（之前 4 列），阶段名称精确化。

### 7. NFR 非功能需求体系化

**决策**：在详细设计文档模板中增加 NFR 章节，包含性能、安全、可用性、可维护性四个子域。

NFR 在编码阶段门控中作为设计完整性检查项：design.md 中 NFR 章节必须包含至少 1 项性能需求和 1 项安全需求。

NFR 级联到测试用例：api-tests.md 自动包含性能验证用例（如 P95 响应时间）。

### 8. DoD 验收标准

**决策**：计划阶段生成的每功能点任务清单中，增加 DoD 验收标准区块。

每功能点 4 维验收：Happy Path (≥1) + Error Path (≥2) + Edge Case (≥1) + Quality (≥1)。

### 9. 审查闭环验证

**决策**：审查问题修复后的验证按严重度分级：

| 严重度 | 验证方式 | 执行者 |
|--------|---------|--------|
| 高 | 重新审查（原视角 + 安全视角交叉检查） | Agent |
| 中 | 重新审查（原视角） | Agent |
| 低 | 抽查（随机 30%） | Agent |

综合报告 (synthesis.md) 增加问题追踪矩阵和关闭条件。

### 10. 数据库迁移管理

**决策**：在变更级目录下增加 `migrations/` 目录，迁移文件按 `{序号}_{子变更}_{描述}.sql` 命名。

编码阶段的 TDD 循环中，涉及数据模型变更的功能点增加"编写迁移脚本 + 编写回滚脚本"步骤。

### 11. 多环境配置

**决策**：扩展 service-guide.md 为四环境配置（dev/test/staging/prod），敏感信息（密码、密钥）通过环境变量引用，禁止明文写入。

### 12. 跨变更冲突检测

**决策**：kflow-guide 和 kflow-code 在操作前检查活跃变更之间的文件修改重叠。变更索引文件增加"影响文件"字段。

### 13. 缺陷根因分类路由

**决策**：kflow-bug-fix 的第一步从"识别问题"扩展为"识别问题 + 分类根因"。

三分法：实现错误(80%) → 修复代码；测试错误(15%) → 修正测试；设计错误(5%) → 触发阶段回退。

### 14. 条件产物引用规范化

**决策**：所有 Skill 文档的输入/输出产物使用标准化图例。

| 图例 | 含义 |
|------|------|
| ✅ 必须 | 文件必须存在 |
| 🔶 条件 | 取决于前置阶段是否执行 |
| ⏭️ 不适用 | 该项目类型不需要 |

## Risks / Trade-offs

### [风险] 大型变更的变更级设计文档过于庞大

→ **缓解**：design.md 按"设计域"组织章节，审查 Agent 按域分配；功能缺陷级走简化流程

### [风险] 设计全部在变更级完成后再编码，可能产生瀑布感

→ **缓解**：保留子变更的分批执行机制。设计完成后，无依赖的子变更立即可以开始编码，后续子变更与前一批测试可并行

### [风险] 阶段回退可能导致子变更连锁反应

→ **缓解**：
- 回退前先评估影响范围（仅当前 vs 接口契约变更 vs 架构级变更）
- 受影响产物保留（标记为"待修订"），不删除
- 用户确认修订方案后再执行

### [风险] 集成测试增加了阶段数量，可能延长开发周期

→ **缓解**：集成测试聚焦跨子变更接口契约，粒度粗、用例少。纯后端项目也需执行但不需要浏览器

### [权衡] 代码审查仅两视角 vs 设计审查四视角

→ **说明**：代码审查聚焦安全+质量两大高频问题域，业务正确性和技术合理性已在设计审查中覆盖，避免重复审查。后续可扩展

### [权衡] 纯后端项目也需集成测试

→ **说明**：纯后端项目虽无前端，但子变更间 API 契约验证同样重要。集成测试不涉及浏览器，用接口调用即可
