## Context

KFlow Skills 体系采用文档驱动的阶段门控机制，但在实际运行中发现5个执行可靠性缺陷。这些问题根源于：(1) 执行类阶段提示词中缺少"自动遍历全部子变更"的明确指令，导致子代理每轮仅处理单个子变更；(2) 原型设计 Skill 的修订模式缺失，新会话中无法正确走编排层入口加载现有原型；(3) 全局 skill-suggestion 捕获机制缺失，三种重要触发模式未被自动记录；(4) 流程中缺少用户最终验收确认环节；(5) 中断恢复仅依赖 .status.md 状态值，缺少产物完整性验证。

## Goals / Non-Goals

**Goals:**
- 编码/代码审查阶段自动遍历全部子变更，消除逐个子变更询问用户
- 原型设计增加修订模式入口，保证无论新会话还是同会话都经过 kflow-prototype-design 编排层
- kflow-init 向 CLAUDE.md 注入全局 skill-suggestion 自动捕获规则
- 集成测试+审计通过后增加用户访问确认门控
- 中断恢复增加阶段产物完整性验证，防止跳过未完成阶段

**Non-Goals:**
- 不修改执行类阶段的核心 TDD/审查逻辑（仅修改遍历指令）
- 不引入 hook 机制（skill-suggestion 捕获通过 CLAUDE.md 全局规则实现）
- 不改变现有阶段门控的状态值体系
- 不修改缺陷修复的三分法根因分类逻辑

## Decisions

### Decision 1: 编码阶段遍历范围 — "全部子变更+全部功能点"双层遍历

**选择**：在重复制模式的"构建阶段专属提示词"中，将遍历指令从"遍历全部功能点"升级为"遍历全部未完成编码的子变更，每个子变更内遍历全部功能点"。

**理由**：执行类阶段已统一为重复制模式（每轮遍历全部工作项），但编码阶段的"工作项"定义仅包含功能点，没有包含子变更。升级为双层遍历后，子代理每轮自动处理所有未完成子变更。

**替代方案**：在 kflow-code 外增加一个"编码调度器" Skill 遍历子变更——否决，因为这会增加额外复杂度，修改提示词即可解决。

### Decision 2: 原型设计修订模式 — 在 CHECK 后增加分支，不新增独立 Skill

**选择**：在 kflow-prototype-design 的 CHECK 步骤后增加"修订模式检测"——prototype/index.html 已存在时走修订模式，加载现有原型+design-prompt.md 约束+合并用户修订需求，然后委托调用底层原型skill。

**理由**：kflow-prototype-design 是编排层入口，负责状态管理+门控+验证。修订模式本质是同一个 Skill 的不同分支，不应拆分为独立 Skill。这样保证入口统一，底层执行引擎可替换。

**关键设计**：修订模式下跳过 INPUT→OPTIMIZE（已有 design-prompt.md），直接进入 DESIGN 步骤，但 prompt 中附加修订需求。

### Decision 3: skill-suggestion 全局捕获 — CLAUDE.md 规则，非 hook

**选择**：kflow-init 向 CLAUDE.md 追加 `## Skill 改进建议自动捕获` section，定义三种触发模式和处理规则。

**理由**：用户明确不使用 hook。CLAUDE.md 是每次会话都加载的文件，放在这里可以确保全局生效。三种触发模式对应三类信息：阻塞原因（因...无法...）、因果链（因...导致...）、用户纠正内容（用户纠正后 AI 回复中的"你说得对"等）。

**关键区分**：记录的是用户纠正的具体内容（业务功能问题），不是 Skill 执行机制问题。但如果纠正内容暴露了 Skill 行为缺陷，则同时记录到 skill-suggestion.md。

### Decision 4: 用户验收确认 — 放在集成测试通过+审计通过之后

**选择**：在 kflow-archive 的门控检查中增加用户验收确认步骤。集成测试通过且审计通过后，启动服务，AskUserQuestion 要求用户访问验证。

**理由**：这是变更交付前的最后一道人工验证门控。放在归档之前是因为归档意味着变更正式完成，归档后不应再回退。用户确认不通过时回到 bug-fix 而非记录 skill-suggestion.md——这是功能缺陷，不是 Skill 问题。

### Decision 5: 中断恢复产物验证 — GATE 步骤增强，不替代状态文件

**选择**：在 kflow-resume 的 GATE 步骤中，增加"产物完整性验证"子步骤。读取 .status.md 后，对每个标记为 ✅ 完成的阶段验证其产物文件和执行轮次。

**理由**：状态文件仍是主要状态源，但增加产物验证作为二次校验。这样不破坏现有状态机制，只是增加安全性。验证项包括：执行轮次=10/10、产物文件存在、traceability 覆盖率达标。

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 编码阶段遍历全部子变更可能增加单轮执行时间 | 重复制模式本身要求10轮自然收敛，单轮时间增加但总轮次不变 |
| 原型修订模式加载现有原型时，用户修订需求可能与现有设计约束冲突 | 修订模式下合并 prompt 时标注冲突点，AskUserQuestion 确认处理 |
| CLAUDE.md 全局规则增加 Token 消耗 | 规则 section 控制在 30 行以内，仅定义触发模式和处理规则 |
| 用户验收确认环节增加人工交互，可能阻塞自动化流程 | AskUserQuestion 提供"跳过确认，直接归档"选项（需用户主动选择） |
| 中断恢复产物验证增加 resume 的检查时间 | 验证仅检查文件存在性和简单字段解析，不执行复杂分析 |
