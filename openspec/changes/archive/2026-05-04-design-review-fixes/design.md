# 设计评审修正：技术设计

## Context

KFlow Skills 体系设计当前版本 v1.5.0，包含 12 个 Skill 的设计文档。2026-05-04 完成设计评审，发现 18 个问题，涉及文件命名一致性、Skill 边界划分、阶段门控完善、checkpoint 结构等。本次修正将设计版本推进到 v1.6.0。

核心约束：
- 设计文档位于 `docs/designs/`，运行时产物位于 `docs/changes/`
- 阶段门控策略不变：状态文件驱动 + 正向/回退门控
- Change → Subchange 层级划分不变
- 所有修改限于设计文档层面，不涉及实际 Skill 实现代码

## Goals / Non-Goals

**Goals:**
- 消除所有文件命名引用不一致（`design.md`、`functional-detailed-design.md` 笔误）
- `kflow-code` 职责精简，审查逻辑提取为独立 Skill
- 集成测试拥有独立 Skill 入口，内聚四分法缺陷修复循环
- `kflow-bug-fix` 简化为纯子变更级，消除上下文检测歧义
- checkpoint 按变更级/子变更级两级存储，恢复路径清晰
- 变更级服务刷新同步点：解决多子变更并行编码后服务状态一致性
- 架构评估自动触发：连续 3 轮同用例失败后自动分析并输出多方案
- 阶段数量描述修正为 9/7，完整反映实际流程

**Non-Goals:**
- 不改变阶段门控的核心机制（状态文件、正向/回退门控）
- 不改变 Change → Subchange 层级划分逻辑
- 不改变四视角审查和代码审查的视角分工
- 不实际实现 Skill 代码（仅设计文档层面）
- 不改变归档设计合并流程的逻辑
- 不改变条件产物引用规范

## Decisions

### D1: 文件名统一

**选择**: 功能设计文件统一为 `functional-design.md`，详细设计文件统一为 `detailed-design.md`

**替代方案**: 
- 合并为单文件 `design.md` → 驳回，功能和详细设计关注点不同，合并后归档时设计合并困难
- 使用 `feat-design.md` / `tech-design.md` → 驳回，中文团队认知负担高

### D2: kflow-code 拆分策略

**选择**: 仅拆分出 `kflow-code-review`，TDD编码、数据库迁移、服务指引、跨变更冲突检测、多 Agent 调度保留在 `kflow-code`

**替代方案**:
- 数据库迁移独立 Skill → 驳回，迁移是编码数据层实现的一部分，拆分导致碎片化
- 多 Agent 调度独立 Skill → 驳回，调度是编码阶段的执行策略而非独立阶段

**关键边界**:
```
kflow-code                 kflow-code-review
├── 数据层 (含迁移)          ├── Agent 1: 安全+规范
├── 业务层                   ├── Agent 2: 质量+性能
├── 接口层                   ├── 门控验证
├── Refactor                ├── 闭环重审
├── 服务指引生成             └── code-review.md
└── 跨变更冲突检测
```

### D3: 集成测试 Skill 与变更级修复内聚

**选择**: `kflow-integration-test` 内聚集成测试执行 + 四分法缺陷修复循环

**替代方案**: 独立的 `kflow-bug-fix-integration` Skill → 驳回，集成测试和修复是强耦合循环（修复后立即重测），拆分导致不必要的 Skill 跳转

**流程**:
```
kflow-integration-test
├── 读取 integration-tests.md
├── 执行集成测试
├── 全通过 → summary.md → 完成
├── 失败 → 四分法根因分类
│   ├── 接口实现错误 → 定位子变更 → 修复 → 重测 → 重新集成
│   ├── 接口契约错误 → 更新契约 → 联动修复 → 重新集成
│   ├── 测试用例错误 → 修正测试用例 → 重新集成
│   └── 架构设计错误 → 架构评估 → 用户决策 → 回退
├── 3轮同用例失败 → 自动触发架构评估
└── 循环直到全通过
```

### D4: kflow-bug-fix 简化

**选择**: `kflow-bug-fix` 仅处理子变更级（三分法），变更级修复内聚到 `kflow-integration-test`

**替代方案**: 拆分为 `kflow-bug-fix` + `kflow-bug-fix-integration` → 驳回，与 D3 内聚决策一致

**关键简化**: 不再需要读取 `.status.md` 判断当前阶段做上下文路由，入口 Skill 即上下文。

### D5: 变更级服务刷新同步点

**选择**: 作为门控规则实现（非独立 Skill），在所有子变更编码+审查完成后、首个测试开始前强制执行

**替代方案**:
- 各子变更各自重启 → 驳回，多子变更共享同一服务，各自重启互相干扰
- 独立 `kflow-service-ready` Skill → 驳回，非用户直接调用阶段，增加认知负担

**执行流程**:
```
门控触发: 所有子变更编码+审查完成
    │
    ▼
1. 收集所有迁移脚本，按序号排序
2. 停止当前运行的服务
3. 编译后端 (service-guide.md)
4. 编译前端 (前后端项目)
5. 执行全部未执行迁移
6. 启动后端服务 (dev)
7. 启动前端服务 (dev)
8. 健康检查 (/health + /db-health)
    │
    ├── 失败 → ❌ 阻塞 (障碍自解循环)
    └── 通过 → 释放门控 → 子变更进入测试
```

### D6: 两级 checkpoint

**选择**: 变更级操作 checkpoint 存 `docs/changes/{change}/checkpoints/`，子变更级操作 checkpoint 存 `docs/changes/{change}/subchanges/{subchange}/checkpoints/`

**归属规则**:
| 阶段 | 级别 |
|------|------|
| 设计探索、原型设计、详细设计 | 变更级 |
| 计划、编码、代码审查、接口单元测试、E2E测试 | 子变更级 |
| 集成测试、归档 | 变更级 |
| 缺陷修复 | 子变更级（从子变更测试触发） |

### D7: 架构评估自动触发

**选择**: 在 `kflow-integration-test` 中实现自动触发，连续 3 轮同一测试用例 ID 失败时自动启动架构评估 Agent

**关键设计点**:
- "同一用例": 按测试用例 ID 追踪
- 自动分析 + 多方案输出（自动），但改造执行需用户确认
- 失败计数器: 用例通过或设计修订后归零

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| Skill 从 12 个增加到 14 个，用户认知负担增加 | kflow-guide 意图识别覆盖所有 Skill，降低记忆负担 |
| kflow-code 和 kflow-code-review 之间的门控依赖增加了流程步骤 | 门控检查自动化，用户无需手动管理状态文件 |
| checkpoints 两级化增加了恢复时的查找复杂度 | 恢复流程优先读子变更级，无则读变更级，规则简单 |
| 变更级服务刷新失败可能阻塞所有子变更测试 | 障碍自解原则覆盖，编译/启动/健康失败均可自动修复 |
| 架构评估自动触发可能产生误报 | 需用户确认后才执行改造方案，误报不会导致破坏性操作 |
