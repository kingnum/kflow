## Context

当前原型设计阶段的验证机制覆盖率低（仅 1 个业务流程最小点击测试），huashu-design 在主 Agent 上下文直接调用导致 HTML 产物污染上下文，10 轮自审和验证轮次均采用"分工制"（每轮只审查部分维度）。功能设计阶段的功能点信息分散在 index.md 和多个 part-NN.md 中，下游阶段缺乏直观的功能全景视图。

此设计涉及两个阶段（kflow-explore、kflow-prototype-design）的架构调整。

## Goals / Non-Goals

**Goals:**
- 将 huashu-design 调用改为子代理方式，利用 design-prompt.md 文件作为唯一真相源
- 升级原型验证体系：新增导航合理性验证（5 轮）+ Playwright 全覆盖验证（5 轮），均采用"重复制"子代理模式
- 将 explore 和 prototype 的 10 轮自审从"分工制"统一改为"重复制"
- 功能设计阶段新增功能结构树，以树状图 + FP-ID + 优先级 + 简述为下游提供清晰全景

**Non-Goals:**
- 不修改 kflow-design（详细设计）阶段的自审模式（不在本次范围内）
- 不修改 huashu-design Skill 内部实现
- 不新增外部依赖

## Decisions

### 1. 子代理调用架构：方式 A（Agent + prompt 内调用 Skill）

**选择**：使用 `Agent(subagent_type="claude", prompt=...)` 子代理，prompt 指示子代理读取 `prototype/design-prompt.md` 并调用 `Skill("huashu-design")`。

**替代方案**：
- 方式 B（isolation worktree）：创建完整 worktree 隔离执行。**拒绝原因**：prototype 产物在同一变更目录下，无需 worktree 隔离，额外开销不必要。

**设计要点**：
- 主 Agent 在 DESIGN 步骤创建子代理，传入"读取 design-prompt.md → 调用 Skill('huashu-design')"的指令
- 子代理执行 huashu-design 的 placeholder → show → full pass 迭代，在其独立上下文中完成
- 子代理完成后返回结果摘要，主 Agent 读取 `prototype/index.html` 验证产物
- 主 Agent 上下文不被 HTML 产物污染，可干净地进入 VERIFY 步骤

### 2. 提示词文件化：design-prompt.md 作为唯一真相源

**选择**：OPTIMIZE 步骤完成后将完整 prompt 写入 `prototype/design-prompt.md`（7 章节模板），用户确认后 DESIGN 步骤从此文件读取。

**文件结构**（7 章节）：
1. 项目背景与设计目标
2. 设计系统（Design System）：色彩方案、字体系统、间距与圆角
3. 菜单与导航结构
4. 页面详细规格（逐页：布局分区、按钮清单、表单清单、数据展示区、状态覆盖、弹窗/抽屉）
5. 业务流程脚本
6. 硬约束：多文件输出 + flow demo 模式 + 离线自包含
7. 高保真要求：参考资源 + 交互细节规格 + 响应式要求

**替代方案**：
- 在内存中传递 prompt（当前方式）。**拒绝原因**：用户无法回溯审查完整 prompt，子代理方式需要文件传递。

### 3. 重复制 vs 分工制（自审和验证）

**选择**：所有轮次验证/自审统一采用"重复制"——每轮子代理/Agent 执行全部维度检查。

**对比**：

| 维度 | 分工制 | 重复制 |
|------|--------|--------|
| 每轮范围 | 部分维度 | 全部维度 |
| 发现问题 | 后期才暴露其他维度问题 | 早期就暴露各类问题 |
| 收敛性 | 不明显 | 自然收敛（后期问题越来越少） |
| 独立性 | 维度间串行 | 每轮独立完整审查 |

**设计要点**：
- explore SELFREV（10 轮）：每轮全维度（完整性+闭环性+必要性+清晰性）
- prototype SELFREV（10 轮）：每轮全维度（覆盖性+一致性+可用性+完整性）
- prototype 导航验证（5 轮）：每轮全 5 项（可达性+返回按钮+表单链+弹窗+闭环）
- prototype Playwright（5 轮）：每轮全 5 项（可达性+按钮+表单+弹窗+端到端）

### 4. VERIFY 步骤重构

**新的 VERIFY 结构**：

```
6.1 CDN 外部依赖扫描（不变，主 Agent 直接执行）
6.2 多文件交叉引用完整性检查（不变，主 Agent 直接执行）
6.3 导航合理性验证（新增，5 轮子代理串行）
6.4 Playwright 验证（升级，5 轮子代理串行）

执行顺序：6.1 → 6.2 → 6.3(5轮) → 6.4(5轮)
前置依赖：CDN 扫描不通过不继续后续步骤
每轮子代理串行：前一轮子代理完成 → 主 Agent 读取报告 → 主 Agent 修复 → 下一轮
```

**导航验证（6.3）每轮检查项**：
- 页面可达性：BFS 遍历，标记孤立/死胡同页面
- 返回/取消按钮：验证语义父页面合理性
- 表单切换链：多步链条 + 提交成功/失败去向
- 弹窗/抽屉导航：全关闭方式 + 嵌套层级 + 上下文恢复
- 跨页面流程闭环：脚本走通 + 能去能回

**Playwright 验证（6.4）每轮检查项**：
- 页面可达性扫描：BFS 走所有 `<a>` 链接，验证加载成功 + pageerror=0
- 按钮/链接全覆盖点击：穷举点击，验证响应
- 表单全覆盖：空提交校验 + 合法提交 + 取消/重置
- 弹窗/抽屉全覆盖：全方式打开/关闭
- 端到端业务流程：按脚本逐条走通

### 5. 功能结构树

**选择**：在 `functional-designs/index.md` 中以 Markdown 树状图替换现有"三、页面导航结构图"。

**格式规范**：
```
## 三、功能结构树

{应用名称}
├── 📁 {模块1}（{n}个功能点）
│   ├── FP-001 {功能名} [P0] — {一句话功能简述}
│   ├── FP-002 {功能名} [P1] — {一句话功能简述}
│   └── ...
├── 📁 {模块2}（{m}个功能点）
│   └── ...
```

**替代方案**：
- 新增 `index.html` 可视化。**拒绝原因**：增加产物碎片化，Markdown 树状图已足够直观，且与 index.md 保持单一文件。

## Risks / Trade-offs

- **子代理失败风险**：huashu-design 生成失败时子代理可能返回模糊错误 → 主 Agent 在子代理完成后验证 `prototype/index.html` 存在性，缺失时重试或降级
- **5 轮 × 2 验证的时间成本**：共 10 个子代理串行执行，耗时较长 → 每轮发现问题后主 Agent 立即修复，减少后期冗余发现；可通过用户确认跳过高置信度轮次（未来优化）
- **重复制自审收敛慢**：最后几轮可能无新发现 → 这是预期行为，证明覆盖充分；SHALL 完成全部 10 轮，不允许提前终止
- **design-prompt.md 文件大小**：穷举页面元素后文件可能很大 → 文件化是必须的（子代理需读取），文件大比上下文膨胀代价小

## Migration Plan

本次变更属于工作流增强，无需数据迁移。部署步骤：
1. 更新设计文档（`docs/designs/skills/`）
2. 更新技能实现（`.claude/skills/` 的 SKILL.md）
3. 更新核心机制文档（`docs/designs/core-mechanisms.md`）
4. 更新相关 openspec 规格文件
5. 后续变更自动使用新流程
