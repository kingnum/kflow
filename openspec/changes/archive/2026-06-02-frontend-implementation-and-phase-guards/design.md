# Design: 前端实现集中化 + 阶段门控增强

## Context

当前 KFlow 体系在前后端项目中有以下设计缺口：

1. **前端实现的责任归属不明确**：原型阶段产出 HTML 交互原型，编码阶段有 PROTO_CONSTRAINTS 约束注入，但缺少「原型到前端工程代码」的转译环节。前端组件代码应该由谁写、怎么写、何时写，在设计上没有明确归属
2. **Plan 阶段任务模板单一**：所有功能点使用统一 TDD 模板（Red → Green → Refactor），但前端 UI 开发不适合纯 TDD（无法为按钮 hover 态或页面布局写单元测试），导致前端 FP 的任务粒度失配
3. **归档阶段缺少硬门控**：当前 audit 通过后归档可自动触发，用户可能在实际验证前就被带入归档流程
4. **回退路径不完整**：当前回退目标仅 design 和 explore 两种，没有 prototype-design 回退路径。编码阶段发现上游问题时的决策流程也未标准化

## Goals / Non-Goals

**Goals:**
- 明确前端实现的责任归属：集中于一个独立子变更，与后端子变更并行开发
- 前端子变更的执行流程：原型转译（非 TDD），覆盖工程骨架 → 逐页实现 → API 对接 → 状态覆盖
- Plan 阶段为前端 FP 生成专属任务模板，区分于后端 TDD 模板
- Archive 阶段设置硬门控：审计通过后 MUST 用户显式确认，禁止自动流转
- 回退机制补全 prototype-design 路径，新增编码阶段发现上游问题的标准化决策流程

**Non-Goals:**
- 不改变后端 TDD 流程和现有子变更结构
- 不改变原型设计阶段的内部流程
- 不改变 E2E 测试阶段的机制
- 不涉及纯后端项目（纯后端无前端子变更和原型阶段）

## Decisions

### Decision 1: 前端集中为一个独立子变更

**选择**：每个 Change 的前端实现统一由一个独立子变更负责。

**理由**：
- 共享文件冲突不可解：前端代码（路由注册、组件树、状态管理、package.json）是高度耦合的，3 个 Agent 并行写前端必然争抢同一批文件，当前"禁止跨域修改+事后统一合并"规则在前端场景下不可行
- 原型转译是全局性的：原型输出的是一个统一的 HTML 交互原型，转译成工程代码时第一件事是搭建工程骨架（脚手架/路由/布局/组件库/状态管理），这是高度内聚的操作，不可拆分
- 设计系统一致性：前端组件库、设计令牌注入、交互模式需要对全局一致性的把控，单一 Agent 天然保证这一点

**替代方案被拒绝**：按业务垂直切分子变更（每个子变更包含后端+前端）。拒绝理由：共享文件冲突会让并行退化为串行等待，且串行等待的复杂度远高于清晰的依赖关系。

**前端 FP > 10 的例外处理**：当前端子变更超过 10 FP 限制时，先完成「骨架子变更」（路由/布局/组件库/设计令牌），再串行追加「页面组子变更」。骨架已建立路由框架和组件库，后续页面子变更只是在框架内添加页面，不涉及共享文件冲突。

### Decision 2: 前端子变更依赖 API 契约而非后端实现

**选择**：前端子变更的依赖条件是「API 契约在 design 阶段已定义」，而非「后端子变更编码完成」。

**理由**：
- detailed-design.md 中已包含稳定的 API 契约（路径/参数/返回值/错误码），前端可以使用 mock 数据并行启动开发
- 后端编码和前端编码可以真正的并行进行，而非串行等待
- 后端编码完成后的"集成对接"仅是将 mock 替换为真实 API，风险远低于串行开发模式

**具体实现**：
- Plan 阶段：前端子变更的依赖标注为「依赖 API 契约（design 阶段已定义）」，非「依赖后端子变更编码完成」
- Code 阶段前端子流程的 3. API 对接阶段：先用 mock 数据开发，后端编码完成后替换为真实 API

### Decision 3: 前端 FP 使用「原型转译」模板替代 TDD 模板

**选择**：Plan 阶段为前端 FP 生成专属任务模板，结构不同于后端 TDD。

**理由**：
- 前端 UI 开发不适合纯 TDD：你很难给"按钮 hover 态背景加深"或"弹窗出现动画"写单元测试
- 前端开发的验证方式不同于后端：应该是视觉对账（对照原型）、交互验证（Playwright）、而非单元测试
- 前端 FP 的输入源是 prototype/ 目录（HTML + design-tokens.css + element-coverage-tree.md），任务结构应围绕这些输入组织

**前端 FP 任务模板结构**：
```markdown
### 前端功能点 {n}: {页面/组件描述}

#### 输入源
- 原型页面: prototype/{page}.html
- 设计约束: design-tokens.css, element-coverage-tree.md, design-system/MASTER.md
- API 契约: detailed-design.md §{api-section}

#### 实现步骤
- [ ] Step 1: 实现组件骨架（结构 + 样式，对齐原型）
- [ ] Step 2: 注入设计令牌（替换硬编码为 var(--xxx)）
- [ ] Step 3: 实现交互状态（hover/active/focus/loading/empty/error）
- [ ] Step 4: 对接 API（mock 数据 → 真实接口）
- [ ] Step 5: 按原型验证一致性（手动对照 prototype/{page}.html）
```

### Decision 4: Archive 阶段禁止自动流转

**选择**：Archive 是唯一不允许自动进入的阶段，audit 通过后 MUST 用户显式确认。

**理由**：
- 归档是不可逆操作（归档后禁止修改），必须由人做最终确认
- 审计虽通过，但用户可能还需要手动验证（启动服务、走一遍业务流程）
- 当前设计虽有"用户确认归档"，但系统自动调度进入 archive 的阶段入口没有被禁止

**具体实现**：
- 阶段流转规则中增加：archive 阶段不得被任何前置阶段（audit）自动调度进入
- audit 完成后输出摘要 + AskUserQuestion：「审计已通过，是否进入归档阶段？」
- kflow-archive SKILL.md 的触发语义强调用户主动调用

### Decision 5: 回退机制扩展

**选择**：回退触发来源从 4 种扩展为 6 种，回退目标增加 prototype-design，新增编码阶段标准化决策流程。

**6 种回退触发来源**：

| # | 触发来源 | 回退目标 | 说明 |
|---|---------|---------|------|
| 1 | 缺陷修复发现设计错误 | design | 已有 |
| 2 | 代码审查发现结构性问题 | design | 已有 |
| 3 | 用户需求变更 | explore | 已有 |
| 4 | 审计发现 | 按问题归属 | 已有 |
| 5 | 编码发现原型交互/视觉/状态问题 | prototype-design | **新增** |
| 6 | 编码发现功能点/业务规则问题 | explore（功能性）/ design（技术性） | **新增** |

**编码阶段发现上游问题的决策流程**：
```
1. 判断根因归属（原型 / 功能设计 / 技术设计）
2. 记录到 skill-suggestion.md
3. AskUserQuestion 确认决策:
   - 确认回退 → 执行回退流程
   - 暂缓此 FP → 标记 ⏸️ 暂缓，继续其他 FP
   - 记录已知问题 → 不阻止继续
4. 回退执行（如用户确认）
```

**原型回退特殊处理**：进入 prototype-design REVISION 模式（加载已有原型 + 用户修订需求 → 修订 → 验证 → 用户确认 → 回退到 design 继续后续流程），而非完全重新执行原型设计。

## Risks / Trade-offs

- **前端子变更成为瓶颈**: 所有后端子变更完成而前端子变更仍在进行时，E2E 测试无法开始。缓解：前端子变更使用 mock 数据与后端并行启动，不等待后端编码完成
- **前端 FP > 10 时拆分为骨架+页面组**: 增加了子变更间的串行依赖。缓解：骨架只需做一次（脚手架/路由/组件库），后续页面组是纯增量工作，串行依赖成本可接受
- **Archive 手动门控可能拖慢节奏**: 用户需要额外确认步骤。缓解：audit 摘要中包含足够的决策信息（测试结果/覆盖率/审计结论），让用户快速判断
- **回退到 prototype-design 后修订范围不确定**: 原型修订可能引发连锁的 design/plan/code 修订。缓解：prototype-design REVISION 模式已有增量修订机制，非完全重做

## Open Questions

1. 前端子变更的「API mock 方案」需要统一约定吗？（Mock 库选择、mock 数据格式）—— 建议由前端子变更 Agent 按项目现有技术栈自行选择，不纳入强制规范
2. 纯后端项目中如果也需要前端（如 Swagger UI / Admin 面板），是否同样适用前端子变更规则？—— 暂不纳入，纯后端项目继续走 ⏭️ 跳过原型设计的现有路径
