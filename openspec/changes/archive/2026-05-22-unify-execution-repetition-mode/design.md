## Context

当前 KFlow Skills 体系中，创造性阶段（explore/prototype/design）的 SELFREV 已统一采用「重复制」——每轮子代理独立执行全部 4 个维度审查，10 轮自然收敛。而执行类阶段（plan/code/code-review/api-test/e2e-test/integration-test/bug-fix）仍采用「Agent 迭代执行」——子代理按复杂度评估生成的「节奏指引」将 10 轮拆分为"前 N 轮重点执行 → 中间优化 → 后 N 轮验证"的增量推进。

这种人为预设的节奏分配带来了「这部分留给后面轮次」的心理暗示，导致子代理可能在早期轮次中疏忽和遗漏。本次改造将执行类阶段的主执行工作统一改为重复制，消除节奏指引，让每轮全量遍历自然收敛。

## Goals / Non-Goals

**Goals:**
- 7 个执行类阶段的主执行工作统一采用重复制：每轮子代理遍历全部工作项执行完整流程
- 移除节奏指引（前 N 轮重点/中间优化/后 N 轮验证的分段策略）
- 保留复杂度评估公式和分级，但标注为「仅信息展示，不驱动执行行为」
- 保留 Tracer Bullet 机制（编码阶段首轮即打通端到端路径）
- 每阶段在 SKILL.md 和设计规格中明确定义「每轮工作内容」
- core-mechanisms.md §15 改写为重复制模式描述

**Non-Goals:**
- 不修改创造性阶段（explore/prototype/design）的 SELFREV——已使用重复制
- 不修改 10 轮下限数量
- 不修改子代理隔离规则
- 不修改主 Agent 验收流程（轮次检查 + 产物检查 + 覆盖率检查）
- 不修改阶段顺序和门控规则
- 不修改 .status.md 格式和轮次计数器机制

## Decisions

### D1: 重复制在编码阶段的具体语义

**选择**：每轮遍历 tasks.md 中全部功能点，对每个 FP 执行完整 TDD 垂直切片（评估状态 → 写/补测试 → 写/改实现 → 验证通过）。已通过的 FP 执行深化检查（模块深度/边界/重构），未通过的 FP 继续推进。每轮输出 traceability.md「编码实现」列更新。

**备选**：每轮只遍历未完成的功能点（跳过已通过的）
**理由**：仅遍历未完成项会导致已通过项的质量无法在后续轮次中被重新审视。编码过程中后续功能点的实现可能破坏前序功能点（回归），必须每轮全量检查。

### D2: 重复制在测试阶段的具体语义

**选择**：每轮运行全部测试用例（api-tests/ 全部接口 + e2e-tests/ 全部场景），已通过的用例仍需重跑验证无回归，失败的用例记录后触发 bug-fix。

**备选**：仅重跑失败用例
**理由**：不重跑已通过用例将丢失回归检测能力。重复制的核心价值正是每轮全量重跑。

### D3: 复杂度评估保留方式

**选择**：保留现有公式（功能点数 × 1 + 接口数 × 1.5 + 场景数 × 2）和分级阈值（低 <20 / 中 20-50 / 高 >50），但在 prompt 中标注为「仅供参考，不驱动执行行为」。复杂度评估结果仅写入 .status.md 备注列作为背景信息。

**备选**：完全移除复杂度评估
**理由**：用户明确要求保留复杂度评估仅用于信息展示。复杂度分作为理解任务规模的参考指标仍有价值，只是不再强制分配轮次重点。

### D4: Tracer Bullet 在重复制中的定位

**选择**：编码阶段保持 Tracer Bullet 先行——首轮遍历全部 FP 时，对每个 FP 的第一个 RED→GREEN 循环 SHALL 是 Tracer Bullet（端到端最简路径）。后续轮次检查 Tracer Bullet 路径是否仍然畅通（不会被新增代码破坏）。

**备选**：废除 Tracer Bullet（因为重复制每轮全量）
**理由**：Tracer Bullet 的价值在于确保端到端路径在首轮即被打通，避免出现"功能实现完毕但全链路不通"的遗漏。重复制解决的是跨轮次遗漏，Tracer Bullet 解决的是首轮内的遗漏——两者互补。

### D5: 轮次迭代指令变更

**选择**：将现有 prompt 中的迭代指令从：
> 「每完成一轮迭代，更新 .status.md 中执行轮次计数器…若当前轮次无新发现且无可执行工作，仍须递增计数器并继续」

变更为：
> 「每轮遍历全部工作项独立执行完整流程。更新 .status.md 中执行轮次计数器。禁止按轮次分段分配工作重点——每轮均须对全部工作项执行完整检查。必须完成全部 10 轮迭代后才可返回验收报告，禁止在第 10 轮前返回。若当前轮次无新发现且无可执行工作，仍须递增计数器并继续。」

**理由**：新增「禁止按轮次分段分配工作重点」和「每轮均须对全部工作项执行完整检查」两条关键约束，消除节奏指引的心理暗示。

### D6: 每阶段工作内容定义方式

**选择**：在每个阶段的 SKILL.md「执行流程」章节新增「每轮工作内容」子章节，用明确的工作项清单 + 流程图定义每轮全量工作。同时更新对应的设计规格文件。

**备选**：仅在 core-mechanisms.md 中统一描述
**理由**：各阶段工作项差异大（FP vs 接口 vs 场景 vs 代码变更），统一描述会过于抽象。阶段级定义确保子代理 prompt 中的工作内容指令清晰具体。

## Risks / Trade-offs

- **[早期轮次耗时增加]** → 每轮全量遍历比增量推进的单轮工作量大。但 10 轮总数不变，总工作量相近（全量遍历的后期轮次收敛后工作很少）。实际影响在可接受范围内。
- **[子代理可能在前几轮过度用力]** → 重复制搭配 10 轮下限，子代理知道还有 9 轮机会，自然会控制每轮深度。此风险低。
- **[复杂度评估变为死数据]** → 信息展示仍有助于主 Agent 理解任务规模，在验收时判断"覆盖率 100% 但复杂度高时是否应额外关注"等场景仍有参考价值。收益 > 保留成本。
- **[改造范围大（14 个文件）]** → 统一的模式定义在 design.md 中一次定稿，各阶段文件按统一模板修改，降低不一致风险。
