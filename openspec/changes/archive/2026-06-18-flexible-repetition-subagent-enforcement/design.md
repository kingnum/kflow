## Context

当前 KFlow Skills 体系的执行类阶段（plan/code/code-review/api-test/e2e-test/integration-test/bug-fix）统一采用重复制模式，强制10轮下限。这一设计旨在确保充分验证，但在实际执行中暴露出三个问题：

1. **效率浪费**：当问题在前几轮已全部修复时，后续轮次成为"空转"，消耗大量 Token 但无实际产出。
2. **子代理规则执行不力**：kflow-bug-triage 路由到 kflow-bug-fix 时，主 Agent 经常直接执行修复工作而非委托给子代理，原因是规则位置不突出且缺少上下文传递。
3. **回退场景僵化**：阶段回退后重新执行仍需完整10轮，但实际影响范围可能很小。

当前状态文件（.status.md）不追踪阶段执行历史，无法区分"首次执行"和"回退重执行"，也无法传递修订影响范围信息。

## Goals / Non-Goals

**Goals:**

- 移除10轮硬编码下限，改为基于阶段执行历史和影响范围的弹性轮次决策
- 强化子代理执行规则，确保所有入口场景（直接触发/triage路由/其他Skill调用）都正确使用子代理
- 在 triage 诊断时一并评估影响范围，为回退后的重复制轮次决策提供依据
- 建立验证门控机制，确保非严格重复制场景下的质量不降低
- 在状态文件中追踪阶段执行历史和修订信息

**Non-Goals:**

- 不改变创造性阶段（explore/prototype-design/design）的执行模式，这些阶段仍由主 Agent 直连执行
- 不修改重复制的核心遍历逻辑（每轮全量遍历、独立完整流程、自然收敛）
- 不改变子代理隔离规则的核心原则（子代理异常时必须重新创建，主 Agent SHALL NOT 接管）
- 不涉及 kflow-integration-test 的四分法修复循环（该 Skill 有独立的修复机制）

## Decisions

### 决策 1：影响范围评估时机

**决定**：在 kflow-bug-triage 诊断时一并执行影响范围评估。

**理由**：
- triage 已经完成了四层溯源诊断，具备评估影响范围的上下文
- 避免在回退目标阶段重复评估，减少冗余工作
- 影响范围评估结果可直接传递给回退目标阶段，形成闭环

**备选方案**：在回退目标阶段执行前单独评估 → 需要重新读取诊断报告和三纲分析结果，增加复杂度

### 决策 2：影响范围分数计算公式

**决定**：`影响分数 = 功能点数 × 1 + 接口数 × 1.5 + 数据模型变更 × 2`

**理由**：
- 功能点是最基本的影响单位，权重为1
- 接口变更涉及前后端联调，复杂度更高，权重1.5
- 数据模型变更影响范围最广（涉及存储、迁移、API），权重2

**备选方案**：
- 简单计数（不区分类型）→ 无法反映不同类型变更的复杂度差异
- 更细粒度的权重（如区分增删改）→ 增加评估复杂度，收益有限

### 决策 3：轮次-分数映射规则

**决定**：
| 影响范围分数 | 轮次 | 验证策略 |
|-------------|------|---------|
| 1-5 | max(3, ceil(分数)) | 受影响项100% + 全量1轮兜底 |
| 6-15 | max(5, ceil(分数/2)) | 受影响项100% + 全量1轮兜底 |
| >15 | 10 | 标准重复制 |

**理由**：
- 最小3轮确保基本的迭代深度（即使影响范围很小）
- 中等影响范围（6-15分）给予5轮基础，兼顾效率和覆盖
- 大范围影响（>15分）等同首次执行，不缩减轮次

**备选方案**：
- 线性映射（轮次=分数）→ 极端情况下轮次过多
- 固定轮次（如统一5轮）→ 无法根据影响范围灵活调整

### 决策 4：非严格重复制的验证门控

**决定**：受影响项100%覆盖验证 + 全量至少1轮兜底

**理由**：
- 受影响项必须100%验证，确保修订/修复的正确性
- 全量至少1轮兜底，防止遗漏关联影响
- 这一标准在效率和质量之间取得平衡

**备选方案**：
- 仅验证受影响项，无全量兜底 → 风险过高，可能遗漏关联问题
- 每轮都全量验证 → 等同标准重复制，失去弹性优势

### 决策 5：子代理执行模式

**决定**：移除"禁止后台运行"硬规则，改为推荐使用前台模式，允许后台运行但需确保权限已配置。

**理由**：
- 后台运行本身不是问题根源，权限未配置才是
- 权限正确配置后，前后台模式都可以正常工作
- 保留灵活性，长时间执行且无需轮间交互时可使用后台模式

**备选方案**：
- 继续禁止后台运行 → 失去灵活性，长时间执行时用户体验不佳
- 强制后台运行 → 主 Agent 无法实时验收和提取轮间摘要

### 决策 6：权限预配置方式

**决定**：在项目 `.claude/settings.json` 中预设 kflow 所需权限。

**理由**：
- 项目级配置确保所有参与者共享相同的权限设置
- 避免每次执行时都需要用户手动批准
- 权限列表明确，便于审查和维护

**备选方案**：
- 用户级全局配置 → 不同项目可能有不同的权限需求
- 子代理 prompt 中动态申请 → 每次都需要用户批准，体验差

### 决策 7：阶段执行历史存储位置

**决定**：在 `.status.md` 中新增「阶段执行历史」表和「最近修订信息」节。

**理由**：
- `.status.md` 是阶段状态的核心文件，所有 Skill 都会读取
- 与现有的阶段状态表整合，便于统一查看
- 不需要新增文件，减少文件管理复杂度

**备选方案**：
- 独立文件（如 `execution-history.md`）→ 增加文件数量，读取时需要额外加载
- 存储在 traceability.md → 职责不清，traceability 专注于覆盖率追踪

## Risks / Trade-offs

### 风险 1：影响范围评估不准确

**风险**：triage 诊断时评估的影响范围可能不准确，遗漏某些受影响的功能点或接口。

**缓解措施**：
- 验证门控要求全量至少1轮兜底，可捕获遗漏的关联影响
- 如果兜底轮次发现新的受影响项，记录到 skill-suggestion.md 用于改进评估规则
- 保守策略：当影响范围不明确时，默认使用完整10轮

### 风险 2：非严格重复制遗漏问题

**风险**：减少轮次后，某些需要多轮迭代才能发现的问题可能被遗漏。

**缓解措施**：
- 受影响项100%验证确保核心问题被覆盖
- 全量至少1轮兜底提供基本的全局检查
- 后续测试阶段（如 api-test/e2e-test）会再次全量验证，形成多层防线

### 风险 3：权限预配置的安全顾虑

**风险**：在 settings.json 中预设大量权限可能带来安全风险。

**缓解措施**：
- 权限列表仅包含 kflow 执行所必需的工具
- 使用受限的 Bash 命令模式（如 `Bash(npm *)`）而非完全开放
- 定期审查权限列表，移除不再需要的权限
- 用户可以随时修改 settings.json 调整权限

### 风险 4：状态文件复杂度增加

**风险**：新增阶段执行历史和修订信息后，.status.md 文件变得更复杂。

**缓解措施**：
- 新增字段有明确的格式规范，便于解析
- 仅在回退场景下需要读取完整历史，正常流程仅需当前状态
- 提供模板文件，确保格式一致性

### 权衡 1：效率 vs 覆盖度

弹性重复制减少了回退场景的轮次，提升了效率，但可能略微降低覆盖度。通过验证门控（受影响项100% + 全量1轮兜底）来平衡这一权衡。

### 权衡 2：自动化 vs 用户控制

影响范围评估和轮次决策完全自动化，用户无需干预。这提升了自动化程度，但减少了用户对轮次决策的控制。用户仍可通过修改 .status.md 中的影响范围分数来手动调整轮次。
