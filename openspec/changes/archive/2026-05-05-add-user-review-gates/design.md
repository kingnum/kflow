## Context

当前体系在原型设计（可选）和详细设计（必须）两个阶段完成后，阶段直接从「AI 审查通过」流转到下一阶段，缺少用户（人）评审确认环节。需要在不破坏现有门控机制的前提下，在两个设计阶段之后各插入一个用户评审门控。

现有相关机制：
- `kflow-explore` 中已使用 `AskUserQuestion` 确认项目类型，可作为交互模式参考
- `.status.md` 已有「审查记录」表格（记录 AI 四视角审查），用户评审记录可按类似模式扩展
- 门控规则定义在 `core-mechanisms.md` 3.4 节，采用声明式检查列表格式

## Goals / Non-Goals

**Goals:**
- 原型设计完成后，用户必须确认原型才能进入详细设计
- 详细设计（含 AI 四视角审查）完成后，用户必须确认设计才能进入计划阶段
- 用户评审结果持久化在 `.status.md` 中，不依赖会话记忆
- 门控检查自动化——下一阶段启动时自动校验用户评审状态
- 原型设计跳过或纯后端项目时，原型评审门控自动跳过

**Non-Goals:**
- 不改变其他阶段的流转规则（计划、编码、测试等）
- 不改变 AI 四视角审查的流程和内容
- 不引入新的文件格式（复用 .status.md）
- 不在编码/测试等执行阶段增加用户评审

## Decisions

### 1. 评审记录存储在 .status.md 中

**选择**：在 `.status.md` 中新增「用户评审记录」表格。

**备选方案**：单独创建 `.user-review.md` 文件。

**理由**：`.status.md` 已是阶段状态的中心化文件，所有门控检查都读取它。新增表格比新文件更简单，门控检查逻辑改动最小。

### 2. 用户交互使用 AskUserQuestion

**选择**：Skill 在阶段末尾调用 `AskUserQuestion`，提供「确认通过」和「需要修订」两个选项。

**备选方案**：用户手动编辑 .status.md 标记确认。

**理由**：与 `kflow-explore` 中确认项目类型的模式一致，交互明确，结果由 Skill 自动写入 .status.md 避免手动编辑错误。

### 3. 评审状态三值模型

| 状态 | 含义 | 触发 |
|------|------|------|
| `⏳ 待评审` | 阶段产物已输出，等待用户评审 | Skill 输出产物后自动标记 |
| `✅ 已确认` | 用户评审通过 | 用户选择「确认通过」 |
| `⚠️ 需修订` | 用户要求修订 | 用户选择「需要修订」→ 触发阶段回退 |

### 4. 详细设计评审在 AI 审查之后

**选择**：AI 四视角审查通过 → 修复问题 → 展示设计和审查综合报告给用户 → 用户评审。

**理由**：AI 审查作为第一道防线发现低级问题，用户评审关注架构决策和业务对齐。用户看到的是经过 AI 审查并修复后的最终设计，而非初稿。

### 5. 门控规则实现方式

**选择**：在现有门控规则列表中新增检查项，格式与现有一致。

进入详细设计阶段的门控，新增：
```
- 检查: (前后端项目 且 原型设计 ≠ ⏭️ 跳过) 用户评审记录中「原型设计」= ✅ 已确认
- 检查: (纯后端 或 原型设计 = ⏭️ 跳过) 跳过此检查
```

进入计划阶段的门控，新增：
```
- 检查: 用户评审记录中「详细设计」= ✅ 已确认
```

### 6. .status.md 新增表格格式

```markdown
## 用户评审记录

| 阶段 | 评审状态 | 评审时间 | 备注 |
|------|---------|---------|------|
| 原型设计 | ✅ 已确认 | 2026-05-05 14:00 | 原型确认通过 |
| 详细设计 | ⏳ 待评审 | - | - |
```

- 原型设计被跳过时，对应行标记为 `⏭️ 跳过`
- 表格不存在时，门控检查视为所有阶段「未评审」

## Risks / Trade-offs

- **流程变慢**：两个用户评审节点会让流程不再全自动。这是有意为之——关键设计决策需要人把关。如果用户及时响应，每次评审增加的时间可控（几分钟到几十分钟）。
- **用户不在场时阻塞**：如果用户发起流程后离开，Skill 会在 AskUserQuestion 处等待。这是预期行为——设计不应在用户缺席时自动推进到执行阶段。
- **状态文件兼容性**：已有变更的 .status.md 缺少用户评审记录表格。门控检查在表格缺失时应提示用户补评审，而非直接崩溃阻塞。
