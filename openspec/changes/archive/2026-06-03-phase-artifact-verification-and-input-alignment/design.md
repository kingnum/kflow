## Context

当前体系存在四类关联问题：

1. **门控缺口**：8 个已知缺口导致下游阶段无法在入口处阻断"输入缺失"的情况，问题延迟到执行中期才暴露
2. **输入源混淆**：各阶段 Skill 输入表不区分后端/前端子变更，前端编码时可能缺少 prototype/* 文件、后端编码时携带无关原型文件
3. **决策延后**：HITL 机制允许编码阶段出现决策点，本质是设计阶段未完成决策的补丁
4. **验证缺失**：缺少一个独立诊断工具来系统化检测上述问题，目前依赖各阶段自身门控（不完整）和 audit（归档时才发现）

## Goals / Non-Goals

**Goals:**
- 补全 8 个门控缺口，确保所有阶段入口阻断输入缺失
- 标准化后端子变更与前端子变更的输入源，消除编码偏差
- 将 HITL 重新定义为设计阶段标记，阻塞执行阶段入口
- 提供独立诊断 Skill 用于全阶段产物验证和修复路由

**Non-Goals:**
- 不改变已有阶段的执行流程核心逻辑
- 不修改子变更划分算法本身（仅修改 HITL 判定语义）
- 不调整 audit 七维度评估体系（verify 是独立工具）

## Decisions

### D1: 子变更类型严格二分（禁止混合）

**选择**: 子变更类型 = `后端子变更` | `前端子变更`，不允许混合类型。

**理由**: 混合子变更导致输入源不明确，编码时前后端实现耦合，无法确定需要哪些原型产物。

**替代方案**: 允许混合但标注"主要类型"——增加了复杂度，输入源判断仍有歧义。

**约束**: 前端子变更的前端 FP 数 > 10 时，先拆为「骨架子变更」（先行）+「页面组子变更」（串行追加），参见 `frontend-implementation-subchange` spec。

### D2: 前端子变更通过 detailed-design.md 获知接口契约

**选择**: 前端子变更不直接引用 functional-designs/ 中的功能点来推导接口，而是通过 detailed-design.md「子变更划分」章节中声明的「依赖API契约」列表获取。

**理由**: functional-designs/ 定义业务行为，不定义接口契约。detailed-design.md 是接口契约的唯一真实来源。前端子变更编码时需要的 API 信息（方法、路径、入参、出参）来自 detailed-design.md §接口设计。

**格式**:
```markdown
### 子变更: frontend-pages
- **类型**: 前端子变更
- **依赖API契约**:
  - POST /auth/login → detailed-design.md §2.1 接口设计
  - GET /user/profile → detailed-design.md §2.1 接口设计
  - GET /dashboard/stats → detailed-design.md §2.2 接口设计
- **包含FP**: FP-001~FP-005
```

### D3: 前端子变更输入限定为核心原型产物

**选择**: 前端子变更仅接受 prototype/ 下的核心产物作为输入：`index.html`, `design-tokens.css`, `element-coverage-tree.md`。排除过程产物：`design-prompt.md`, `design-system/MASTER.md` 等。

**理由**: 过程产物包含 AIGC 提示词和设计系统参考文件，不包含精确的实现规格。`design-tokens.css` 是精确值（CSS变量定义），`element-coverage-tree.md` 是精确元素清单，`index.html` 是精确页面结构——这三者足以驱动前端实现。

**替代方案**: 保留 `design-system/MASTER.md` 作为参考——但 MASTER.md 是设计系统说明文档，非实现规格。实现应严格对齐 `design-tokens.css` 中的 CSS 变量值，而非 MASTER.md 中的描述。

### D4: HITL 作为设计阶段标记

**选择**: HITL 不再作为执行类型，改为"设计不完整标记"。设计阶段划分子变更时，若存在未决决策，标记为 HITL 并阻塞进入 plan。所有进入 plan 的子变更 MUST 是 AFK。

**AFK 新判定标准**（移除「UI/UX方向决策」条件）:
| 条件 | 说明 |
|------|------|
| 无架构选择决策点 | 所有技术选型在设计文档中已明确 |
| 无设计方案确认需求 | 接口契约、数据模型已完整定义 |
| 所有实现路径已明确 | 详细设计文档覆盖了所有功能点的实现路径 |

原「UI/UX方向决策」条件移除——因为原型设计阶段已有用户评审门控，原型确认后不存在未决 UI 决策。

### D5: kflow-verify 定位

**选择**: kflow-verify 是独立诊断 Skill，非流程阶段。触发时机：手动调用 + 归档前自动触发（integration-test 通过后）。不阻塞阶段流转（诊断报告是信息输出，修复由用户决策驱动）。

**与 kflow-audit 的关系**:
```
kflow-verify (独立诊断)               kflow-audit (流程阶段)
├── 全阶段产物诊断                       ├── 归档门控
├── 输入源完整性检查                     ├── 七维度评分
├── 交叉引用一致性                       ├── 效率统计
├── 修复路由建议                         └── 阻断归档
└── 不影响 .status.md
```

### D6: 门控规则的子变更类型标注

**选择**: 门控规则 §3.4 中对每条规则显式标注 `[后端SC]`/`[前端SC]`/`[全部]` 适用性。新增前端专属门控检查项。

## Risks / Trade-offs

- **[GAP-4~5 强制 prototype/* 存在]** → 前端子变更缺少原型时被阻断。缓解：原型设计阶段可选的逻辑不变，但若跳过原型设计且存在前端子变更，plan 入口需 AskUserQuestion 确认。
- **[HITL 阻塞 plan 入口]** → 设计阶段未关闭决策时子变更卡住。缓解：设计阶段用户评审流程确保决策全部关闭；如有遗漏，kflow-verify 诊断可提前发现。
- **[门控规则膨胀]** → 每条门控增加检查项可能拖慢阶段入口。缓解：检查项均为文件存在性验证（Read/Glob），非内容审查。
- **[子变更严格二分]** → 某些场景可能需要共享前后端逻辑（如类型定义）。缓解：共享类型定义由变更级 agent 在「集成对接」步骤统一管理，不属于子变更域。
