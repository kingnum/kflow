## Context

KFlow Skills 体系已定义了完整的阶段门控机制，但在实际执行中暴露出 6 个实践缺口：

1. **服务启动模糊**：测试阶段的 WAIT_SYNC 步骤依赖 agent 自行推断如何启动服务，未显式绑定 service-guide.md
2. **playwright-cli 误用**：子代理将 playwright-cli 的 snapshot+ref 交互模式误解为需要编写测试脚本
3. **原型设计被跳过**：explore 完成后进入 design 时没有强制询问是否需要原型设计
4. **re-init 无对齐检查**：Skill 体系升级后，未归档变更可能停留在旧阶段列表中
5. **原型→代码一致性缺失**：目前仅在 E2E 阶段用截图对比原型，耗 token 且是事后检测
6. **子代理工具切换**：子代理在模型 API 报错时自行改用 Bash 执行 Playwright 命令，绕过 /playwright-cli 约束

已有 `subagent-isolation-rule` 定义了"主代理不得接管子代理"的机制，但缺少反向约束"子代理不得自作主张换工具"。

## Goals / Non-Goals

**Goals:**
- 服务启动显式绑定 service-guide.md，消除 agent 推断
- playwright-cli 使用模式从"编写脚本"回归到"snapshot+ref 交互"
- 前后端项目在 explore 完成后必须询问是否进入原型设计
- re-init 时输出未归档变更的阶段对齐报告
- 从原型 HTML 自动提取设计令牌/元素清单/导航树，编码+审查阶段 Grep 对账
- 子代理禁止因模型 API 报错自行更换工具，改为阻塞+上报

**Non-Goals:**
- 不修改阶段门控的基本流程（阶段数量、顺序、依赖关系不变）
- 不使用截图对比原型（保留像素级对比方案供后续探索）
- 不自动修改未归档变更的 .status.md（仅输出报告）
- 不要求主代理在子代理异常时接管（已有 subagent-isolation-rule 覆盖）

## Decisions

### D1: 设计令牌来源——从实际原型 HTML 自动提取，不从 design-prompt.md 文本描述提取

**理由**：design-prompt.md 是对设计的文本描述，与原型实际 HTML 之间必然存在信息损耗。从原型 HTML 的 `<style>` 块和 CSS 变量中机械提取的令牌是 100% 准确的。

**替代方案**：从 design-prompt.md 第二章提取——被否决，因为 OPTIMIZE 步骤是人工录入，可能记错/记漏。

### D2: spec 文件生成时机——绑定到"用户确认通过"时点

**理由**：原型会多次迭代（v1→v2→v3），在迭代过程中生成 spec 会导致 spec 与最终原型不一致。只有在用户说"确认通过"后才生成，spec 是最终确认版原型的结构快照。

### D3: 原型一致性验证——清单对账替代截图对比

**理由**：截图对比需要视觉分析，耗 token 且难以自动化。Grep 源码中的元素名/颜色值与 spec 文件对账是纯文本操作，轻量且可重复执行。

**局限**：只能验证"是否存在"和"是否使用了设计令牌"，不能验证"渲染效果是否一致"。这是可接受的权衡——结构正确 + 使用设计令牌 = 80% 的一致性。

### D4: 子代理工具禁令——阻塞+上报，不降级

**理由**：如用户反馈所述，"无论出现什么情况，子代理若无法正常执行，主代理也不见得可以正常执行"。因此子代理遇到工具不可用时，不应降级到其他工具，而应阻塞+上报，由用户决策。

### D5: 原型决策标记——用 .status.md 中的 prototype_decision 字段控制幂等

**理由**：避免重复询问。无论用户走哪条路径（直接 explore、guide 引导），询问逻辑都只触发一次。

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 设计令牌提取不完整——原型中使用了但未在 :root 中定义的样式 | element-spec.md 同时提取内联 style 中的样式值 |
| Grep 对账误报——源码中使用了等价但不同名的值 | code-review 阶段对 Grep 结果做人工确认 |
| huashu-design 不支持自动提取 CSS 变量 | 退化为从 design-prompt.md 第二章提取，标注"来源: 文本描述" |
| prototype_decision 标记残留导致无法重新询问 | 用户可手动删除 .status.md 中的该标记 |
| 子代理阻塞后用户无法快速恢复 | 阻塞时输出明确的恢复指令（安装工具/确认工具可用性） |
