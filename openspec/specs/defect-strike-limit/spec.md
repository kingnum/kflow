## Purpose

定义子变更级缺陷修复的三次尝试上限机制，防止无限自动循环，与集成测试的「三回合架构评估」保持一致。
## Requirements
### Requirement: 子变更修复三次尝试上限

系统 SHALL 在子变更级缺陷修复循环中对同一失败限制最多 3 次自动修复尝试，第 3 次失败后 SHALL 升级到用户决策。

#### Scenario: 首次修复尝试
- **WHEN** 子变更测试失败且为首次尝试修复
- **THEN** 系统分析根因并按二分法分类
- **AND** 采用针对性修复方案

#### Scenario: 第二次修复尝试
- **WHEN** 子变更同一测试用例首次修复后仍然失败
- **THEN** 系统 SHALL 采用替代方案（不同工具或方法）
- **AND** 不得重复首次尝试的相同方案

#### Scenario: 第三次修复尝试
- **WHEN** 子变更同一测试用例第二次修复后仍然失败
- **THEN** 系统 SHALL 重新审视问题假设，考虑更广泛的替代方案
- **AND** 不得重复前两次尝试的方案

#### Scenario: 第三次失败后升级
- **WHEN** 子变更同一测试用例三次修复尝试均失败
- **THEN** 系统 SHALL 停止自动修复循环
- **AND** SHALL 使用 AskUserQuestion 升级用户，提供三个选项：保留当前状态 / 跳过该功能点 / 触发设计阶段回退
- **AND** SHALL 在修复报告中记录三次尝试的方案和失败原因

#### Scenario: 不同测试用例独立计数
- **WHEN** 子变更内有多个测试用例失败
- **THEN** 系统 SHALL 对每个测试用例独立计数尝试次数
- **AND** 一个用例达到上限不影响其他用例的正常修复

#### Scenario: 根因变更重置计数
- **WHEN** 修复过程中根因分类发生变更（如从实现错误变更为设计错误）
- **THEN** 系统 SHALL 重置尝试计数
- **AND** 按新的根因分类重新开始修复流程

## Requirements

### Requirement: 修复尝试上限与迭代轮次的层级关系

系统 SHALL 在缺陷修复阶段并行维护两个独立指标：10轮迭代（外层循环下限）和 3次修复尝试上限（内层单用例上限）。两者独立运作，不互相替代。

#### Scenario: 10轮迭代内嵌3次修复上限

- **WHEN** 缺陷修复阶段 Agent 迭代子代理执行第 N 轮（N ∈ [1, 10]）
- **AND** 某测试用例在第 N 轮中修复失败
- **THEN** Agent 遵循 3次尝试上限规则进行修复（上限）
- **AND** 无论该用例修复结果如何，Agent 继续完成当前轮次其他工作
- **AND** 当前轮次完成后递增轮次计数器，进入第 N+1 轮

#### Scenario: 3次上限触发不终止轮次迭代

- **WHEN** 第 N 轮中某测试用例 3次修复尝试均失败（触发3次上限）
- **THEN** Agent 升级用户决策（保留/跳过/回退）
- **AND** 升级用户不终止当前轮次——Agent 继续处理其他未触发上限的问题
- **AND** 当前轮次完成后继续下一轮
- **AND** 已触发上限的用例在后续轮次中标记为「用户决策中」，不再尝试修复

#### Scenario: 所有用例均触发上限时的处理

- **WHEN** 某轮次中所有失败用例均已触发3次上限
- **THEN** Agent 确认无更多可执行工作
- **AND** 后续剩余轮次递增计数器并标注「所有问题已升级用户决策，无可修复项」
- **AND** 不因无工作而跳过轮次——10轮计数器仍须递增至 10/10

#### Scenario: 10轮下限与3次上限的计数器独立

- **WHEN** Agent 迭代子代理执行缺陷修复
- **THEN** 轮次计数器（执行轮次: N/10）约束整体迭代循环次数（下限）
- **AND** 修复尝试计数（每用例独立）约束单用例修复次数（上限）
- **AND** 两个计数器独立递增，互不干扰
