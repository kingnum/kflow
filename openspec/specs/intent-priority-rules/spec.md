# intent-priority-rules Specification

## Purpose
定义意图识别关键词映射的优先级规则，当多个关键词同时命中时按优先级裁决。包含 triage 路由和修复类上下文判断逻辑。
## Requirements
### Requirement: 关键词意图优先级

系统 SHALL 为意图识别关键词映射定义优先级，当多个关键词同时命中时按优先级裁决。

> **变更说明**：新增 triage 路由关键词组（反馈/报告问题/报bug/提bug → kflow-bug-triage），与修复类关键词同级（优先级 1）。修改"修复"关键词增加上下文判断逻辑：有活跃 bug-fix 子阶段时路由到 bug-fix，否则路由到 triage。原有"问题"关键词从修复类移至 triage 类。

#### Scenario: 优先级排序
- **WHEN** 系统进行意图识别
- **THEN** 关键词优先级从高到低为：
  1. 问题诊断类（反馈、报告问题、报bug、提bug、问题）→ kflow-bug-triage
  2. 修复类（修复、Bug、缺陷）→ 上下文判断（见下方）
  3. 测试类（测试、QA、验收、E2E）
  4. 归档类（归档、完成、结束）
  5. 流程类（流程、指引、帮助、如何）
  6. 创建类（新功能、开始、创建、开发）

#### Scenario: 修复类关键词上下文判断
- **WHEN** 用户输入命中修复类关键词（修复、Bug、缺陷）
- **THEN** 系统 SHALL 检查当前活跃变更是否存在正在执行的 bug-fix 子阶段
- **AND** 若存在 → 路由到 kflow-bug-fix（继续已有修复工作）
- **AND** 若不存在 → 路由到 kflow-bug-triage（新问题诊断）
- **AND** 若无法判断 → 使用 AskUserQuestion 询问用户意图

#### Scenario: 多关键词同时命中
- **WHEN** 用户输入同时命中多个优先级的关键词（如"测试一下新功能"命中优先级3"测试"和优先级6"新功能"）
- **THEN** 系统按最高优先级裁决
- **AND** 路由到优先级最高的匹配 Skill
- **AND** 在指引输出中注明检测到的多意图及裁决依据

#### Scenario: 相同优先级多关键词命中
- **WHEN** 用户输入命中同一优先级内的多个关键词
- **THEN** 系统列出匹配项供用户选择
- **AND** 不使用 AskUserQuestion 阻塞，而是在指引输出中提供可点击选项

#### Scenario: 无关键词命中
- **WHEN** 用户输入无法匹配任何关键词
- **THEN** 系统显示全套流程概览
- **AND** 引导用户选择阶段

### Requirement: 优先级覆盖规则

系统 SHALL 支持特定场景下的优先级覆盖。

#### Scenario: 活跃变更自动路由
- **WHEN** 用户输入为继续类关键词（继续、下一步、接着）且存在活跃变更
- **THEN** 系统忽略其他关键词命中
- **AND** 直接路由到活跃变更的当前阶段

#### Scenario: 明确阶段名称直接匹配
- **WHEN** 用户输入包含明确的 Skill 名称或阶段名称（如"设计探索"、"kflow-design"）
- **THEN** 系统直接路由到对应 Skill
- **AND** 跳过关键词优先级裁决

### Requirement: 指引错误记录增强

系统 SHALL 在记录指引错误时包含优先级相关信息。

#### Scenario: 优先级误判记录
- **WHEN** 用户纠正基于优先级的意图识别结果
- **THEN** 系统在 `docs/skill-suggestion.md` 中记录：
  - 用户原始输入
  - 命中的关键词及各自优先级
  - 做出的路由决策
  - 用户期望的正确路由
- **AND** 标注问题类型为"优先级裁决错误"
