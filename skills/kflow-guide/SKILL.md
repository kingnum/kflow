# 角色

KFlow 流程入口。不做阶段执行，只做意图识别和路由分发。将用户的自然语言输入映射到正确的阶段 Skill，并检测项目上下文（类型、活跃变更、冲突）。正则解析优先（继续/恢复/resume + 变更名 -> RESUME 路由到 kflow-resume），关键词意图识别映射，活跃变更检测，项目类型判断，NEW CHANGE 创建前指引（类型分类+命名建议+冲突预检），定向指引，指引错误记录到 skill-suggestion.md。

# 任务

接收用户输入 -> 变更名正则解析（优先级最高）-> 关键词意图识别 -> 活跃变更检测 -> 项目类型判断 -> NEW CHANGE 指引（条件触发）或 定向指引（条件触发）-> 输出流程概览和下一步建议 -> 指引错误记录（按需）。

# 门控检查

此阶段为独立指引阶段，无前置门控要求。不产生持久化文件（除 skill-suggestion.md 错误日志外）。用于帮助用户确定当前应该使用的 Skill。

# 输入要求

| 产物 | 图例 | 说明 |
|------|------|------|
| 用户自然语言输入 | ✅ 必须 | 描述想要做什么，或"继续/恢复/resume + 变更名" |
| 项目上下文 | 🔶 条件 | 由本 Skill 自动检测，无需用户提供 |

# 输出产物

在对话中输出流程指引信息，不产生持久化文件。

| 产物 | 说明 | 图例 |
|------|------|------|
| 流程指引 | 对话中输出流程概览和下一步建议 | ✅ 必须 |
| 定向指引 | 对话中输出特定变更的定向状态信息 | 🔶 条件 |
| NEW CHANGE 指引 | 对话中输出新变更创建前指引 | 🔶 条件 |
| skill-suggestion.md | 指引错误时追加错误记录 | 🔶 条件 |

# 执行流程

## 总流程

```
INPUT -> PARSE(变更名正则/设计修订检测) -> INTENT(关键词映射)
    -> CHANGES(活跃变更) -> TYPECHECK(项目类型) -> GUIDE(输出) -> ERRORLOG(按需)
```

```
┌─────────────────────────────────────────────────────────────────┐
│                        GUIDE WORKFLOW                            │
├─────────────────────────────────────────────────────────────────┤
│  1. INPUT      接收用户输入                                       │
│  2. PARSE      变更名正则/设计修订解析（优先级最高）              │
│     ├── /继续|恢复|resume\s+([a-z0-9-]+)/ -> RESUME 路由        │
│     ├── /指引|引导|guide\s+([a-z0-9-]+)/  -> 定向指引            │
│     ├── 设计修订触发词 -> DESIGN_REVISION 路由                   │
│     └── 无匹配 -> 进入 INTENT 步骤                               │
│  3. INTENT     意图识别（关键词映射）                             │
│     ├── 新变更意图 -> NEW CHANGE                                  │
│     ├── 识别成功 -> 确定目标 Skill                                │
│     └── 无法识别 -> 显示全套流程概览                              │
│  4. CHANGES    活跃变更检测                                       │
│     ├── 无活跃变更 -> 引导创建新变更（NEW CHANGE 指引）          │
│     ├── 单个活跃变更 -> 自动选择该变更                            │
│     └── 多个活跃变更 -> 列出供用户选择                            │
│  5. TYPECHECK  项目类型判断                                       │
│     ├── 前后端项目 -> 显示 11 阶段流程                            │
│     └── 纯后端项目 -> 显示 9 阶段流程                             │
│  6. GUIDE      输出流程概览和下一步建议                           │
│  7. ERRORLOG   (如有错误) 记录到 skill-suggestion.md              │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤 1：INPUT — 接收用户输入

接收用户的自然语言输入，不做任何预处理，原样传递给后续步骤。

## 步骤 2：PARSE — 变更名正则解析（优先级最高）

> **Plan Mode 绕过**：正则匹配「继续/恢复/resume + 变更名」模式时，SHALL NOT 进入 Plan Mode，直接进入 Skill 路由链。此规则优先级高于 Claude Code Plan Mode 自动触发。详见 CLAUDE.md 强制规则。

在意图识别之前，先用正则从用户输入中提取变更标识名称（kebab-case）：

| 正则模式 | 匹配示例 | 路由分支 |
|---------|---------|---------|
| `/继续\|恢复\|resume\s+([a-z0-9-]+)/` | "继续 add-user-auth" | RESUME 路由 -> 调用 `kflow-resume` |
| `/指引\|引导\|guide\s+([a-z0-9-]+)/` | "指引 add-user-auth" | 定向指引分支（步骤 7） |
| `/(设计需调整\|功能设计需调整\|原型.*调整\|详细设计.*调整\|接口设计.*改\|修改.*设计\|调整.*设计)/` | "功能设计需调整"、"原型需调整"、"详细设计需调整" | DESIGN_REVISION 路由（步骤 2b） |
| 无变更名 "继续/恢复/resume" | "继续"（无变更名） | 按活跃变更数处理（步骤 5） |

**变更名提取规则**：
- 变更名为 kebab-case 格式（小写字母+数字+连字符）
- 提取后验证：检查 `docs/changes/{change-name}/` 目录是否存在
- 变更名不存在时 -> 提示变更不存在并列出活跃变更

### RESUME 路由（正则命中时）

```
RESUME 路由流程:

1. 提取变更名: 正则捕获组 ([a-z0-9-]+)
2. 调用 Skill 工具: Skill(skill="kflow-resume", args="<change-name>")
3. 不额外处理: resume 负责 VERIFY -> STATE -> LOCATE -> GATE -> SUMMARIZE -> DISPATCH
```

### 无变更名 RESUME 处理

| 活跃变更数 | 处理 |
|-----------|------|
| 0 个 | 提示没有可恢复的变更，引导创建新变更 |
| 1 个 | 自动选择该变更，调用 `kflow-resume` |
| >1 个 | 列出所有活跃变更供用户选择，选择后调用 `kflow-resume` |

### DESIGN_REVISION 路由（设计修订正则命中时）

```
DESIGN_REVISION 路由流程:

1. 解析用户输入，关键词→目标设计目录映射:
   ├── 含"功能"/"需求" → functional-designs
   ├── 含"原型"/"UI"/"交互" → prototype
   └── 含"详细"/"接口"/"架构" → detailed-design

2. ROUTE 分流到目标设计 Skill（REVISION 模式）:
   ├── functional-designs → Skill(skill="kflow-explore", args="REVISION")
   ├── prototype → Skill(skill="kflow-prototype-design", args="REVISION")
   └── detailed-design → Skill(skill="kflow-design", args="REVISION")

3. 目标 Skill 完成后 → 进入「设计修订后处理流程」（§设计修订后处理）
```

## 步骤 3：INTENT — 意图识别关键词映射

当正则未命中时，按关键词优先级映射：

### 关键词映射表

| 关键词 | 识别意图 | 推荐 Skill | 优先级 |
|--------|---------|-----------|--------|
| 反馈、报告问题、报bug、提bug、问题 | 问题诊断（triage） | `kflow-bug-triage` | 1 |
| 状态、进度、查看、总结 | 状态查看 | `kflow-status` | 1 |
| 流程、指引、帮助、如何 | 流程概览 | 显示完整流程 | 1 |
| 修复、Bug、缺陷 | 缺陷修复（上下文判断） | 见下方上下文判断 | 1 |
| 继续、下一步、接着 | 继续活跃变更 | 继续当前阶段 | 2 |
| 初始化、环境配置、工具推荐、设置 | 环境初始化 | `kflow-init` | 2 |
| 新功能、开始、创建、开发 | 新变更启动 | `kflow-explore` | 3 |
| 原型设计、UI 设计、交互设计 | 原型设计 | `kflow-prototype-design` | 3 |
| 测试、QA、验收、E2E | 测试阶段 | `kflow-e2e-test`（前后端项目） | 4 |
| 详细设计、技术设计、架构设计、设计审查 | 详细设计 | `kflow-design` | 4 |
| 任务计划、任务清单、实现计划 | 计划阶段 | `kflow-plan` | 4 |
| 代码审查、审查代码、review | 代码审查 | `kflow-code-review` | 5 |
| 编码实现、TDD、功能实现 | 编码阶段 | `kflow-code` | 5 |
| 审计、评估、检查、审核 | 审计阶段 | `kflow-audit` | 5 |
| 诊断、验证产物、检查产物、产物完整性、输入源检查、verify | 产物诊断 | `kflow-verify` | 5 |
| 集成测试、跨子变更测试 | 集成测试 | `kflow-integration-test` | 6 |
| 归档、完成、结束 | 归档阶段 | `kflow-archive` | 6 |
| 设计需调整、功能设计需调整、原型需调整、详细设计需调整、接口设计要改、修改设计、调整设计 | 设计修订 | DESIGN_REVISION 路由（步骤 2b） | 0 |

### 修复类关键词上下文判断

当用户输入命中修复类关键词（修复、Bug、缺陷）时：
- 存在活跃 bug-fix 子阶段 → 路由到 `kflow-bug-fix`（继续已有修复）
- 不存在活跃 bug-fix → 路由到 `kflow-bug-triage`（新问题诊断）
- 无法判断 → 使用 AskUserQuestion 询问用户意图

### 变更类型分类关键词（NEW CHANGE 分支专用）

| 关键词 | 变更类型 | 后续流程 |
|--------|---------|---------|
| 添加、新增、实现、开发、做、增加 | 功能需求 | 标准流程 |
| 修复、Bug、问题、报错、异常、修 | 功能缺陷 | 简化流程 |
| 搭建、创建平台、系统、完整、产品 | 产品需求 | 完整流程（可能需要多变更） |

### 意图识别优先级规则

```
优先级判定规则:

0. 正则解析优先（最高优先级）:
   ├── 输入匹配「继续/恢复/resume + 变更名」-> RESUME 路由到 kflow-resume
   ├── 输入匹配「指引/引导/guide + 变更名」-> 定向指引分支
   └── 正则未命中 -> 进入关键词匹配

1. 按上表中各匹配关键词的最高优先级值选择
2. 同优先级时:
   ├── 检查活跃变更当前阶段，优先选择符合当前流程方向的关键词
   ├── 无法从上下文判断时，使用 AskUserQuestion 确认
   └── 用户未明确纠正的记录到 skill-suggestion.md

3. 特殊排除规则:
   ├── 关键词"测试"在有活跃变更且当前阶段为编码/代码审查时 -> 不触发
   ├── 关键词"修复/Bug/缺陷"在有活跃变更且当前阶段为集成测试时 -> 引导到 kflow-integration-test
   ├── 关键词"修复/Bug/缺陷"-> 执行上下文判断（有活跃 bug-fix -> bug-fix，否则 -> triage）
   └── 关键词"归档"在有活跃变更且未全部完成时 -> 提示先完成前置阶段

4. 歧义处理:
   ├── 用户明确指代某 Skill 名称时直接触发该 Skill
   └── 上下文无法确定时 -> 显示匹配的选项供用户选择（最多3项）

5. RESUME 无变更名处理:
   ├── 0 个活跃变更 -> 引导创建新变更
   ├── 1 个活跃变更 -> 自动选择该变更，调用 kflow-resume
   └── >1 个活跃变更 -> 列出所有活跃变更供用户选择

6. NEW CHANGE vs 活跃变更裁决:
   ├── 用户表达新变更意图 + 有活跃变更 -> 先确认是否创建新变更（不同于继续）
   └── 用户未明确 -> AskUserQuestion 确认
```

## 步骤 4：CHANGES — 活跃变更检测

```
活跃变更检测:

┌─────────────────────────────────────────────────────────────────┐
│  1. SCAN     扫描 docs/changes/ 目录                             │
│  2. FILTER   过滤未归档变更                                       │
│     └── 排除 docs/archive/ 目录下的变更                          │
│  3. READ     读取各变更的 .status.md                              │
│  4. COUNT    统计活跃变更数量                                     │
│     ├── 0 个 -> 提示创建新变更（进入步骤 6 NEW CHANGE 指引）     │
│     ├── 1 个 -> 自动选择该变更                                   │
│     └── > 1 个 -> 列出变更供用户选择                             │
└─────────────────────────────────────────────────────────────────┘
```

NEW CHANGE vs 活跃变更裁决：
- 用户表达新变更意图 + 有活跃变更 -> 先确认是否创建新变更（不同于继续）
- 用户未明确 -> AskUserQuestion 确认

## 步骤 5：TYPECHECK — 项目类型判断

| 检测标识 | 判断为 |
|---------|--------|
| package.json 前端框架依赖（react, vue, angular, next, nuxt, svelte） | 前后端项目 |
| 前端源文件（.vue, .tsx, .jsx, .svelte） | 前后端项目 |
| 前端构建配置（vite.config, webpack.config） | 前后端项目 |
| 以上均不存在 | 纯后端项目 |

| 项目类型 | 流程阶段 | 可选阶段 |
|---------|---------|---------|
| 前后端项目 | 11 阶段 | 原型设计可选 |
| 纯后端项目 | 9 阶段 | 无可选阶段（跳过原型设计和浏览器自动化测试） |

## 步骤 6：NEW CHANGE 创建前指引（条件触发）

当用户表达新变更意图（新功能/修复Bug/搭建平台）且不指定变更名时，提供创建前指引，不创建任何文件：

```
NEW CHANGE 指引流程:

┌─────────────────────────────────────────────────────────────────┐
│                 NEW CHANGE GUIDANCE WORKFLOW                     │
├─────────────────────────────────────────────────────────────────┤
│  1. CLASSIFY   变更类型分类                                       │
│     ├── 产品需求: 搭建/创建平台/系统/完整/产品                    │
│     ├── 功能需求: 添加/新增/实现/开发/做/增加                     │
│     └── 功能缺陷: 修复/Bug/问题/报错/异常/修                      │
│  2. DETECT     检测项目类型                                       │
│     ├── 前后端项目（存在前端框架依赖/前端源文件）                 │
│     └── 纯后端项目（无前端工程）                                  │
│  3. SUGGEST    建议变更名称                                      │
│     ├── 功能需求: add-{feature-name} (kebab-case)                 │
│     ├── 功能缺陷: fix-{issue-name} (kebab-case)                   │
│     └── 产品需求: {platform-name} (kebab-case)                    │
│  4. CONFLICT   跨变更冲突预检                                     │
│     ├── 扫描 docs/changes/ 下活跃变更的功能点                    │
│     ├── 与新变更功能意图做语义比对                                │
│     └── 有重叠 -> 提示可能冲突 + 列出冲突变更                    │
│  5. OUTPUT     输出创建前指引                                     │
│      ├── 变更类型、建议名称                                      │
│      ├── 项目类型、适用流程阶段                                  │
│      ├── 冲突预检结果                                            │
│      └── 下一步建议: 使用 kflow-explore 开始设计探索             │
└─────────────────────────────────────────────────────────────────┘
```

### NEW CHANGE 指引输出格式

```markdown
## 新变更指引

### 变更类型
**{产品需求/功能需求/功能缺陷}**

### 建议变更名称
`{suggested-change-name}`

### 项目类型
{前后端项目/纯后端项目} -> {N} 阶段流程

### 适用流程阶段
{根据项目类型和变更类型的阶段列表}

### 跨变更冲突预检
{无冲突 / ⚠️ 可能存在冲突: 列出冲突变更和功能点}

### 下一步建议
请使用 `kflow-explore` 开始设计探索。变更初始化将在 explore 阶段完成。
```

## 步骤 7：定向指引（条件触发）

当正则解析识别到「指引/引导/guide + 变更名」模式时，输出针对该变更的定向状态信息：

```
定向指引流程:

┌─────────────────────────────────────────────────────────────────┐
│                 TARGETED GUIDANCE WORKFLOW                       │
├─────────────────────────────────────────────────────────────────┤
│  1. VERIFY    检查 docs/changes/{change}/ 目录存在               │
│     └── 不存在 -> 提示变更不存在 + 列出活跃变更                  │
│  2. READ      读取变更状态文件                                   │
│     ├── .status.md（变更基本信息和阶段状态）                      │
│     └── subchanges/*/.status.md（子变更进度矩阵）                │
│  3. OUTPUT    输出定向指引                                       │
│      ├── 变更基本信息（描述、类型、项目类型、当前阶段）           │
│      ├── 流程位置图（✅/🔄/⏳/⏭️/❌/⚠️ 标注各阶段）            │
│      ├── 子变更进度矩阵（如有）                                  │
│      └── 可用操作建议（当前应使用的 Skill）                      │
└─────────────────────────────────────────────────────────────────┘
```

### 定向指引输出格式

```markdown
## 定向指引：{change-name}

### 变更信息
| 字段 | 值 |
|------|-----|
| 描述 | {变更描述} |
| 类型 | {变更类型} |
| 项目类型 | {项目类型} |
| 当前阶段 | {阶段名称} |

### 流程位置
{各阶段 ✅/🔄/⏳/⏭️ 标注}

### 子变更进度
{如有子变更，显示进度矩阵}

### 建议操作
请使用 `{recommended-skill}` {操作说明}
```

## 步骤 8：GUIDE — 流程概览输出

```markdown
## KFlow 流程概览

### 项目类型: {前后端项目|纯后端项目}

### 阶段流程
{根据项目类型显示对应流程图}

### 各阶段说明
| 阶段 | Skill | 核心功能 |
|------|-------|---------|
| 设计探索 | kflow-explore | 需求澄清、功能点拆分、项目类型检测 |
| 原型设计 | kflow-prototype-design | HTML 原型设计（前后端项目可选） |
```

# 与其他 Skill 的关系

- **独立阶段**：可随时调用，不依赖其他阶段
- **引导进入**：根据意图识别结果引导进入对应 Skill
- **路由到 `kflow-resume`**：识别 RESUME 模式后直接调用 `kflow-resume` 进行中断恢复
- **路由到 `kflow-bug-triage`**：用户反馈关键词（反馈/报告问题/报bug/提bug/问题）匹配时路由到 triage
- **路由到 `kflow-bug-fix`**：修复类关键词 + 存在活跃 bug-fix 子阶段时路由继续已有修复
- **NEW CHANGE 指引 -> `kflow-explore`**：新变更指引输出后建议用户使用 `kflow-explore` 开始设计探索
- **定向指引**：识别「指引/引导/guide + 变更名」模式时输出定向状态信息
- **无前置阶段**：无需门控检查
- **不创建文件**：仅输出指引信息（除 skill-suggestion.md 错误记录）

# 核心提醒

- 正则解析优先级最高，在关键词匹配之前执行
- 意图识别按优先级值选择，同优先级结合活跃变更上下文判断
- 无变更名 RESUME 时：0 活跃->引导新变更，1 活跃->自动 resume，>1 活跃->列出供选择
- 指引错误必须记录到 skill-suggestion.md，便于后续优化关键词映射
- 本 Skill 不创建变更目录和文件，变更初始化在 kflow-explore 阶段完成
- 输出时根据项目类型显示对应流程（前后端 11 阶段 / 纯后端 9 阶段）
- 跨变更冲突预检通过语义比对，非精确匹配
- NEW CHANGE 指引不创建任何文件，仅给出建议和冲突预警
