# 角色

设计探索入口。所有变更的起点，负责需求澄清、项目类型检测（前后端 vs 纯后端）、功能点原子级拆分、领域词汇表（CONTEXT.md）管理。**不在此阶段划分子变更**——子变更划分后置到详细设计阶段完成后基于完整设计认知执行。

# 任务

```
接收用户需求
  → 创建变更目录和状态文件
  → 探索项目上下文
  → 检测项目类型 + 用户确认
  → 判断变更粒度（产品级/功能需求级/功能缺陷级）
  → 需求澄清 + CONTEXT.md 管理（首次构建或增补 + 对齐检查）
  → 功能点原子级拆分 + 关联关系分析
  → 输出 functional-designs/（index.md + part-NN.md 目录化结构）
  → 10 轮自循环审查（完整性/闭环性/必要性/清晰性）
  → 原型设计决策门控（AskUserQuestion，仅前后端项目，prototype_decision 标记幂等）
  → 标记阶段完成
```

# 门控检查

此阶段为**入口阶段**，无前置门控要求。需自建变更目录和状态文件。

输出产物自检：
- `functional-designs/index.md` 必须存在，含版本号（格式：`主版本.次版本.修订号`）和统一修订记录表（合并原需求变更记录与修订记录，含修订类型枚举列）、项目类型字段
- `functional-designs/index.md` **不再包含**子变更划分方案（子变更划分后置到 design 阶段）
- `functional-designs/` 至少包含一个 `part-NN.md` 分册文件
- `.status.md` 必须存在并标记设计探索阶段为 ✅ 完成
- `self-reviews/explore/` 目录下必须存在 10 个自审报告文件

# 输入要求

| 产物 | 图例 | 说明 |
|------|------|------|
| 用户需求描述 | ✅ 必须 | 产品功能、功能需求、功能缺陷 |
| 已有需求文档/缺陷描述 | 🔶 条件 | 如存在则引用，不存在时基于用户口头描述 |

> **图例说明**：✅ 必须 ＝ 不可或缺的前置输入；🔶 条件 ＝ 满足特定条件时需要；⏭️ 跳过 ＝ 被跳过的阶段，产物不存在。

# 输出产物

| 产物 | 文件 | 图例 | 内容要求 |
|------|------|------|---------|
| 功能设计文档 | `docs/changes/{change}/functional-designs/` | ✅ 必须 | 需求描述、项目类型、功能点清单（每功能点含：用户故事、所属页面与菜单、可执行操作、表单项定义、业务规则、业务流程上下文）、功能点关联关系、功能结构树（模块→功能点树状图，含FP-ID/优先级/简述）、核心业务流程图、变更类型判断、版本号+统一修订记录表（合并原需求变更记录与修订记录，格式：版本/日期/修订类型/修订内容/影响功能点/触发阶段） |
| 领域词汇表 | `CONTEXT.md`（项目根目录） | ✅ 必须 | 项目级领域术语定义，首次构建或增补。每术语含：定义、别名、边界 |
| 状态文件 | `docs/changes/{change}/.status.md` | ✅ 必须 | 标记设计探索阶段完成，含项目类型字段、执行备注 |
| 自审报告 | `docs/changes/{change}/self-reviews/explore/` | ✅ 必须 | 10 轮自审报告，文件名格式：`{YYYYMMDD}-{HHMMSS}.md` |

# 执行流程

## 总流程

```
设计探索阶段流程:

┌──────────────────────────────────────────────────────────────────┐
│                      EXPLORE WORKFLOW                             │
├──────────────────────────────────────────────────────────────────┤
│  1. INIT      → 创建变更目录和状态文件                             │
│  2. CONTEXT   → 探索项目上下文（文件、文档、提交记录）              │
│  3. TYPEDET   → 检测项目类型（前后端 vs 纯后端）                  │
│  │   ├── 检测 package.json 前端框架依赖                          │
│  │   ├── 检测 .vue/.tsx/.jsx 等前端源文件                       │
│  │   ├── 检测 vite.config/webpack.config 构建配置               │
│  │   └── 判断结果 → 前后端项目 或 纯后端项目                     │
│  4. CONFIRM   → 用户确认项目类型（AskUserQuestion）              │
│  5. CLASSIFY  → 判断变更粒度（产品级/功能需求级/功能缺陷级）       │
│  6. CLARIFY   → 澄清需求 + 收集用户意图 + CONTEXT.md 管理        │
│  │   ├── 检测 CONTEXT.md 是否存在                                │
│  │   ├── 不存在 → 首次构建（从需求描述提炼领域术语）             │
│  │   ├── 已存在 → 增补检查（新术语/模糊术语对齐）                │
│  │   └── 对齐检查（功能点术语 vs CONTEXT.md 定义）               │
│  7. SPLIT     → 拆分功能点到原子级                                │
│  8. RELATE    → 分析功能点关联关系                                │
│  9. OUTPUT    → 输出功能设计文档（含版本号+修订记录区+功能结构树）│
│  10. SELFREV  → 10 轮自循环审查（子代理串行 + 重复制）           │
│  │   每轮: 启动独立 Agent(subagent) 子代理                       │
│  │   子代理全四维度独立检查（完整性/闭环性/必要性/清晰性）         │
│  │   子代理发现问题 → 直接修复 + 生成自审报告                     │
│  │   主 Agent 读报告 + 确认修复 → 启动下一轮子代理                │
│  │   报告路径: self-reviews/explore/{YYYYMMDD}-{HHMMSS}.md       │
│  │   串行执行 10 轮，不可提前终止                                 │
│  11. PROTO_GATE → 原型设计决策门控（仅前后端项目）             │
│  │   ├── 检查 .status.md 中是否存在 prototype_decision 标记     │
│  │   ├── 不存在 → AskUserQuestion: "检测到 {n} 个 UI 功能点，   │
│  │   │   是否进入原型设计阶段？"                                │
│  │   │   ├── 确认创建原型 → prototype_decision=已选择原型设计    │
│  │   │   └── 跳过原型设计 → prototype_decision=已跳过           │
│  │   └── 已存在 → 跳过询问，直接进入下一阶段                    │
│  12. COMPLETE → 更新状态文件，标记阶段完成                        │
└──────────────────────────────────────────────────────────────────┘
```

## 步骤 1：INIT — 创建变更目录和状态文件

```
1. 确定变更名称（kebab-case 格式）:
   ├── 功能需求: add-{feature-name}
   ├── 功能缺陷: fix-{issue-name}
   └── 产品需求: {platform-name}
2. 创建目录 docs/changes/{change}/:
   ├── functional-designs/ 目录
   └── self-reviews/explore/ 目录
3. 创建 docs/changes/{change}/.status.md（变更级状态文件）:
   └── 填写基本信息: 创建时间、变更类型（初始标记）、项目类型（待检测）、
       变更描述、当前阶段=设计探索
```

## 步骤 2：CONTEXT — 探索项目上下文

使用 Glob/Grep/Read 快速了解项目概况：

```
1. 项目结构和关键目录
2. package.json 中的框架依赖
3. 已有文档（docs/prototype/、CONTEXT.md 的存在性）
4. 活跃变更情况（docs/changes/ 下的现有变更）
```

## 步骤 3：TYPEDET — 检测项目类型

| 检测标识 | 存在时判断为 |
|---------|------------|
| package.json 前端框架依赖（react, vue, angular, next, nuxt, svelte） | 前后端项目 |
| 前端源文件（.vue, .tsx, .jsx, .svelte, .angular） | 前后端项目 |
| 前端构建配置（vite.config.*, webpack.config.*, tsconfig.json） | 前后端项目 |
| 以上均不存在 | 纯后端项目 |

## 步骤 4：CONFIRM — 用户确认项目类型

使用 AskUserQuestion 展示检测结果：
- 显示检测到的项目类型（前后端项目 / 纯后端项目）
- 用户可确认或手动修正项目类型
- 修正结果记录到 `functional-designs/index.md`

### 项目类型对后续阶段的影响

| 项目类型 | 原型设计阶段 | E2E 测试阶段 | 集成测试阶段 |
|---------|------------|------------|------------|
| 前后端项目 | 可选 | 必须 | 必须 |
| 纯后端项目 | ⏭️ 跳过 | ⏭️ 跳过 | 必须 |

## 步骤 5：CLASSIFY — 判断变更粒度

根据用户输入关键词自动判断，并按粒度调整流程深度：

| 粒度 | 关键词特征 | 流程深度 | 产出要求 |
|------|-----------|---------|---------|
| **产品级** | "搭建/创建平台/系统/完整/产品" | 完整流程 | 需求描述 + 项目类型 + 功能点清单 + 技术方案建议 |
| **功能需求级** | "添加/新增/实现/做/增加" | 标准流程 | 需求描述 + 项目类型 + 功能点清单 + 关联关系 |
| **功能缺陷级** | "修复/Bug/问题/报错/异常" | 简化流程 | 缺陷描述 + 项目类型 + 影响分析 + 修复范围 |

## 步骤 6：CLARIFY — 需求澄清 + CONTEXT.md 管理

### 6a. 领域词汇表管理

`CONTEXT.md` 是项目级领域词汇表，作为所有后续阶段术语引用的**唯一真实来源**。

```
1. 检测 CONTEXT.md 是否存在
2. 不存在 → 首次构建:
   ├── 从用户需求描述中提取关键领域概念
   ├── 为每个概念编写: 定义、别名、边界
   ├── 按字母顺序排列术语
   └── 写入 CONTEXT.md（含版本号和修订记录）
3. 已存在 → 增补检查:
   ├── 检查新出现的术语是否已在词汇表中
   ├── 新术语 → 按首次构建规则添加到 CONTEXT.md
   ├── 用户使用模糊/过载术语 → 提议精确术语并写入
   └── 更新版本号（次版本 +1）和修订记录
4. 对齐检查（SHALL 执行）:
   ├── 检查功能点描述中使用的术语是否与 CONTEXT.md 定义一致
   ├── 功能设计文档中的实体名称 SHALL 使用 CONTEXT.md 中的术语
   └── 发现不一致时 → 更新功能设计文档或更新 CONTEXT.md 定义
```

#### CONTEXT.md 文件格式

详见 [references/context-template.md](references/context-template.md) — 文件格式模板、术语提炼规则（定义精确性/别名必查/边界明确/过载术语处理/禁止实现耦合）、消费链。

### 6b. 需求澄清

使用 AskUserQuestion 澄清模糊需求。输出功能设计时覆盖以下域内内容：

- **页面结构/菜单层级**：用户可见的页面导航结构
- **可执行操作**：每个页面的具体操作、触发方式、可见条件
- **表单项定义**：字段名、字段类型、校验规则、默认值
- **业务规则**：前置条件、校验规则、触发条件、后置结果
- **业务流程**：用户视角的完整业务流程闭环（从触发点到最终结果）

## 步骤 7：SPLIT — 拆分功能点到原子级 + 标记 FP 类型

### 7a. 原子级判定标准

| 粒度 | 判断标准 | 示例 |
|------|---------|------|
| 过粗 | 包含多个独立行为 | "用户管理功能" ❌ |
| 合适 | 单一明确行为 | "用户注册"、"用户登录" ✅ |
| 过细 | 实现细节级别 | "添加注册按钮" ❌ |
| 无法归类（跨层） | 同时涉及后端和前端判定标准 | "实现注册功能（含 API + 页面）" ❌ → 必须拆分 |

### 7b. FP 类型判定

每个功能点在拆分完成后 SHALL 标记为「后端」或「前端」：

| 类型 | 判定标准 | 示例 |
|------|---------|------|
| **后端** | API 实现、服务逻辑、数据模型、数据库操作、业务规则引擎、外部服务集成 | "实现注册 API"、"实现用户表迁移" |
| **前端** | 页面组件、UI 组件、路由配置、状态管理、样式实现、CSS 变量注入 | "实现注册表单页面"、"实现导航守卫" |

### 7c. 无法归类则强制拆分

| 步骤 | 操作 |
|------|------|
| 检测 | 当 FP 同时涉及后端和前端判定标准 |
| 判定 | 粒度过粗，MUST 继续拆分 |
| 拆分 | 拆为独立后端 FP + 前端 FP |
| 禁止 | SHALL NOT 以混合形式写入 functional-designs/ |

### 7d. 拆分后关联标注

跨层拆分后的后端 FP 和前端 FP SHALL 在「关联功能点」列相互引用，前端 FP 描述中标注「关联后端 FP: {FP-ID}」。

### 功能点数量与后续处理

> **重要**：子变更划分不在本阶段执行——在详细设计阶段完成后基于完整设计认知划分。

| 条件 | 处理方式 |
|------|---------|
| 功能点 ≤ 10 | 详细设计阶段确认后，创建单一子变更 |
| 功能点 > 10 且 ≤ 20 | 详细设计阶段完成后拆分为多个子变更 |
| 功能点总数 > 20 | 提示用户拆分为多个变更 |

## 步骤 8：RELATE — 分析功能点关联关系

| 关系类型 | 定义 | 标注格式 |
|----------|------|---------|
| 前置依赖 | A 必须在 B 前完成 | `A -> B` |
| 后置依赖 | B 完成后才可执行 A | `B <- A` |
| 并行关系 | A 和 B 可同时实现 | `A \|\| B` |
| 包含关系 | A 是 B 的子功能 | `B ⊃ A` |

## 步骤 9：OUTPUT — 输出功能设计文档

### FP 清单表格格式

`functional-designs/index.md` 和 `part-NN.md` 中 FP 清单 SHALL 包含「类型」列：

**index.md**：功能点ID | 名称 | 简述 | 类型 | 优先级 | 关联功能点 | 依赖功能点 | 所在分册

**part-NN.md**：功能点ID | 名称 | 类型 | 优先级 | 关联功能点 | 依赖功能点

统计信息 SHALL 增加：后端 FP 数量、前端 FP 数量、跨层关联 FP 对

### 输出目录结构

采用 **index.md + part-NN.md 目录化结构**：

- 条目 ≤ 30：`index.md` + 一个 `part-01.md`
- 条目 > 30：`index.md` + N 个 `part-NN.md`（每个分册 ≤ 30 条目）

### 每个功能点必须包含

| 内容 | 说明 |
|------|------|
| 用户故事 | 以用户视角描述功能价值 |
| 所属页面与菜单 | 功能所在的页面路径和菜单层级 |
| 可执行操作 | 触发方式（点击/输入/选择等）+ 可见条件 |
| 表单项定义 | 字段名、字段类型、校验规则、默认值 |
| 关联配置项 | 配置项名称、关联功能点ID、关联类型、配置值变化→功能行为变化描述 |
| 业务规则 | 前置条件、校验规则、触发条件、后置结果 |
| 业务流程上下文 | 该功能点在整体用户流程中的位置 |

### version 号与修订记录

`functional-designs/index.md` 头部包含：

```markdown
> **版本**: 1.0.0
> **创建时间**: {YYYY-MM-DD}
> **修订记录**:
> | 版本 | 日期 | 修订内容 | 触发阶段 |
> |------|------|---------|---------|
> | 1.0.0 | {YYYY-MM-DD} | 初始版本 | 设计探索 |
```

版本号格式：`主版本.次版本.修订号`（如 1.0.0 → 1.1.0 → 1.1.1 → 2.0.0）。

### 需求变更记录表

`functional-designs/index.md` 必须包含需求变更记录表：

```markdown
## 需求变更记录

| 序号 | 时间 | 变更描述 | 影响功能点 | 触发阶段 | 处理状态 |
|------|------|---------|-----------|---------|---------|
| 1 | 2026-05-03 14:00 | 新增双因素认证 | FP-005 | 详细设计 | 已修订 |
| 2 | 2026-05-04 10:00 | 支付接口从 A 改为 B | FP-012 | 编码 | 处理中 |
```

### 配置项影响矩阵

`functional-designs/index.md` 必须包含配置项影响矩阵，建立功能点与配置项的交叉关联视图：

```markdown
## 配置项影响矩阵

| 配置项名称 | 类型 | 默认值 | 受影响功能点ID列表 | 影响类型汇总 |
|-----------|------|--------|-------------------|-------------|
| {CONFIG_KEY} | {string/number/boolean} | {默认值} | FP-001, FP-003, FP-007 | {行为变更/流程分支/功能开关/...} |
```

> 配置项影响矩阵用于在功能设计阶段识别配置变更的波及范围，为后续详细设计、测试用例设计提供输入。每行记录一个配置项及其影响的全部功能点。

## 步骤 10：SELFREV — 10 轮自循环审查（子代理串行 + 重复制）

### 子代理上下文文件加载（基础层 + 创意层）

自审子代理 prompt 中 SHALL 包含以下 kflow-shared 文件：

- skills/kflow-explore/references/state-values.md（摘要）
- skills/kflow-explore/references/gates.md（当前阶段相关门控）
- skills/kflow-explore/references/self-review.md

> **子代理隔离规则**：自审子代理异常时 MUST 重新创建（新 Agent 调用），主代理 SHALL NOT 接管自审执行。同一任务最多重试 3 次（1 次初始 + 3 次重试），全部失败后标记 ⚠️ 阻塞并提示用户。重新创建的子代理是全新 Agent，不依赖前上下文。

### 自审模式：子代理串行 + 重复制

> **v2.2.0 变更**: 自审执行方式从"当前 Agent 自身执行"改为"子代理（Agent subagent）串行执行"。每轮启动独立子代理，子代理拥有独立上下文避免"作者盲点"。

> **v2.1.0 变更**: 从"分工制"（每轮只审查部分维度）改为"重复制"——每轮独立执行全部四个维度（完整性+闭环性+必要性+清晰性），10 轮形成自然收敛。

| 对比 | 分工制（旧） | 重复制（新） |
|------|------------|------------|
| 每轮范围 | 部分维度 | **全部四个维度** |
| 发现节奏 | 后期才暴露其他维度问题 | 早期就暴露各类问题 |
| 收敛趋势 | 不明显 | 自然收敛（后期问题越来越少） |

### 审查维度与检查规则

详见 [references/self-review-dimensions.md](references/self-review-dimensions.md) — 四维度检查规则（完整性/闭环性/必要性/清晰性）及子代理执行说明。

### 自审执行流程（子代理串行 + 重复制）

```
10 轮自审执行流程（子代理串行，每轮全维度）:

1. 主 Agent 启动第一轮子代理:
   Agent(
     subagent_type="claude",
     description="Explore 自审 Round 1",
     prompt="读取 functional-designs/ 下的所有产物文件，按完整性、闭环性、必要性、清晰性全部四个维度独立检查。发现问题直接修复产物文件，生成审查报告到 self-reviews/explore/{YYYYMMDD}-{HHMMSS}.md。仅修复确认的问题，不做重构或额外改进。"
   )
2. 子代理返回审查报告路径
3. 主 Agent 读取报告，确认修复内容
4. 修复不合理 → 主 Agent 补充修复
5. 启动下一轮子代理（Round N+1），步骤同 Round 1
6. SHALL NOT 并行启动多个子代理（串行执行）
7. 重复直至完成全部 10 轮
8. 全部完成后标记阶段完成
```

### 强制执行规则

- SHALL 完成全部 10 轮自审，不允许提前终止
- 即使连续多轮无新问题也必须完成全部 10 轮
- SHALL 每轮启动独立子代理（Agent subagent），不允许主 Agent 自身执行自审
- SHALL NOT 并行启动多个子代理（串行执行，前一轮完成后再启动下一轮）
- 自审全部完成后标记阶段完成，释放 prototype 或 design 阶段门控

## 步骤 11：PROTO_GATE — 原型设计决策门控（仅前后端项目）

> **来源**: refine-skill-execution-rules 变更。探索完成后必须询问是否需要原型设计，避免原型设计被无意跳过。使用 `.status.md` 中的 `prototype_decision` 标记控制幂等，避免重复询问。

### 触发条件

| 条件 | 处理 |
|------|------|
| 项目类型 = 纯后端 | ⏭️ 跳过此步骤 |
| `.status.md` 中已有 `prototype_decision` 标记 | ⏭️ 跳过询问（幂等） |
| 前后端项目且无 `prototype_decision` 标记 | 执行 AskUserQuestion |

### AskUserQuestion 流程

```
Question: "检测到 {n} 个 UI 功能点。是否进入原型设计阶段？"
Options:
  - "确认创建原型"（推荐）→ 写入 prototype_decision=已选择原型设计 → 后续阶段进入 kflow-prototype-design
  - "跳过原型设计" → 写入 prototype_decision=已跳过 → 后续阶段直接进入 kflow-design
```

### 标记写入

在 `.status.md` 中写入：
```markdown
prototype_decision: 已选择原型设计 | 已跳过
```

此标记在整个变更生命周期内有效，后续阶段（如 kflow-guide、kflow-design）读取此标记决定流程走向。

## 阶段边界约束

### 域内内容（explore 负责）

| 内容类别 | 说明 |
|---------|------|
| 页面结构/菜单层级 | 用户可见的页面导航结构 |
| 可执行操作 | 用户在每个页面可执行的具体操作 |
| 表单项定义 | 字段名、类型、校验规则、默认值 |
| 业务规则 | 前置条件、校验规则、触发条件、后置结果 |
| 业务流程 | 用户视角的完整业务流程闭环 |
| 功能点拆分 | 原子级功能点、依赖关系、优先级 |

### 域外内容（禁止）

| 内容类别 | 说明 | 应由哪个阶段处理 |
|---------|------|----------------|
| 技术架构选型 | 框架、组件库、中间件选择 | design |
| 数据模型设计 | 实体定义、表结构、索引设计 | design |
| 接口定义 | API 路径、方法、入参出参 | design |
| 代码实现 | 任何代码文件 | code |
| 子变更划分 | 执行单元划分 | design |
| UI 原型 | 视觉设计和交互原型 | prototype |

### 越界处理

当 explore 阶段发现信息不足以完成功能设计时：
1. 在当前阶段文档中标记"待补充"并说明缺少什么
2. 记录到 `docs/skill-suggestion.md`
3. 禁止自创新文档来填补信息空缺
4. 禁止越界输出技术方案或接口定义

- **后续阶段**：前后端项目 → 原型设计（可选）或直接进入详细设计；纯后端项目 → 直接进入详细设计

# 核心提醒

- **FP 类型标记**：每个功能点 SHALL 标记为「后端」或「前端」，无法归类的 FP 必须继续拆分，类型作为后续子变更划分类型校验的依据
- **子变更划分不在此阶段执行**：后置到详细设计阶段完成后基于完整设计认知划分
- **10 轮自审强制执行（子代理串行 + 重复制）**：每轮启动独立子代理执行全部四个维度（完整性+闭环性+必要性+清晰性），子代理边审边修，串行不可并行，即使无新问题也必须完成全部 10 轮，不可提前终止
- **两动作规则**：每 2 次外部信息收集操作后立即保存关键发现到 functional-designs/
- **CONTEXT.md**：项目级领域词汇表，首次构建后持续增补，后续阶段只引用不修改
- **functional-designs/ 结构**：index.md（含版本号、需求变更记录表、配置项影响矩阵）+ part-NN.md 目录化结构
- **functional-designs/index.md 版本号**：格式为 `主版本.次版本.修订号`，初始版本为 `1.0.0`
- **变更粒度**：产品级 → 完整流程；功能需求级 → 标准流程；功能缺陷级 → 简化流程
- **禁止越界**：不得输出技术方案、接口定义、数据模型、子变更划分方案
- **变更名称**：使用 kebab-case 格式
- **原型设计决策门控**：探索完成后 SHALL 询问是否进入原型设计（前后端项目），写入 `prototype_decision` 标记控制幂等，避免重复询问
- **CONTEXT.md 对齐检查**：增补后必须执行术语对齐检查
