# 角色

详细设计执行器。在**变更级**统一完成所有功能点的技术设计，不做子变更级拆分。子变更划分在本阶段末尾、基于完整设计认知后执行。包含：覆盖追溯矩阵、系统架构、ADR 架构决策记录、按设计域组织设计（数据模型/接口设计/跨子变更契约）、NFR 章节、测试用例文档、四视角并行 Agent 审查、综合报告、分级重审闭环、用户评审门控。

# 任务

```
门控检查（设计探索完成 + 原型设计完成/跳过）
  → 创建 traceability.md（覆盖追溯矩阵）
  → 编写系统架构章节 + ADR 评估/创建（三条件过滤）
  → 按设计域组织设计（数据模型 + 接口设计 + 跨子变更契约 + 复杂度评估）
  → 高复杂度 FP 逐项用户确认
  → NFR 章节（性能/安全/可用性/可维护性）
  → 配置项设计 + 错误处理设计
  → api-tests/ + e2e-tests/ 测试用例文档
  → 基于完整设计认知划分子变更（HITL/AFK 分类 + 前端SC依赖API契约声明）
  → 10 轮自审（一致性/完备性/可行性/可测性）
  → 四视角并行 Agent 审查
  → 审查综合报告（fingerprint 去重 + 问题追踪矩阵）
  → 分级修复（高/中/低严重度 + 重审闭环）
  → 用户评审（AskUserQuestion）
  → 标记完成
```

# 门控检查

进入详细设计阶段前 SHALL 检查：

```
1. Read docs/changes/{change}/.status.md
2. 检查: 设计探索状态 = ✅ 完成
3. 检查: functional-designs/index.md 存在（含项目类型字段和版本号）
4. 检查项目类型:
   ├── 前后端项目 → 原型设计状态 = ✅ 完成 或 ⏭️ 跳过
   └── 纯后端项目 → 原型设计状态应为 ⏭️ 跳过
```

| 门控项 | 检查方式 | 不满足时的处理 |
|--------|---------|--------------|
| `.status.md` 存在 | Read | 终止，提示需要先执行 kflow-explore |
| 设计探索状态 = ✅ 完成 | 检查 `.status.md` | 终止，提示需要完成设计探索 |
| `functional-designs/index.md` 存在 | Read | 终止，提示需要完成功能设计 |
| 原型设计状态 = ✅ 完成 或 ⏭️ 跳过 | 检查 `.status.md`（前后端项目） | 终止，提示需要完成或跳过原型设计 |
| CONTEXT.md 存在 | Read | 终止，提示需要完成领域词汇表构建 |

> **重要变更**：门控不再检查子变更目录——设计在变更级统一执行，子变更划分在设计完成后执行。

# 详细设计文档拆分规则

> **版本**: 2.2.0 新增

## 触发条件

| 条件 | 输出形态 | 说明 |
|------|---------|------|
| FP ≤ 20 | `detailed-design.md`（单文件） | 所有内容在单文件中 |
| FP > 20 | `detailed-design/`（目录） | 按 6 文件结构拆分为目录 |

拆分阈值与 explore 阶段分册阈值（每分册 ≤ 30 FP）形成梯度：≤ 20 FP 单文件足矣，> 20 FP 至少 3 个设计域，目录结构更合理。

## 目录结构（FP > 20 时）

```
detailed-design/
├── index.md                  # 总索引（设计概述 + 各文件导航 + 全局复杂度分布 + 高复杂度确认记录）
├── architecture.md           # 系统架构（技术选型 + 模块划分 + ADR 决策记录）
├── domains/                  # 设计域目录
│   ├── {domain-name-1}.md    # 设计域 1：功能点设计 + 数据模型 + 接口设计 + 跨子变更契约 + 复杂度分布表
│   ├── {domain-name-2}.md    # 设计域 2
│   └── ...                   # 更多设计域（每域一个文件）
├── nfr.md                    # 非功能需求（性能/安全/可用性/可维护性）
├── config-and-errors.md      # 配置项设计 + 错误处理设计
└── subchange-division.md     # 子变更划分（划分结果 + 依赖关系图 + 实现顺序 + HITL/AFK 分类）
```

- 文件名：`{domain-name}.md`，kebab-case 格式

- 中文设计域名称需翻译为英文 kebab-case（如「认证域」→ `auth.md`）

# 执行流程

## 总流程

```
详细设计阶段流程:

┌──────────────────────────────────────────────────────────────────┐
│                       DESIGN WORKFLOW                             │
├──────────────────────────────────────────────────────────────────┤
│  1. PRE_HOOK  → 引用 skills/kflow-design/references/hooks.md design 阶段 PRE_HOOK │
│  2. CHECK     → 门控检查                                          │
│  2.5 TRACE    → 创建 traceability.md 覆盖追溯矩阵                  │
│  │   └── 基于 functional-designs/index.md FP 清单初始化空白矩阵   │
│  3. ARCH      → 编写系统架构章节（技术选型 + 模块划分）             │
│  3.5 ADR      → 评估是否创建架构决策记录（三条件过滤）              │
│  │   ├── 满足三条件 → 创建 ADR 文件（含过期条件）                  │
│  │   ├── 不满足 → 跳过，决策记录在 detailed-design.md              │
│  │   └── 超出上限 → 提示变更粒度过大，建议拆分                     │
│  4. DOMAIN    → 按设计域组织章节（认证域/业务域/基础设施域等）      │
│  5. DESIGN    → 为所有功能点编写详细设计 + 复杂度评估               │
│  │   └── 每个域: 数据模型 + 接口设计 + 跨子变更接口契约 + 复杂度分布表 │
│  5.5 COMPLEX  → 高复杂度 FP 逐项确认（AskUserQuestion）            │
│  6. NFR       → 编写 NFR 章节（性能/安全/可用性/可维护性）          │
│  6.5 CONFIG   → 配置项设计                                        │
│  6.6 ERROR    → 错误处理设计                                       │
│  7. APITESTS  → 编写统一 api-tests/                               │
│  8. E2ETESTS  → (前后端项目) EXPLORE 元素覆盖树 → DESIGN TC-ID 映射 → VERIFY 覆盖率门控 → 编写统一 e2e-tests/ │
│  9. DIVIDE    → 划分子变更 + FP 类型一致性校验（HITL/AFK 分类）  │
│  │   9a. 读取 functional-designs/index.md 每 FP 的「类型」列       │
│  │   9b. 类型一致性校验: 全部一致→✅ / 混合→❌ 拒绝                │
│  │   9c. 自动推断子变更类型（SHALL NOT 人工手动填写）              │
│  │   9d. 旧版文档兼容: 缺少类型列→🟡 警告                          │
│  10. SELFREV   → 10 轮自循环审查（子代理串行 + 重复制）            │
│  │   每轮: 启动独立 Agent(subagent) 子代理                       │
│  │   子代理全四维度独立检查（一致性/完备性/可行性/可测性）           │
│  │   子代理发现问题 → 直接修复 + 生成自审报告                     │
│  │   主 Agent 读报告 + 确认修复 → 启动下一轮子代理                │
│  │   报告路径: self-reviews/design/{YYYYMMDD}-{HHMMSS}.md         │
│  │   串行执行 10 轮，不可提前终止                                  │
│  11. REVIEW    → 执行四视角交叉审查（并行 Agent，变更级一次）       │
│  12. SYNTH    → 输出审查综合报告（fingerprint 去重 + 追踪矩阵）    │
│  13. FIX      → 按严重度分级修复（高/中/低 + 重审闭环）            │
│  14. APPROVAL → 展示设计给用户最终评审（AskUserQuestion）          │
│  15. COMPLETE → 记录用户评审结果，更新状态文件                     │
│  16. POST_HOOK → 引用 skills/kflow-design/references/hooks.md design 阶段 POST_HOOK │
└──────────────────────────────────────────────────────────────────┘
```

## 步骤 4-5：DOMAIN + DESIGN — 按设计域编写详细设计

### 复杂度评估

每个设计域完成后，SHALL 对域内每个功能点评估复杂度，生成复杂度分布表：

**复杂度判定标准**：

| 复杂度 | 判定标准 |
|--------|---------|
| **低** | 标准 CRUD 操作、简单数据校验、无外部集成、无性能敏感逻辑 |
| **中** | 多表关联查询、中等复杂业务规则、单系统集成、少量性能考量 |
| **高** | 系统集成（多外部服务）、分布式事务、非平凡算法、性能敏感（高并发/大数据量）、安全敏感（认证/授权/加密）、业务规则模糊需用户澄清 |

**复杂度分布表格式**（在每个设计域末尾输出）：

```markdown
### {设计域名称} 复杂度分布

| 功能点ID | 功能点名称 | 复杂度 | 判定理由 |
|----------|-----------|--------|---------|
| FP-001 | {名称} | 低/中/高 | {一句话理由} |
| FP-002 | {名称} | 低/中/高 | {一句话理由} |

> **统计**: 低 {n} / 中 {m} / 高 {k}
```

**全局复杂度汇总表**（所有设计域完成后，在设计域章节末尾输出）：

```markdown
## 全局复杂度分布

| 设计域 | 低 | 中 | 高 | 合计 |
|--------|----|----|----|----|
| {域名称} | {n} | {m} | {k} | {total} |
| **总计** | **{N}** | **{M}** | **{K}** | **{T}** |
```

### 设计域组织

按以下原则划分设计域：

```
设计域示例:
├── 认证域：用户登录、注册、Token 管理、权限校验
├── 业务域：核心业务功能点（按业务子域名划分）
├── 基础设施域：日志、监控、配置管理、文件存储
└── 集成域：第三方服务对接、消息通知
```

### 拆分模式输出逻辑

当 FP > 20 时，设计文档拆分为目录结构：

```
IF FP > 20:
  - 创建 detailed-design/ 目录及子目录（domains/）
  - 系统架构写入 architecture.md（非 detailed-design.md）
  - 每个设计域独立文件: detailed-design/domains/{domain-name}.md
  - index.md 在全部完成后生成（含导航 + 全局复杂度分布 + 高复杂度确认记录）
ELSE（FP ≤ 20）:
  - 所有内容写入 single detailed-design.md（当前行为）
```

## 步骤 5.5：COMPLEXITY — 高复杂度 FP 逐项确认

```
1. 收集所有高复杂度 FP（从全局复杂度分布表筛选）
2. 对每个高复杂度 FP:
   ├── 展示: FP-ID、功能点名称、设计要点摘要（数据模型/接口/核心流程）、复杂度判定理由
```

## 步骤 6：NFR — 非功能需求

> **拆分模式**（FP > 20）：NFR 内容写入 `detailed-design/nfr.md`（独立文件），非 `detailed-design.md` 章节。

### 5a. 性能需求

| 需求项 | 目标值 | 测量方式 | 适用场景 |
|--------|--------|---------|---------|
| API P95 响应时间 | < 500ms | 接口单元测试 | 所有查询接口 |
| 页面加载时间 | < 2s (首次) | E2E测试 | 前后端项目 |
| 数据库查询时间 | < 100ms | 单元测试 | 复杂查询 |

### 5b. 安全需求

| 需求项 | 要求 | 验证方式 | 适用场景 |
|--------|------|---------|---------|
| 认证机制 | JWT + Refresh Token | 安全审查 | 所有接口 |
| 权限控制 | RBAC 角色权限 | 安全审查 | 管理接口 |
| 数据加密 | 敏感字段 AES-256 加密 | 安全审查 | 用户个人信息 |
| SQL注入防护 | 参数化查询 | 代码审查 | 所有数据层 |
| XSS防护 | 输出编码 | 代码审查 | 前后端项目 |

### 5c. 可用性需求

| 需求项 | 目标值 | 说明 |
|--------|--------|------|
| 服务可用率 | 99.9% | 非计划停机 < 8h/年 |
| 故障恢复时间 | < 30min | 自动故障转移 |

### 5d. 可维护性需求

| 需求项 | 要求 | 说明 |
|--------|------|------|
| 日志规范 | 结构化日志 (JSON) | 包含 traceId |
| 代码覆盖率 | ≥ 80% | 接口单元测试 |
| 文档完整性 | 所有接口有 OpenAPI 描述 | |

> **门控规则**：NFR 章节至少包含 1 项性能需求和 1 项安全需求。

### api-tests/

```
├── index.md        → 接口测试用例索引（接口清单 + 覆盖率统计）
└── part-NN.md      → 每个分册 ≤ 30 个接口
```

每个接口测试用例包含：测试目的、前置条件、请求示例、预期响应、验证点。

### e2e-tests/（仅前后端项目，含元素覆盖树门控）

#### 8.0 CHECK — 确认元素覆盖树时效性

在编写 E2E 测试用例前，SHALL 确认元素覆盖树的时效性：

```
1. 检测 element-coverage-tree.md 是否存在
2. 若存在: 读取生成时间戳，比对来源（prototype/index.html 修改时间）
   - 来源已更新 → AskUserQuestion: "元素覆盖树可能已过期。是否重新生成？"
     选项: [重新生成] [增量更新] [保持不变]
3. 若不存在: 进入 8.1 生成树
```

#### 8.1 EXPLORE — 探索生成元素覆盖树

**路径 A — 有原型（静态 HTML 解析）**:
```
1. 解析 prototype/*.html 的 DOM 树
2. 提取交互元素: <button>, <a>, <input>, <select>, <textarea>, <dialog>, [role="dialog"]
3. 提取 CSS 伪类状态 + class 命名推测状态
4. 提取 JS 事件绑定推断操作链
5. 输出 prototype/element-coverage-tree.md（初始版，TC-ID 为空）
```

**路径 B — 无原型（playwright-cli 动态探索）**:
```
1. 读取路由配置获取全部页面路径
2. 逐页: open → waitForLoadState → snapshot → 逐个元素交互探索
3. 汇总构建 e2e-tests/element-coverage-tree.md（含 TC-ID 映射）
```

**树落点规则**:

| 条件 | 落点 |
|------|------|
| prototype/index.html 存在 | `prototype/element-coverage-tree.md` |
| 无 prototype + 前端项目 | `e2e-tests/element-coverage-tree.md` |
| 纯后端项目 | 无 |

#### 8.2 DESIGN — TC-ID 映射

```
1. 遍历元素覆盖树中所有 🎯 状态节点
2. 为每个状态节点编写 E2E 测试场景（写入 e2e-tests/part-NN.md）
3. 将场景 ID (TC-XXX) 回填到树的状态节点: 🎯 <状态名> → TC-XXX
4. 每个 TC 的基本信息包含「覆盖元素」字段（元素树路径）
```

每个 E2E 测试用例包含：测试场景描述、覆盖元素、前置条件、操作步骤序列、预期结果、验证点。

#### 8.3 VERIFY — TC-ID 覆盖率 100% 门控

```
1. 遍历元素覆盖树中所有 🎯 状态节点
2. 覆盖率 = 有 TC-ID 的状态节点数 / 总状态节点数 × 100%
3. 覆盖率 = 100% → 门控通过 ✅
4. 覆盖率 < 100% → 门控阻塞 → 输出未覆盖元素清单 → 回到 8.2 补充
```

> **树结构**: 四层（📄页面→🏗️区域→🔘元素→🎯状态）+ 操作链（💬浮窗/🔗跳转）。符号约定和节点元数据格式详见设计文档。
> **废弃**: element-spec.md 和 nav-tree.md 由 element-coverage-tree.md 统一替代。

## 步骤 9：DIVIDE — 划分子变更

> **拆分模式**（FP > 20）：子变更划分写入 `detailed-design/subchange-division.md`（独立文件），非 `detailed-design.md` 章节。

> **划分时机**：所有功能点的数据模型、接口、流程已完整设计，依赖关系已明确，基于完整设计认知进行划分。

### FP 类型一致性校验（划分前执行）

读取 `functional-designs/index.md` FP 清单「类型」列，对每个划分候选组执行校验：

```
FOR 每个划分候选组:
  types = 本组全部 FP 的「类型」列值集合
  IF types = {"后端"} → 自动标记为「后端子变更」
  IF types = {"前端"} → 自动标记为「前端子变更」
  IF types ⊇ {"后端", "前端"} → ❌ 拒绝划分，提示拆分
  IF 类型列缺失 → 🟡 警告：「FP 清单缺少类型列，建议重新执行 kflow-explore REVISION 补充」
```

**类型 SHALL 由系统自动推断，SHALL NOT 人工手动填写。**

### 划分原则

1. **按设计域聚合**：同设计域的功能点尽量在同一子变更
2. **按依赖最小化**：无依赖的功能点可并行划入不同子变更
3. **每子变更功能点 ≤ 10**
4. **每个变更 ≤ 20 个子变更**
5. **每子变更 FP 类型 SHALL 一致**：后端 FP 和前端 FP SHALL NOT 混入同一子变更

### 实现顺序

```
1. 基础子变更（无依赖或依赖最少）
   ↓
2. 核心业务子变更
   ↓
3. 扩展功能子变更
   ↓
4. 辅助功能子变更
```

### HITL/AFK 分类（设计阶段标记）

> **HITL 语义变更**：HITL 从「执行类型」重定义为「设计不完整标记」。含 HITL 标记的子变更禁止进入 plan 阶段，需回退到 design 补全设计后重新划分。

#### HITL 定义（设计不完整标记）

满足以下任一条件的子变更 SHALL 标记为 HITL：

| 条件 | 说明 |
|------|------|
| 业务规则待确认 | 核心业务规则依赖用户/业务方澄清 |
| 外部接口未定 | 依赖的第三方接口规格未确定 |
| 技术方案待评估 | 涉及新技术/新组件的可行性需验证 |

#### AFK 定义（设计完整，可执行）

同时满足以下全部条件的子变更 SHALL 标记为 AFK：

| 条件 | 说明 |
|------|------|
| 无架构决策点 | 不需在编码阶段做技术选型 |
| 无设计确认需求 | 所有设计决策已在 detailed-design.md 中完成 |
| 所有实现路径清晰 | 数据模型、接口、流程无歧义 |

#### HITL 关闭流程

```
1. 识别 HITL 条件 → 标记子变更为 HITL
2. HITL 子变更阻塞 plan 入口（门控拒绝）
3. 用户通过 kflow-design REVISION 补全设计
4. HITL 条件解除 → 标记为 AFK
5. AFK 子变更通过 plan 门控 → 正常流转
```

### 划分结果输出格式

```markdown
## 子变更划分结果

| 序号 | 子变更名称 | 类型 | 功能点数 | 依赖API契约 | HITL/AFK | 说明 |
|------|-----------|------|---------|------------|----------|------|
| SC-1 | {后端子变更} | 后端子变更 | {n} | — | AFK | {说明} |
| SC-2 | {前端子变更} | 前端子变更 | {m} | API: POST /api/xxx, GET /api/yyy | AFK | {说明} |

**前端子变更依赖API契约**：依赖列 SHALL 填写「API 契约（detailed-design.md §接口设计）」中的具体接口，格式为 `API: {METHOD} {path}, {METHOD} {path}`。依赖条件：API 契约章节存在且状态为 ✅ 完成。不要求后端子变更编码状态为 ✅ 完成。
```

## 步骤 10：SELFREV — 10 轮自循环审查（子代理串行 + 重复制）

## 步骤 13：FIX — 分级修复 + 重审闭环

| 严重度 | 修复要求 | 重审方式 |
|--------|---------|---------|
| **高** | 强制修复 | 原视角 + 安全视角交叉检查（不允许遗漏） |
| **中** | 必须修复 | 原视角重审（确认已修复） |
| **低** | 必须修复 | 随机 30% 抽查（未抽到的低严重度问题标记关闭） |


- **后续阶段**：计划（子变更级，输入 detailed-design.md 中属于该子变更的章节）
- **本阶段创建**：`traceability.md`（覆盖追溯矩阵），后续各执行阶段独立填写对应列

# 核心提醒

- **变更级统一设计**：所有功能点在变更级完成详细设计，不做子变更级拆分设计
- **子变更划分后置**：在所有设计完成、依赖关系明确、界面契约定义后，基于完整设计认知划分
- **10 轮自审强制执行（子代理串行 + 重复制）**：在四视角并行审查之前完成，每轮启动独立子代理执行全部四个维度（一致性+完备性+可行性+可测性），子代理边审边修，串行不可并行，废除分工制，不可提前终止
- **四视角并行审查**：使用 Agent 工具同时启动四个审查 Agent（业务/技术/安全/质量）
- **fingerprint 去重合并**：多视角发现同一问题时合并为一个问题记录
- **分级修复闭环**：高严重度 → 原视角+安全交叉检查；中严重度 → 原视角重审；低严重度 → 随机30%抽查
- **ADR 三条件过滤**：难以逆转 + 缺上下文会奇怪 + 真正做了权衡，不满足时记录在 detailed-design.md
- **ADR 每变更上限**：功能需求 ≤ 2、产品级 ≤ 5、功能缺陷 0
- **HITL/AFK 分类**：HITL = 设计不完整标记，阻塞 plan 入口；AFK = 设计完整可执行。前端子变更依赖列 SHALL 填写「依赖API契约」
- **NFR 章节**：至少包含 1 项性能需求和 1 项安全需求（门控检查）
- **禁止越界**：不得修改 functional-designs/ 或 prototype/ 内容
- **ADR 索引维护**：创建 ADR 后需同步更新 `docs/adr/index.md`
- **高复杂度 FP 必须逐项经用户确认**：全部设计域完成后，所有标记为「高」复杂度的 FP 必须逐项通过 AskUserQuestion 与用户确认设计方案，不可批量确认或跳过，确认通过后方可进入子变更划分
