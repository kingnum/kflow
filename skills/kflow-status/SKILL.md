# 角色

变更状态总结器。扫描所有未归档变更，汇总各变更阶段状态和子变更进度矩阵。输出含子变更5阶段进度展示、加权整体进度、依赖关系说明。纯查询阶段，不产生文件。支持读取 docs/changes/index.md 索引文件补充元信息。

# 任务

扫描 docs/changes/ 目录 -> 过滤未归档变更（排除 archive/）-> 尝试读取 docs/changes/index.md 索引文件 -> 读取各变更的 .status.md 文件 -> 汇总各变更阶段状态和子变更进度矩阵 -> 计算加权整体进度 -> 输出状态汇总表格。

# 门控检查

此阶段为独立查询阶段，无前置门控要求。自动扫描 docs/changes/ 目录下的所有未归档变更。支持读取 docs/changes/index.md 索引文件（如存在）。不产生任何文件。

# 输入要求

| 产物 | 图例 | 说明 |
|------|------|------|
| 变更目录 | ✅ 必须 | 自动扫描 `docs/changes/` 目录 |
| docs/changes/index.md | 🔶 条件 | 变更管理索引文件，如存在则优先使用其元信息 |

# 输出产物

在对话中输出状态汇总表格，不产生文件。

| 产物 | 说明 | 图例 |
|------|------|------|
| 未归档变更总览表 | 所有活跃变更的概览信息 | ✅ 必须 |
| 各变更详细进度 | 每个变更的阶段状态表 | ✅ 必须 |
| 子变更进度矩阵 | 每个变更的子变更 5 阶段进度展示 | 🔶 条件（当子变更数 > 0 时） |
| 依赖关系与状态说明 | 子变更间依赖和当前状态说明 | 🔶 条件（当子变更数 > 0 时） |

# 执行流程

## 总流程

```
SCAN(扫描 docs/changes/) -> FILTER(排除 archive/) -> INDEX(读 index.md 如存在)
    -> READ(各变更 .status.md) -> AGGREGATE(汇总计算) -> OUTPUT(输出表格)
```

```
┌─────────────────────────────────────────────────────────────────┐
│                       STATUS WORKFLOW                            │
├─────────────────────────────────────────────────────────────────┤
│  1. SCAN       扫描 docs/changes/ 目录                           │
│  2. FILTER     过滤未归档变更（排除 archive/ 子目录）            │
│  3. INDEX      尝试读取 docs/changes/index.md（如存在）          │
│     └── 索引文件存在 -> 补充活跃变更列表信息                     │
│     └── 索引文件不存在 -> 仅从扫描结果获取                       │
│  4. READ       读取各变更的 .status.md 文件                      │
│  5. AGGREGATE  汇总各变更阶段状态和子变更进度                     │
│  6. OUTPUT     输出状态汇总表格（含子变更进度矩阵）              │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤 1：SCAN — 扫描变更目录

使用 Glob 扫描 `docs/changes/*/` 下的所有变更目录。对于每个目录，提取变更名称（目录名）。

## 步骤 2：FILTER — 过滤未归档变更

```
过滤逻辑:

1. 对于每个候选变更 dir:
   ├── Glob: docs/archive/*/{dir}/.status.md 是否存在？
   ├── 或 Glob: docs/archive/*-{dir}/ 目录是否存在？
   ├── 已归档 -> 排除
   └── 未归档 -> 保留

2. 排除 docs/changes/index.md 自身（非变更目录）
3. 空目录（无 .status.md）-> 标记为"未初始化"，仍纳入列表
```

## 步骤 3：INDEX — 读取索引文件

如果 `docs/changes/index.md` 存在，从中获取：
- 活跃变更列表（名称、类型、创建时间）
- 已归档变更列表
- 变更元信息（描述、影响文件等）

索引文件不存在时，从目录扫描结果构建信息。

## 步骤 4：READ — 读取状态文件

```
读取逻辑:

1. 对每个未归档变更:
   ├── Read: docs/changes/{change}/.status.md
   │   ├── 提取: 变更描述、变更类型、项目类型、当前阶段
   │   ├── 提取: 阶段状态表（各阶段 ✅/🔄/⏳/⏭️/❌/⚠️ 状态）
   │   └── 提取: 创建时间（如有）
   ├── 如有子变更:
   │   ├── Glob: docs/changes/{change}/subchanges/*/.status.md
   │   ├── 提取每个子变更的阶段状态
   │   └── Read: 各子变更的 tasks.md（获取功能点数和依赖关系）
   └── 如无 .status.md -> 标记为"未初始化"
```

## 步骤 5：AGGREGATE — 汇总计算

### 状态值含义与进度映射

| 状态值 | 含义 | 进度映射 |
|--------|------|---------|
| ✅ 完成 | 阶段已完成，产物已生成 | 100% |
| 🔄 进行中 | 阶段正在执行 | 按任务完成比例估算 |
| ⏳ 待开始 | 阶段未开始 | 0% |
| ⏭️ 跳过 | 阶段被跳过（可选阶段或不适用） | 不计入进度（移除权重） |
| ❌ 阻塞 | 阶段被阻塞，无法继续 | 保持当前进度 |
| ⚠️ 有问题 | 阶段完成但有遗留问题 | 80% |
| ⚠️ 需修订 | 阶段产物需回退重做 | 50% |

### 阶段权重表

| 阶段 | 前后端项目 | 纯后端项目 |
|------|-----------|-----------|
| 设计探索 | 10% | 10% |
| 原型设计 | 10% | 0%（跳过） |
| 详细设计 | 25% | 25% |
| 计划 | 10% | 10% |
| 编码 | 20% | 20% |
| 代码审查 | 5% | 5% |
| 接口单元测试 | 10% | 10% |
| E2E 测试 | 10% | 0%（跳过） |
| 集成测试 | 10% | 10% |

### 整体进度计算规则

```
变更整体进度 = Σ(阶段进度 × 阶段权重) / Σ(阶段权重)

计算步骤:
1. 根据项目类型确定阶段列表
2. 对每个阶段:
   ├── ⏭️ 跳过 -> 移出计算，该阶段权重不参与分母
   ├── ✅ 完成 -> 进度 = 100%
   ├── 🔄 进行中 -> 进度 = 估算比例（从 tasks.md checkbox 统计）
   ├── ⚠️ 需修订 -> 进度 = 50%
   ├── ⚠️ 有问题 -> 进度 = 80%
   ├── ❌ 阻塞 -> 保持当前进度值
   └── ⏳ 待开始 -> 进度 = 0%
3. 加权求和: Σ(进度 × 权重) / Σ(权重)
4. 结果取整到 1%
```

### 子变更进度矩阵构建

```
子变更进度矩阵构建:

1. 遍历 docs/changes/{change}/subchanges/*/.status.md
2. 提取每个子变更的:
   ├── 功能点数（从 tasks.md checkbox 总数）
   ├── 5 阶段状态: 计划 / 编码 / 代码审查 / 接口单元测试 / E2E测试
   ├── 依赖关系（从 tasks.md 的「依赖子变更」字段）
   ├── 执行类型: AFK（无人值守）/ HITL（需人工介入）
   └── 当前阶段（从 .status.md 的「当前阶段」字段）
3. 按依赖顺序排列子变更:
   ├── 拓扑排序：无依赖的在前
   └── 有依赖的排在依赖项之后
4. 计算各子变更加权完成度 = 各阶段进度加权平均
```

## 步骤 6：OUTPUT — 输出格式

### 输出结构

```
未归档变更总览表（一览所有变更）:
  -> 各变更详细进度（阶段状态表）:
      -> 子变更进度矩阵（如有子变更，展示 5 阶段进度）:
          -> 依赖关系与状态说明
```

### 未归档变更总览表

```markdown
## 未归档变更状态汇总

| 变更名称 | 类型 | 项目类型 | 当前阶段 | 整体进度 | 子变更数 | 创建时间 |
|----------|------|---------|----------|----------|----------|----------|
| ecommerce-init | 产品需求 | 前后端项目 | 编码 | 45% | 3 | 2026-04-29 |
| add-user-auth | 功能需求 | 纯后端项目 | 详细设计 | 60% | 1 | 2026-04-29 |
| fix-login-bug | 功能缺陷 | 前后端项目 | 集成测试 | 80% | 1 | 2026-04-28 |
```

**字段说明**:
- **整体进度**: 加权计算后的百分比
- **子变更数**: 0 表示变更级阶段尚未进行到子变更划分
- **创建时间**: 从 index.md 或 .status.md 获取，无数据时填 `—`

### 各变更详细进度

```markdown
### 详细进度

#### ecommerce-init (产品需求级, 前后端项目) - 3个子变更

| 阶段 | 状态 | 进度 |
|------|------|------|
| 设计探索 | ✅ 完成 | 100% |
| 原型设计 | ✅ 完成 | 100% |
| 详细设计 | ✅ 完成 | 100% |
| 计划 | ✅ 完成 | 100% |
| 编码 | 🔄 进行中 | 45% |
| 代码审查 | ⏳ 待开始 | 0% |
| 接口单元测试 | ⏳ 待开始 | 0% |
| E2E测试 | ⏳ 待开始 | 0% |
| 集成测试 | ⏳ 待开始 | 0% |
```

若变更的阶段状态为纯后端项目（无原型设计和 E2E 测试阶段），则不显示被跳过的阶段；若阶段标记为 ⏭️ 跳过，则该行显示 ⏭️ 跳过 而非省略。

### 子变更进度矩阵

子变更数 > 0 时输出，展示每个子变更的 5 阶段进度：

```markdown
**子变更进度矩阵**:

| 子变更 | 功能点 | 类型 | 计划 | 编码 | 代码审查 | 接口单元测试 | E2E测试 | 依赖 |
|--------|--------|------|------|------|---------|-------------|---------|------|
| user-auth | 10 | AFK | ✅ | ✅ | ✅ | ✅ | ✅ | 无 |
| order-management | 8 | AFK | ✅ | 🔄 | ⏳ | ⏳ | ⏳ | user-auth |
| payment-integration | 6 | HITL | ✅ | ⏳ | ⏳ | ⏳ | ⏳ | order-management |
```

**字段说明**:
- **类型**: AFK（无人值守，Agent 闭环）/ HITL（需人工介入，如支付、审核等）
- **依赖**: 列出依赖的其他子变更名称，"无"表示无依赖
- **E2E测试**: 纯后端项目显示 ⏭️
- 子变更按依赖拓扑排序

### 依赖关系与状态说明

```markdown
**依赖关系与状态说明**:
- `order-management` 依赖 `user-auth`：user-auth 需完成后方可推进
- `payment-integration` 依赖 `order-management`：当前被阻塞，等待 order-management 编码完成
- `payment-integration` 为 HITL 类型，包含需要人工介入的支付对接环节
```

### 空状态输出

当无未归档变更时：

```markdown
## 未归档变更状态汇总

当前没有未归档的变更。

> 使用「做/添加/实现/搭建 + 功能描述」创建新变更，或使用 `/kflow-guide` 获取流程指引。
```

# 子变更 5 阶段进度展示说明

子变更进度矩阵展示每个子变更在其生命周期中的 5 个阶段状态：

| 阶段 | 说明 | 状态值 |
|------|------|--------|
| 计划 | 任务清单和 DoD 验收标准准备 | ✅/🔄/⏳/⏭️ |
| 编码 | TDD 流程编码实现 | ✅/🔄/⏳/⏭️ |
| 代码审查 | 两视角并行审查闭环 | ✅/🔄/⏳/⏭️ |
| 接口单元测试 | 子变更级接口测试 | ✅/🔄/⏳/⏭️ |
| E2E 测试 | 端到端测试（仅前后端项目） | ✅/🔄/⏳/⏭️ |

纯后端项目的子变更 E2E 测试列固定为 ⏭️。

# 索引文件读取规则

docs/changes/index.md 的优先级：

```
1. 索引文件存在:
   ├── 活跃变更列表: 以索引文件为准
   ├── 元信息（创建时间、类型）: 优先使用索引文件数据
   └── 索引文件与目录不一致: 标注差异，以目录扫描为准

2. 索引文件不存在:
   └── 全部信息从目录扫描和 .status.md 读取
```

# 与其他 Skill 的关系

- **独立阶段**：可随时调用，不依赖其他阶段
- **无前置阶段**：无需门控检查
- **无后续阶段**：仅用于状态查看，不触发后续流程
- **被 guide 调用**：当用户表达"状态/进度/查看/总结"意图时，guide 路由到此 Skill

# 核心提醒

- **纯查询阶段**：不产生任何文件，不修改任何状态
- **⏭️ 跳过阶段不计入进度**：权重从分母中移除，重新分配
- **子变更进度矩阵展示每个子变更的 5 阶段进度和依赖关系**
- **index.md 存在时优先使用其元信息**，与目录不一致时标注差异
- **空目录（无 .status.md）标记为"未初始化"**，仍纳入列表
- **子变更按依赖拓扑排序**：无依赖在前，有依赖在后
- **HITL 类型子变更特别标注**，提醒需要人工介入
- **加权进度取整到 1%**，避免虚假精度
