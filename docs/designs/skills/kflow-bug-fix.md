# kflow-bug-fix（缺陷修复阶段）

> **版本**: 参见仓库根目录 `VERSION` 文件
> **阶段**: 缺陷修复（按需阶段，子变更级，二分法根因分类）

---

## 基本信息

```yaml
name: kflow-bug-fix
description: 缺陷修复阶段 - 系统化调试、子变更级二分法根因分类路由（实现错误/测试错误）、缺陷修复报告。按需阶段，仅处理子变更级缺陷修复。入口仅限测试阶段自动发现（B路径），用户反馈统一由 kflow-bug-triage 处理。变更级集成测试缺陷修复由 `kflow-integration-test` Skill 内聚处理。阶段钩子引用 `skills/kflow-bug-fix/references/hooks.md`（同触发阶段的钩子配置）。
```

## 门控检查

> **机制说明**：门控规则定义在 [core-mechanisms/03-status-and-tasks.md](../core-mechanisms/03-status-and-tasks.md#34-门控规则)

进入缺陷修复阶段前检查：
- 存在测试失败的测试轮次报告（子变更级：接口单元测试或 E2E 测试）

> **注意**：用户反馈统一由 `kflow-bug-triage` 处理（A路径），本 Skill 入口仅限测试阶段自动发现（B路径）。

缺陷修复完成后：
- 本地验证通过
- 输出缺陷修复报告（含根因分类）
- 实现错误/测试错误 → 返回测试阶段

自动跳过规则：
- 测试阶段一轮通过 → 自动跳过缺陷修复阶段

> **注意**：本 Skill 仅处理子变更级缺陷修复。集成测试失败时使用 `kflow-integration-test` Skill 内置的四分法修复循环。

## 根因分类（二分法）

### 分类标准

| 根因类型 | 判断依据 | 占比估计 | 处理路由 |
|---------|---------|---------|---------|
| 实现错误 | 代码实现与设计文档一致但逻辑有误、未处理边界条件、编码错误 | ~80% | 修复代码 |
| 测试错误 | 测试用例预期值与设计不符、测试数据准备错误、测试环境配置问题 | ~20% | 修正测试 |

> **变更说明**：原三分法中的"设计错误"分类已上移到 `kflow-bug-triage`。当三次尝试后升级用户且用户选择设计回退时，引导用户通过 kflow-bug-triage 进行正式诊断。

### 决策流程

```
根因分析 Agent 完成分析后:

1. 对比测试失败信息与 detailed-design.md 中的设计定义
2. 检查代码实现是否与设计一致:
   ├── 一致但仍有 Bug → 实现错误（默认分类）
   ├── 不一致 → 检查不一致的原因:
   │   └── 实现疏忽 → 实现错误
   └── 测试断言值与设计预期不符 → 测试错误
```

## 修复尝试上限

为防止子变更级缺陷修复无限自动循环，SHALL 遵循以下尝试上限规则：

### 递进策略

同一测试用例的三次尝试遵循递进策略：

| 尝试次数 | 策略 | 说明 |
|----------|------|------|
| 第 1 次 | 直接修复 | 按二分法根因分类采用针对性方案，快速定位并修复 |
| 第 2 次 | 替代方案 | 采用不同工具或方法，不得重复第 1 次尝试的相同方案 |
| 第 3 次 | 重新思考 | 重新审视问题假设，考虑更广泛的替代方案，不得重复前两次方案 |
| 超出上限 | 升级用户 | 使用 AskUserQuestion 升级用户，提供三个选项：保留当前状态（人工介入）/ 跳过该功能点（标记为已知问题）/ 通过 kflow-bug-triage 进行诊断 |

### 计数规则

| 规则 | 说明 |
|------|------|
| 独立计数 | 不同测试用例的尝试次数独立计算，一个用例达到上限不影响其他用例 |
| 根因变更重置 | 修复过程中根因分类发生变更时（如从实现错误变更为设计错误），重置尝试计数 |
| 跨轮次累加 | 同一测试用例在不同轮次中的尝试次数累加计算 |
| 上限兜底 | 无论根因类型，同一用例累计 3 次失败后 SHALL 升级用户 |

### 修复报告记录

每次修复尝试 SHALL 在缺陷修复报告中记录：
- 当前尝试次数（第 N/3 次）
- 采用的修复策略（直接修复 / 替代方案 / 重新思考）
- 与之前尝试的方案差异说明（第 2 次起必填）
- 失败原因分析（如适用）

## Feedback Loop 构建 (DISCOVER 1a)

### 概述

缺陷修复的第一步是构建一个快速、确定性、Agent 可运行的 pass/fail 信号，用于验证缺陷是否存在及修复是否有效。

### 10 种方法优先级排序

按优先级从高到低尝试，自动化优先：

| 优先级 | 方法 | 说明 | 适用场景 |
|--------|------|------|---------|
| 1 | 失败测试 at nearest seam | 运行最近接口的已有失败测试 | 接口单元测试框架可用 |
| 2 | Curl/HTTP 脚本 | 针对 dev server 运行 HTTP 请求脚本 | 后端 API 缺陷 |
| 3 | CLI 调用 with fixture | 输入 fixtured 数据 → diff snapshot 输出 | CLI 工具、数据处理 |
| 4 | Headless 浏览器脚本 | Playwright/Puppeteer 脚本 | 前端 UI 缺陷 |
| 5 | Replay captured trace | 回放捕获的 HAR/请求 trace | 有抓包 trace 可用 |
| 6 | Throwaway harness | 最小化子系统 harness | 复杂集成场景 |
| 7 | Property/Fuzz loop | 1000 随机输入 fuzz 循环 | 数据相关缺陷 |
| 8 | Bisection harness | git bisect run 自动化 | 回归缺陷 |
| 9 | Differential loop | 旧版本 vs 新版本 diff 对比 | 性能回归 |
| 10 | HITL bash script (last resort) | 人工操作的脚本 | 需要手动步骤的缺陷 |

### Loop 优化

构建成功后，尝试优化：

| 优化维度 | 目标 | 方法 |
|---------|------|------|
| 速度 | 2 秒确定性 loop 优于 30 秒不稳定 loop | 缓存 setup、跳过无关 init |
| 信号清晰度 | 断言具体症状，非模糊"不对劲" | 精确断言失败模式 |
| 确定性 | 每次运行结果一致 | 固定时间、随机种子、文件系统隔离 |

### 无法构建 Loop 时的处理

所有 10 种方法均失败时：

1. 停止并明确列出已尝试方法
2. 请求用户提供：
   - a) 可复现环境访问权限
   - b) 捕获的 artifact（HAR 文件、日志 dump、core dump、带时间戳的屏幕录制）
   - c) 添加临时生产 instrumentation 的授权
3. 禁止不通过 feedback loop 直接进入假设阶段

## 非确定性缺陷处理 (DISCOVER 1b)

### 复现率提升策略

对于不能 100% 复现的缺陷，不追求 100% 复现，而是提升复现率至可调试水平：

| 策略 | 说明 |
|------|------|
| Loop 触发 100× | 在循环中重复触发，统计数据 |
| 并行化 | 同时运行多个 loop 实例 |
| 增加压力 | 提高并发度、数据量 |
| 缩小时间窗口 | 聚焦最可能触发的关键路径 |
| 注入 sleeps | 在竞态条件怀疑点注入延时 |

### 复现率分级

| 复现率 | 分类 | 处理方式 |
|--------|------|---------|
| ≥ 50% | 可调试 | 进入正常 DEBUG → HYPOTHESISE 流程 |
| 1% - 50% | 边缘可调试 | 提升复现率后再进入假设阶段 |
| < 1% | 不可调试 | 转向"无法构建 loop"流程，请求用户协助 |

## 可证伪多假设生成 (DISCOVER 1c)

### 假设格式

每个假设 SHALL 使用可证伪格式：

```
如果 <X> 是原因，则 <改变 Y> 会让 bug 消失 / <改变 Z> 会让它更严重
```

### 生成规则

| 规则 | 说明 |
|------|------|
| 数量 | 生成 3-5 个假设 |
| 可证伪性 | 每个假设 SHALL 给出可证伪预测 |
| 不可证伪 → 丢弃 | 无法给出可证伪预测的假设视为"感觉"，丢弃或锐化 |
| 排序 | 按可能性排序后展示给用户 |
| 用户参与 | 用户可以基于领域知识重排序 |

### 单变量测试

| 规则 | 说明 |
|------|------|
| 每次仅改变一个变量 | 确保测试结果可归因 |
| 测试结果映射 | 将每次测试结果映射到对应假设的预测 |
| 反馈循环验证 | 最终验证假设时使用 Phase 1 构建的 loop |

## Regression Test at Correct Seam (步骤 3.5)

### Seam 正确性评估

修复方案确定后，在修复前评估测试 seam：

| 评估项 | 说明 |
|--------|------|
| seam 是否锻炼真实 bug 模式 | 测试能否复制触发 bug 的完整调用链 |
| 正确 seam | 先写回归测试确认失败 → 应用修复 → 确认通过 |
| seam 太浅 | single-caller 测试无法复制真实调用链 → 标记"无正确 seam" |

### 无正确 seam 时的处理

标记为发现并记录到 fix-report：

1. 记录为什么不存在正确 seam
2. 建议需要哪种架构改进（如模块拆分、接口重构）
3. 这些建议在 post-mortem 阶段传递给 kflow-audit

### 原始场景验证

修复完成后：
1. 重新运行 Phase 1 构建的 feedback loop
2. 确认原始（未最小化）场景不再复现缺陷

## Post-mortem 检查点 (步骤 7)

### 触发条件

修复已完成且本地验证通过后自动执行。

### 检查内容

| 检查项 | 内容 |
|--------|------|
| 复盘问题 | "什么可以防止此缺陷发生？"、"此缺陷的根本原因为何未被早期阶段捕获？" |
| 架构改进识别 | 是否存在架构改进可以防止同类缺陷（如缺少正确的测试 seam、模块耦合导致变更波及） |
| 假设记录 | 最终正确的假设及证明过程；备选假设被排除的原因 |
| 调试日志清理 | 所有调试 instrumentation 已清理；throwaway prototype 已删除或移动到标记位置；临时文件已清理 |

### 架构改进建议传递

| 来源 | 目标 | 内容 |
|------|------|------|
| fix-report "关联建议" 字段 | kflow-audit 审计评估 | 架构层面的改进机会 |
| Regression test seam 问题 | kflow-audit 审计评估 | 测试 seam 缺失报告 |
| 假设记录 | fix-report "最终正确的假设" 字段 | 供未来调试参考 |

### 无改进时的处理

post-mortem 未发现架构改进机会时，简要记录"无架构改进建议"，检查点仍标记完成。

## 缺陷修复报告格式

```markdown

## 基本信息
- **发现时间**: {YYYY-MM-DD HH:MM}
- **修复级别**: 子变更级
- **发现阶段**: {测试轮次或其他}
```

## 重复制（执行类阶段）

> ⚠ **子代理强制规则**（参见 skills/kflow-bug-fix/references/repetition.md §12）：本阶段（缺陷修复）主工作 MUST 通过 Agent 子代理执行，主 Agent 仅负责调度和验收，SHALL NOT 直接执行缺陷修复主工作（缺陷诊断/根因分析/代码修复/测试修正等），无例外。子代理崩溃时轮次级重试（≤3 次），全部失败标记 ⚠️ 阻塞。

缺陷修复阶段属于执行类阶段，采用重复制模式。子代理每轮遍历全部失败用例，每个独立分析+修复+验证，内层 3 次修复上限独立共存于外层目标轮次迭代中，自然收敛。目标轮次由弹性轮次决策规则确定（参见 `skills/kflow-bug-fix/references/repetition.md` §14）：

- **首次执行**：10 轮（标准重复制）
- **回退重执行**：按 .status.md「最近修订信息」中的影响范围分数决定轮次
  - 分数 1-5：max(3, ceil(分数)) 轮
  - 分数 6-15：max(5, ceil(分数/2)) 轮
  - 分数 >15：10 轮
- **弹性模式验证门控**（目标轮次 < 10 时）：受影响项 100% + 全量 ≥1 轮兜底 + 产物完整性（参见 `skills/kflow-bug-fix/references/repetition.md` §15）

### 每轮工作内容

**遍历项**：全部失败用例

**每轮流程**：
1. 遍历全部失败用例，每个独立执行完整修复流程：
   - Phase 1: BUILD LOOP（10 种方法优先级尝试，优化 loop）
   - Phase 2: REPRODUCE（确定性/非确定性缺陷策略）
   - Phase 3: HYPOTHESISE（3-5 个可证伪假设 → 单变量测试）
   - 二分法根因分类 → 修复 → 验证循环
   - Regression test at correct seam 评估
   - Post-mortem 检查点（复盘 + 架构建议 + 日志清理 + 假设记录）
2. 已修复的用例仍需验证无回归
3. 单用例 3 次修复上限独立共存于内层（不终止外层轮次）

**每轮产物**：fix-report 或 design-error-report

**轮次结束后**：更新 .status.md 执行轮次计数器

### 复杂度评估

复杂度评估仅信息展示，不驱动执行行为：

```
复杂度分 = 失败用例数 × 1.5 + 非确定性缺陷 × 2

低复杂度 (< 20 分) / 中复杂度 (20-50 分) / 高复杂度 (> 50 分)
```

分级阈值保留但仅用于信息分类，复杂度分写入 .status.md 备注列标注「仅供参考，不驱动执行行为」。无论复杂度高低，均须完成目标轮次（由 `skills/kflow-bug-fix/references/repetition.md` §14 弹性轮次决策确定）迭代后方可返回。

### 目标轮次与3次修复上限的层级关系

缺陷修复阶段并行维护两个独立指标：

| 维度 | 目标轮次 | 3次修复上限 |
|------|---------|-----------|
| 约束层级 | 外层循环（整体迭代轮次） | 内层循环（单用例修复尝试） |
| 方向 | 下限（floor）——确保充分迭代 | 上限（ceiling）——阻止无限重试 |
| 计数范围 | 每阶段统一计数 1-N（N 由弹性轮次决策确定） | 每用例独立计数 1-3，跨轮次累加 |
| 触发后果 | 未达 N/N → 拒收，继续执行 | 达到 3 次 → 升级用户决策 |
| 互动关系 | 保障整体迭代深度 | 上限触发不终止轮次，仍须递增至 N/N |

**场景**：第 2 轮某用例 3 次修复均失败 → 升级用户决策 → Agent 继续处理当前轮次其他可修复问题 → 递增至目标轮次 N/N → 满轮后返回。极端情况所有用例均触发上限 → Agent 标注「所有问题已升级用户决策，无可修复项」并继续递增计数器至 N/N。

### 执行流程

```
1. 复杂度评估 → 写入 .status.md 备注列（仅信息展示，不驱动执行行为）

1.5 INIT → 主 Agent 按 §14 弹性轮次决策确定目标轮次 N，写入 .status.md 执行轮次为 1 / N
    └── 判定流程: 读取 .status.md「阶段执行历史」表 → 无记录则首次执行(N=10) / 有记录则回退重执行(读取影响范围分数按 §14.2 映射)

2. 构建阶段专属提示词
   ├── 输入: 测试轮次报告 + detailed-design.md + functional-designs/
   ├── 执行要求:
   │   ├── Phase 1: BUILD LOOP（10 种方法优先级尝试，优化 loop）
   │   ├── Phase 2: REPRODUCE（确定性/非确定性缺陷策略）
   │   ├── Phase 3: HYPOTHESISE（3-5 个可证伪假设 → 单变量测试）
   │   ├── 二分法根因分类 → 修复 → 验证循环
   │   ├── 修复尝试上限: 每用例 3 次（上限），第 4 次升级用户
   │   ├── Regression test at correct seam 评估
   │   └── Post-mortem 检查点（复盘 + 架构建议 + 日志清理 + 假设记录）
   ├── 输出: fix-report（含 loop+假设+seam+post-mortem）
   ├── 重复制遍历指令: 「每轮遍历全部失败用例独立执行完整修复流程。更新 .status.md 中执行轮次计数器为当前轮次号。禁止按轮次分段分配工作重点——每轮均须对全部失败用例执行完整检查。必须完成全部 N 轮迭代后才可返回验收报告，禁止在目标轮次前返回。若当前轮次无新发现且无可执行工作（如所有问题已升级用户决策），仍须递增计数器并继续。3次修复上限约束单用例修复尝试次数（内层），N轮约束整体迭代轮次（外层），两者独立共存。」（N = 目标轮次，由 §14 确定）
   └── 完成承诺: COMPLETED

3. 启动 Agent 迭代子代理 (Agent(description, prompt, run_in_background))
   └── 子代理内维持现有多 Agent 并行分析 + 二分法根因分类

4. 主 Agent 验收
   ├── 轮次: .status.md 执行轮次 = N / N（N 为目标轮次，由弹性轮次决策确定）
   ├── 产物: fix-report 存在且完整
   ├── Feedback loop: Phase 1 构建的 loop 已记录
   ├── 假设记录: 3-5 个假设及验证结果已记录
   ├── 修复验证: 本地验证通过 + 原始场景验证通过
   ├── Seam 评估: Regression test seam 正确性已评估
   ├── Post-mortem: 复盘检查点已完成，日志已清理
   ├── 根因分类: 分类结论正确（实现错误/测试错误）
   └── 弹性模式验证门控（当 N < 10 时）: 受影响项 100% + 全量 ≥1 轮兜底 + 产物完整性
```

### 验收结果处理

- **通过** → 返回测试阶段重新执行（含 post-mortem 完成标记）
- **轮次不足** → 拒收，直接重新启动 Agent 子代理继续执行，不进入 AskUserQuestion
- **轮次达标但产物不合格** → 记录 skill-suggestion.md → AskUserQuestion 询问重跑

## 与其他 Skill 的关系

- **输入来自**：`kflow-api-test`（接口单元测试失败）、`kflow-e2e-test`（E2E 测试失败）、`kflow-bug-triage`（L4 实现问题路由）
- **输出给**：测试阶段（重新测试）
- **前置阶段**：测试（子变更级：接口单元测试或 E2E 测试），或 kflow-bug-triage L4 路由
- **后续阶段**：返回测试阶段（实现/测试错误）
- **关系说明**：仅处理子变更级，变更级集成测试修复由 `kflow-integration-test` 内聚处理。用户反馈统一由 `kflow-bug-triage` 处理（A路径），本 Skill 入口仅限测试阶段自动发现（B路径）
- **回写 bugs/ 目录**：A 路径修复完成后 SHALL 回写 `bugs/bug-NNN-NNN.md` 修复记录节并更新 `bugs/index.md` 状态；B 路径修复报告留在子变更目录，若对应 bug 已在 bugs/ 登记则同步追加摘要
- **执行模式**：重复制（内嵌多 Agent 并行分析），弹性轮次决策（首次执行10轮/回退按影响范围分数），3次修复上限独立共存，复杂度评估仅信息展示，主 Agent 验收闭环
