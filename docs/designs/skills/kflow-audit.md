## 门控检查

> **机制说明**：审计门控规则定义在 [core-mechanisms/04-gates-and-transitions.md](../core-mechanisms/04-gates-and-transitions.md#63-归档条件)

审计触发方式：
- **归档前自动触发**：集成测试通过后进入归档流程时，自动执行审计作为归档门控
- **手动调用**：用户直接调用 kflow-audit，对指定变更或全部活跃变更执行审计

归档门控规则：
- 阻塞级问题（缺少必须产物、阶段跳转、审查未闭环）→ 阻断归档
- 严重级问题（测试覆盖率不足、审查问题未全部关闭）→ 阻断归档
- 建议级问题（文档格式不规范、测试轮次偏多）→ 不阻断归档，记录改进建议

## 审计维度

### 七维度评估体系

| 维度 | 权重 | 检查项 | 评分标准 |
|------|------|--------|---------|
| 流程合规性 | 22% | 阶段顺序遵守、跳阶段检查、可选阶段处理、回退流程规范性 | 0-100 |
| 产物完整性 | 23% | 必须产物存在性、条件产物处理、产物内容质量（非空、无占位符） | 0-100 |
| 审查质量 | 20% | 四视角审查执行、代码审查执行、审查问题闭环（追踪矩阵全部关闭） | 0-100 |
| 测试充分性 | 15% | 接口单元测试通过率、E2E测试通过率、集成测试执行、多轮测试记录 | 0-100 |
| 缺陷管理 | 10% | 根因分类执行、修复闭环、设计错误回退触发正确性 | 0-100 |
| 效率指标 | 5% | 各阶段耗时、测试轮次（理想 ≤ 2 轮）、缺陷发现-修复周期 | 0-100 |
| Post-mortem 汇总 | 5% | 读取所有子变更 fix-report 的关联建议，汇总架构改进机会 | 0-100 |

**总评分 = Σ(维度得分 × 权重)**，满分 100 分。

### Post-mortem 汇总评估

审计时 SHALL 读取所有子变更的 fix-report 文件，汇总架构改进建议。

**汇总流程**：

```
1. 扫描所有子变更 fix-report 目录
   └── subchanges/*/test-reports/fix-reports/*.md

2. 提取每个 fix-report 的"关联建议"字段
   └── 架构改进建议、测试 seam 缺失报告

3. 按类型归类：
   ├── 测试 seam 缺失 → 测试架构改进机会
   ├── 模块耦合 → 模块化改进机会
   ├── 接口契约 → 契约设计改进机会
   └── 其他 → 通用改进建议

4. 汇总到审计报告的改进建议章节
```

**评分标准**：

| 评分 | 条件 |
|------|------|
| 100 | 所有 fix-report 已完成 post-mortem，架构建议已汇总 |
| 70 | 大部分 fix-report 已完成 post-mortem，少数未完成 |
| 40 | 部分 fix-report 缺失关联建议字段 |
| 0 | 无 fix-report 或均未完成 post-mortem |

**评估内容**：

| 检查项 | 说明 |
|--------|------|
| Post-mortem 完成率 | 所有子变更 fix-report 是否包含 post-mortem 章节 |
| 架构改进识别 | 是否有被打捞出的架构级改进机会 |
| 建议质量 | 建议是否具体可执行（非"改进代码质量"等空泛建议） |
| 建议传递 | 汇总后的架构改进机会是否可传递给下一变更的设计阶段 |

## 执行流程

```
审计阶段流程:

┌─────────────────────────────────────────────────────────────┐
│                      AUDIT WORKFLOW                          │
├─────────────────────────────────────────────────────────────┤
│  1. PRE_HOOK  → 引用 `skills/kflow-audit/references/hooks.md` audit 阶段 PRE_HOOK (轻量) │
│  │   ├── CHECK_STATE → 验证前置阶段状态                       │
│  │   ├── RELOAD → 重读全量产物, cross-reviews/, test-reports/, .status.md │
│  │   └── 不需要服务                                            │
│  2. SCOPE     → 确定审计范围                                  │
│  │   ├── 指定变更 → 审计单个变更                               │
│  │   └── 全部活跃变更 → 审计所有未归档变更                      │
│  3. COLLECT   → 收集审计材料                                   │
│  │   ├── .status.md (阶段状态、耗时)                          │
│  │   ├── functional-designs/ (功能设计)                      │
│  │   ├── detailed-design.md (详细设计)                        │
│  │   ├── cross-reviews/* (审查报告)                          │
│  │   └── test-reports/*/summary.md (测试报告)                 │
│  4. EVALUATE  → 七维度并行评估（含 Post-mortem 汇总）         │
│  │   ┌──────────────────────────────────────────────────┐    │
│  │   │ 并行 Agent 评估:                                   │    │
│  │   │ Agent 1: 流程合规性 + 产物完整性                    │    │
│  │   │ Agent 2: 审查质量                                  │    │
│  │   │ Agent 3: 测试充分性 + 缺陷管理                      │    │
│  │   │ Agent 4: 效率指标 + Post-mortem 汇总                │    │
│  │   └──────────────────────────────────────────────────┘    │
│  5. SCORE     → 计算七维度评分和总评分                         │
│  6. CLASSIFY  → 问题分级（阻塞/严重/建议）                     │
│  │   ├── 阻塞级 → 阻断归档，要求修复                           │
│  │   ├── 严重级 → 阻断归档，建议修复后可覆盖                    │
│  │   └── 建议级 → 不阻断，记录到 skill-suggestion.md           │
│  7. ROUTE     → 审计回退路由                                   │
│  │   ├── 问题归属设计阶段 → 设计阶段 ⚠️ 需修订                  │
│  │   ├── 问题归属编码阶段 → 编码阶段 ⚠️ 需修订                  │
│  │   └── 问题归属测试阶段 → 测试阶段 ⚠️ 需修订                  │
│  8. REPORT    → 输出审计报告                                   │
│  9. POST_HOOK → 引用 `skills/kflow-audit/references/hooks.md` audit 阶段 POST_HOOK (轻量) │
│  │   └── UPDATE_STATE → 更新 .status.md                       │
│ 10. COMPLETE  → 审计通过（无阻塞/严重问题）或触发回退           │
└─────────────────────────────────────────────────────────────┘
```

## 审计回退路由

审计发现问题是第 4 种回退触发来源（与缺陷修复、代码审查、用户需求变更并列）：

```
审计回退流程:

1. 确定问题归属阶段:
   ├── 设计探索/详细设计问题 → 对应设计阶段 ⚠️ 需修订
   ├── 编码问题 → 编码阶段 ⚠️ 需修订
   └── 测试问题 → 对应测试阶段 ⚠️ 需修订

2. 重置后续阶段:
   - 目标阶段之后的所有阶段状态 → ⏳ 待开始
   - 受影响产物标记为"待修订"

3. 联动评估:
   ├── 仅当前变更受影响 → 单独回退
   ├── 接口契约变更 → 依赖该接口的子变更联动回退
   ├── 架构级变更 → 所有子变更联动回退
   └── 审计发现 → 按审计问题归属阶段确定回退范围和联动子变更

4. 修复后重新审计:
   - 修复完成 → 目标阶段 ⚠️ 需修订 → ✅ 完成
   - 后续阶段逐个重新执行
   - 重新进入归档流程时再次触发审计
```

## 与其他 Skill 的关系

- **触发来源**：归档流程自动触发、用户手动调用
- **回退路由**：按问题归属阶段回退到 `kflow-explore`、`kflow-design`、`kflow-code`、`kflow-api-test`、`kflow-e2e-test`
- **门控关系**：归档门控（`kflow-archive` 的前置条件）
- **输出给**：`kflow-archive`（审计通过后）
