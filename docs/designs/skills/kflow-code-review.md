## 门控检查

> **机制说明**：门控规则定义在 [core-mechanisms/03-status-and-tasks.md](../core-mechanisms/03-status-and-tasks.md#34-门控规则)

进入代码审查阶段前检查：
- 子变更编码状态 = ✅ 完成
- 子变更 tasks.md 中所有实现任务已完成
- 所有功能点 TDD 循环完成（全 Green + 已提交）
- 不满足则 ❌ 阻塞，提示先完成编码

## 执行流程

```

┌─────────────────────────────────────────────────────────────┐
│                   CODE REVIEW WORKFLOW                       │
├─────────────────────────────────────────────────────────────┤
│  1. PRE_HOOK  → 引用 `skills/kflow-code-review/references/hooks.md` code-review 阶段 PRE_HOOK (轻量) │
│  │   ├── CHECK_STATE → 验证前置阶段状态                       │
│  │   └── RELOAD → 重读 service-guide.md, CONTEXT.md, detailed-design.md, .status.md │
│  │   └── 不需要服务（纯静态分析，不启动/停止服务）              │
│  2. CHECK     → 门控检查（编码完成状态）                     │
│  3. COLLECT   → 收集审查材料（代码变更、详细设计、tasks.md）  │
│  3.5 CROSS_TIER → 跨层越界检测                                 │
│  │   ├── 后端SC: Grep .tsx/.jsx/.vue 文件 + 硬编码颜色值       │
│  │   ├── 前端SC: Grep migrations/ + ORM注解 + 服务端路由       │
│  │   └── 排除 node_modules/.next/dist/.git 等目录             │
│  4. PARALLEL  → 两视角并行审查                               │
│  │   ┌─────────────────────┐  ┌─────────────────────┐       │
│  │   │ Agent 1: 安全+规范   │  │ Agent 2: 质量+性能   │       │
│  │   │ ├── SQL 注入         │  │ ├── N+1 查询         │       │
│  │   │ ├── XSS/CSRF        │  │ ├── 内存泄漏         │       │
│  │   │ ├── 敏感信息泄露     │  │ ├── 错误处理完整性    │       │
│  │   │ ├── 编码规范        │  │ ├── 代码复杂度       │       │
│  │   │ └── 依赖安全漏洞     │  │ └── 性能瓶颈         │       │
│  │   └─────────────────────┘  └─────────────────────┘       │
│  4.5 PROTO_CHECK → 原型对账验证（条件触发）                   │
│  │   ├── 设计令牌对账: Grep 源码硬编码颜色值 vs design-tokens │
│  │   ├── 元素覆盖对账: Grep 源码 vs element-spec.md 清单     │
│  │   ├── 路由覆盖对账: 提取路由配置 vs nav-tree.md 页面列表  │
│  │   └── 对账结果写入代码审查报告                             │
│  5. GATE     → 门控判定                                     │
│  │   ├── Agent 1 高=0 且 Agent 2 高=0 且 Agent 2 中<3       │
│  │   │   → ✅ 审查通过 → 输出报告 → 完成                     │
│  │   └── 任一不满足 → ❌ 阻塞                               │
│  │       ├── 输出完整问题清单                                │
│  │       ├── 标记子变更编码阶段为 ❌ 阻塞                     │
│  │       └── 修复完成后重新触发 kflow-code-review           │
│  6. REPORT   → 输出 code-review.md                          │
│  7. POST_HOOK → 引用 `skills/kflow-code-review/references/hooks.md` code-review 阶段 POST_HOOK (轻量) │
│  │   └── UPDATE_STATE → 更新 .status.md                      │
│  8. COMPLETE → 更新子变更状态（编码+审查完成）               │
└─────────────────────────────────────────────────────────────┘
```

## 审查视角详解

### Agent 1: 安全+规范

| 检查项 | 说明 | 严重度判定 |
|--------|------|-----------|
| SQL 注入 | 是否使用参数化查询，禁止字符串拼接 | 高 |
| XSS/CSRF | 输出编码、CSRF token 防护 | 高 |
| 敏感信息泄露 | 密钥/token/密码是否明文，日志是否脱敏 | 高 |
| 编码规范 | 命名规范、代码格式、注释规范 | 低 |
| 命名对齐 | 代码命名（模块/类/函数）是否与 CONTEXT.md 领域词汇表冲突 | 中 |
| 依赖安全漏洞 | 第三方依赖是否存在已知漏洞 | 中 |

### Agent 2: 质量+性能

| 检查项 | 说明 | 严重度判定 |
|--------|------|-----------|
| N+1 查询 | 循环内数据库查询、批量操作优化 | 高 |
| 内存泄漏 | 未关闭的资源、循环引用、事件监听器 | 高 |
| 错误处理完整性 | 异常捕获、错误返回、事务回滚 | 中 |
| 代码复杂度 | 圈复杂度 > 10、函数 > 50 行 | 中 |
| 性能瓶颈 | 同步阻塞、大对象加载、缺少索引 | 中 |

### 门控规则

| 条件 | 判定 |
|------|------|
| Agent 1 高严重度 = 0 且 Agent 2 高严重度 = 0 且 Agent 2 中严重度 < 3 | ✅ 审查通过 |
| Agent 1 高严重度 ≥ 1 或 Agent 2 高严重度 ≥ 1 或 Agent 2 中严重度 ≥ 3 | ❌ 阻塞 |

### 跨层越界检测（步骤 3.5，在原型对账之前执行）

> **版本**: 当前变更新增

两视角并行审查前，SHALL 对子变更源码目录执行跨层越界检测，验证代码产出不超越子变更类型边界。

#### 后端子变更越界检测

| 检测项 | Grep 模式 | 严重度 | 排除规则 |
|--------|----------|--------|---------|
| 前端源文件 | `*.tsx`、`*.jsx`、`*.vue`、`*.svelte` | 🟡 警告 | `node_modules/`、`.next/`、`dist/`、`build/` |
| 样式文件 | `*.css`、`*.scss`、`*.less`（排除 `globals.css`） | 🟡 警告 | 同上 |
| 硬编码颜色值 | `#[0-9a-fA-F]{3,6}` 或 `rgb(` | 🔵 建议（≥ 3 次标记） | `*.d.ts`、`*.test.*`、`*.spec.*` |
| prototype 路径引用 | `prototype/` | 🟡 警告 | `node_modules/`、`__tests__/`、`mocks/` |

#### 前端子变更越界检测

| 检测项 | Grep 模式 | 严重度 | 排除规则 |
|--------|----------|--------|---------|
| 数据库迁移脚本 | `migrations/` 或 `schema.sql` 或 `*.prisma` | 🟡 警告 | `node_modules/`、`.next/`、`dist/` |
| ORM 模型定义 | `@Entity`、`@Table`、`prisma.model`、`mongoose.Schema` | 🟡 警告 | `*.test.*`、`*.spec.*`、`node_modules/` |
| 服务端路由注册 | `app.use(`、`app.post(`、`router.get(`、`@PostMapping`、`@GetMapping` | 🟡 警告 | `*.test.*`、`*.spec.*`、`__tests__/` |

#### 越界检测输出

检测结果写入审查报告「跨层一致性」章节，位于原型对账章节之前：
- 🟡 警告：标记为审查建议，不阻塞审查通过
- 🔵 建议：标记为审查建议，不阻塞审查通过
- 原型对账 SHALL 仅在前端子变更时执行（不受越界检测结果影响）

### 原型对账验证（条件触发，仅当原型 spec 文件存在时）

| 对账维度 | 方法 | 判定 |
|---------|------|------|
| 设计令牌使用 | Grep 源码中硬编码颜色值（#xxx、rgb()、rgba()、hsl()），与 design-tokens.css 中的 CSS 变量值比对 | 源码中硬编码的值在 design-tokens.css 有对应变量 → ❌ "应使用 var(--xxx) 替代"；使用了 var(--xxx) → ✅ |
| 元素覆盖 | 读取 element-spec.md 中的按钮/表单/导航清单，Grep 源码中逐项搜索对应名称 | 缺失项 → ❌ "原型中定义了 {元素名}，源码中未找到"；多余项记录但不阻塞 |
| 路由覆盖 | 读取 nav-tree.md 中的页面列表，提取前端路由配置文件中的所有路径 | 未覆盖的页 → ❌ "原型中定义了页面 {page}，路由配置中未找到" |

对账结果输出：
- 设计令牌使用统计（使用 var(--xxx) 数 vs 硬编码数）
- 元素覆盖率（已覆盖数 / 总数 × 100%）
- 路由覆盖率（已覆盖页面数 / 总页面数 × 100%）
- 元素覆盖率 < 100% → 标记为审查缺陷
- 设计令牌硬编码数 > 0 → 标记为审查建议

## 闭环验证

修复完成后按严重度分级重审：

| 严重度 | 验证方式 | 说明 |
|--------|---------|------|
| 高 | 原视角 + 安全视角交叉检查 | 两个视角均确认修复后才关闭 |
| 中 | 原视角重新审查 | 确认修复不引入新问题 |
| 低 | 随机 30% Agent 验证 + 70% 开发者自查 | 抽样验证，减轻审查负担 |

## 重复制（执行类阶段）

> ⚠ **子代理强制规则**（参见 skills/kflow-code-review/references/repetition.md §12）：本阶段（代码审查）主工作 MUST 通过 Agent 子代理执行，主 Agent 仅负责调度和验收，SHALL NOT 直接执行审查主工作（安全审查/质量审查/问题追踪等），无例外。子代理崩溃时轮次级重试（≤3 次），全部失败标记 ⚠️ 阻塞。

代码审查阶段属于执行类阶段，采用弹性重复制模式。目标轮次由弹性轮次决策确定（首次执行 10 轮，回退重执行按影响范围分数缩减）。子代理每轮遍历全部待审查子变更，每个子变更内两视角独立执行全部检查项。

> 通用规范（复杂度公式、轮次执行细节、prompt 规范、弹性轮次决策、验证门控）参见 `skills/kflow-code-review/references/repetition.md`

### 阶段特定参数

- **遍历项**：全部编码完成但审查未通过的子变更
- **每轮工作**：两视角并行审查（安全+规范 / 质量+性能）+ 原型对账验证（条件触发）+ 分级重审闭环
- **复杂度权重**：修改文件数 × 1 + 新增代码行数(百行) × 1.5
- **产物要求**：code-review.md 问题追踪矩阵更新，Agent 1 高 = 0 且 Agent 2 高 = 0 且 Agent 2 中 < 3

## 与其他 Skill 的关系

- **输入来自**：`kflow-code`（编码阶段，子变更级）
- **输出给**：接口单元测试（`kflow-api-test`，子变更级）或 `kflow-bug-fix`（审查阻塞时按需）
- **前置阶段**：编码
- **后续阶段**：接口单元测试（`kflow-api-test`，子变更级）
- **关系说明**：从 `kflow-code` 拆分出的独立 Skill，原 `kflow-code` 不再包含代码审查子阶段
- **执行模式**：弹性重复制（内嵌两视角并行审查），目标轮次由弹性轮次决策确定（参见 skills/kflow-code-review/references/repetition.md §14），复杂度评估仅信息展示，主 Agent 验收闭环
