## 门控检查

> **机制说明**：门控规则定义在 [core-mechanisms/03-status-and-tasks.md](../core-mechanisms/03-status-and-tasks.md#34-门控规则)

进入编码阶段前检查：
- .status.md 存在
- 计划状态 = ✅ 完成
- 所有子变更 tasks.md 文件存在
- 当前子变更的依赖子变更已完成编码和接口单元测试（如有依赖）
- NFR 章节完整性检查（detailed-design.md 中 NFR 章节含性能+安全需求）
- 跨变更冲突检测（检查活跃变更文件重叠）
- docs/service-guide.md 存在（init 可预生成草稿）或自动生成服务指引
- 原型清单检查（条件）：如 prototype/index.md 存在且清单含 entry 角色文件，读取清单中声明的文件路径并注入为编码约束

## 数据库迁移管理

### 迁移脚本规范

当编码涉及数据模型变更时，在变更级 `migrations/` 目录创建迁移脚本：

| 文件 | 命名格式 | 说明 |
|------|---------|------|
| 迁移脚本 | `{序号}_{子变更}_{描述}.sql` | 正向迁移（DDL/DML） |
| 回滚脚本 | `{序号}_{子变更}_{描述}_rollback.sql` | 回滚操作 |

示例：
```
migrations/
├── 001_user-auth_create_users.sql
├── 001_user-auth_create_users_rollback.sql
├── 002_order-management_create_orders.sql
```

## 多 Agent 并行编码策略

### 策略概述

```
┌─ 前端子变更 + 无依赖后端子变更 并行编码 ──────────────────┐
│  Agent FE: 前端子变更编码（原型转译流程，mock 数据）       │
│  Agent 1: subchange-1 编码（无依赖，后端）                │
│  Agent 2: subchange-2 编码（无依赖，后端）                │
└──────────────────────────────────────────────────────────┘
                    ↓
┌─ 有依赖后端子变更顺序编码 ────────────────────────────────┐
│  等待依赖完成 → 开始编码                                  │
└──────────────────────────────────────────────────────────┘
                    ↓
┌─ 集成对接 ───────────────────────────────────────────────┐
│  后端编码完成 → 前端替换 mock 为真实 API                  │
└──────────────────────────────────────────────────────────┘
```

**前端子变更独立并行策略**：
- 前端子变更不依赖后端子变更编码完成，仅依赖 API 契约（design 阶段已定义）
- 前端子变更可与无依赖的后端子变更同时并行启动
- 使用 mock 数据进行开发，后端子变更编码完成后执行集成对接

### Agent 分配原则

1. 检测子变更依赖关系图
2. 所有子变更进入 code 阶段 MUST 是 AFK（HITL 已在设计阶段关闭并转为 AFK）
3. 按依赖关系分组：
   - AFK 无依赖子变更 → 可并行启动
   - AFK 有依赖子变更 → 等待依赖项完成后再启动

### 子变更执行策略

所有进入 code 阶段的子变更 MUST 是 AFK——设计决策已全部在设计阶段关闭，编码阶段不出现决策点。子变更按依赖关系图并行/顺序执行：

| 执行策略 | 条件 | 执行方式 |
|---------|------|---------|
| 可并行 | AFK + 无依赖 | 同时启动多个 Agent |
| 依赖完成后再启动 | AFK + 有依赖 | 等待依赖子变更完成后启动 |

### 冲突预防规则

| 规则 | 说明 | 执行方式 |
|------|------|---------|
| 独立域约束 | 每个 Agent 仅修改分配的子变更目录 | 明确约束范围 |
| 禁止跨域修改 | 禁止修改其他子变更代码 | Prompt 中明确 "Do NOT change other code" |
| 共享文件锁定 | 共享配置文件需等待其他 Agent 完成 | 顺序处理共享文件 |
| 跨变更冲突预防 | 编码前检查活跃变更文件重叠 | 读取 index.md 对比影响文件 |
| 结果合并验证 | Agent 完成后检查文件冲突 | 验证无交叉修改 |

### 变更级同步收敛

多 Agent 并行编码完成后，由上层 Agent 统一执行变更级收敛：

```
变更级同步收敛流程:

1. WAIT_ALL   → 等待所有子变更编码 Agent 完成
2. COLLECT    → 收集所有子变更的代码变更和迁移脚本
3. VERIFY     → 验证无交叉修改（共享文件冲突检查）
4. COMPILE    → 编译验证（统一编译项目，验证退出码，不启动服务）
5. MIGRATE    → 按序号合并并排序所有迁移脚本
6. REVIEW     → 触发 kflow-code-review（每个子变更独立审查）
7. SYNC       → 全部通过后释放门控 → 进入测试服务刷新同步点
```

| 收敛步骤 | 执行者 | 说明 |
|---------|--------|------|
| 等待完成 | 上层 Agent | 确认所有并行 Agent 完成编码和提交 |
| 冲突检查 | 上层 Agent | 对比各子变更修改文件列表，检测交叉修改 |
| 迁移合并 | 上层 Agent | 收集所有迁移脚本，按序号排序，冲突时提示用户裁定 |
| 编译验证 | 上层 Agent | 统一编译项目（仅验证可编译，不启动持久服务） |
| 代码审查 | `kflow-code-review` | 每个子变更独立触发代码审查 Skill |
| 门控释放 | 上层 Agent | 全部子变更编码+审查通过后释放 |

### 共享文件冲突预防

多 Agent 并行编码时存在共享文件（如 `.gitignore`、`package.json`、公共配置文件）的并发修改风险：

1. 编码启动前识别共享文件清单
2. Agent 仅修改分配的子变更目录，禁止修改共享文件
3. 需修改共享配置时，等待所有 Agent 完成后由上层 Agent 统一修改
4. 冲突检测时发现非法跨域修改 → 回退该 Agent 的变更，重新执行

### 前后端子变更文件隔离规则

前端子变更与后端子变更的代码域天然隔离，但共享配置文件（如前端路由注册入口、公共类型定义）存在交叉风险：

| 规则 | 说明 |
|------|------|
| 前端域 | 前端子变更操作：`src/pages/`、`src/components/`、`src/router/`、`src/store/`、`src/styles/`、`src/layouts/` |
| 后端域 | 后端子变更操作：`src/api/`、`src/services/`、`src/models/`、`src/db/`、`src/middleware/` |
| 共享文件 | `package.json`、`tsconfig.json` 等由变更级 agent 统一管理，子变更 Agent 禁止修改 |
| 跨域变更 | 如需同时修改前后端共享文件，由变更级 agent 在「集成对接」步骤统一处理 |
| 违规处理 | 发现跨域修改 → 回退该 Agent 的变更，记录到 skill-suggestion.md |

**前端 FP > 10 时的文件隔离**：

```
前端 FP > 10:
  ├── 骨架子变更（先行）: 脚手架/路由/布局/组件库/设计令牌
  │   └── 统一创建共享基础设施（路由入口、全局布局、公共组件、状态管理框架）
  └── 页面组子变更（串行追加）: 各页面组件文件（独立文件，无共享冲突）
      └── 在骨架已建立的框架内增量添加页面，各自操作独立的页面组件文件
```

### 子代理工具切换禁令

子代理在编码执行过程中遇到工具调用错误（如 API 返回 500、参数校验失败等）时，SHALL NOT 自行切换替代工具或绕过错误。正确做法：标记当前功能点为 ❌ 阻塞，上报主 Agent 等待决策。主 Agent 分析错误原因后决定重试、调整 prompt 或升级用户。

## FP 类型二次校验（编码前）

> **版本**: 当前变更新增

编码阶段在进入 TDD 任务循环前，SHALL 对子变更内各 FP 进行类型二次校验：

```
FOR 每个待实现 FP:
  1. 读取 functional-designs/index.md 中该 FP 的「类型」列
  2. 比较 FP 类型与子变更类型:
     ├── FP 类型 = 子变更类型 → ✅ 进入编码
     ├── FP 类型 ≠ 子变更类型 → ⏭️ 跳过该 FP
     └── 类型列缺失 → 🟡 警告（旧版兼容），不跳过
  3. 被跳过的 FP SHALL 记录到编码报告中
```

**跳过示例**：
- 前端子变更 tasks.md 中混入了后端 FP（如"实现注册 API"）→ 跳过该 FP，不编写后端代码
- 后端子变更 tasks.md 中混入了前端 FP（如"实现注册页面"）→ 跳过该 FP，不编写前端代码

**编码报告格式**（被跳过的 FP）：
```markdown
### 跳过的功能点
| FP-ID | 名称 | FP 类型 | 子变更类型 | 跳过原因 |
|-------|------|--------|-----------|---------|
| FP-005 | 实现注册 API | 后端 | 前端子变更 | FP 类型与子变更类型不匹配 |
```

## TDD 哲学

### 核心原则

TDD 的核心目的是通过测试验证系统行为，而非锁定实现细节。

| 原则 | 说明 |
|------|------|
| 测试行为而非实现 | 测试 SHALL 描述系统做什么（行为），而非怎么做（实现） |
| 仅用公共接口 | 测试 SHALL 仅通过公共接口验证行为，禁止测试私有方法 |
| 内部重构不应破坏测试 | 如果改实现不变行为导致测试失败 → 测试与实现耦合 |
| Mock 仅用于外部边界 | Mock SHALL 仅用于数据库、外部服务等外部边界，禁止 mock 内部 collaborator |

### 好测试 vs 坏测试

| 好测试 | 坏测试 |
|--------|--------|
| 验证可观察行为 | 验证内部方法调用 |
| 通过公共接口 | 调用私有方法 |
| 重构不破坏 | 重构即破坏 |
| 可读性高（描述业务行为） | 可读性低（描述实现步骤） |
| 一次一个行为 | 一次测试多个行为 |

### Vertical Slice（垂直切片）— 正确模式

TDD 循环 SHALL 采用垂直切片方式，每次实现一个完整的端到端行为切片（schema → logic → API → test）：

```
功能点1: RED(写1个测试) → GREEN(最小实现) → REFACTOR → ✓
功能点2: RED(写1个测试) → GREEN(最小实现) → REFACTOR → ✓
功能点3: RED(写1个测试) → GREEN(最小实现) → REFACTOR → ✓
```

### Tracer Bullet 先行

第一个 RED→GREEN 循环 SHALL 是一个 tracer bullet — 证明端到端路径可用的最简实现：

| Tracer Bullet 要求 | 说明 |
|--------------------|------|
| 横切所有集成层 | 从接口到数据层全路径打通 |
| 最简实现 | 不包含完整业务逻辑，仅证明路径可用 |
| 可运行 | 必须能编译通过并产生可验证的输出 |
| 后续增量 | 每个后续 RED→GREEN 基于前一切片增量添加 |

### 水平切片反模式 — 警告

> **⚠️ 警告**：禁止先写所有测试再写所有代码的"水平切片"反模式。

水平切片的危害：

```
❌ 错误: 水平切片（先写所有测试 → 再写所有代码）

Layer 1: 写所有 Controller 测试 → 写所有 Controller 代码
Layer 2: 写所有 Service 测试 → 写所有 Service 代码
Layer 3: 写所有 Repository 测试 → 写所有 Repository 代码

问题: 测试假想行为 — 测试是基于"我认为数据库应该怎么表现"而非"数据库实际怎么表现"编写的
      重构不可靠 — 重构实现时这些测试会大面积失败
```

```
✅ 正确: 垂直切片 / tracer bullet

Slice 1: Controller 测试 → Controller → Service → Repository → 全链路通过
Slice 2: Controller 测试 → Controller → Service → Repository → 全链路通过
Slice 3: ...

每次只写一个测试 → 一个实现，每个切片完整可运行
```

### TDD 每周期检查清单

每个 TDD 周期完成后 SHALL 自检：

#### RED 阶段（测试已写，待确认失败）

- [ ] 测试描述行为而非实现
- [ ] 测试使用公共接口
- [ ] 如果执行内部重构，此测试不应失败

#### GREEN 阶段（代码已实现，测试通过）

- [ ] 代码最小 — 仅满足当前测试，无多余逻辑
- [ ] 无投机性功能（"将来可能用到"的代码）
- [ ] 无提前优化（"这里太慢"在没有数据支持时不做优化）

#### REFACTOR 阶段（测试全通过，重构代码）

- [ ] 消除重复（DRY — 相同逻辑不出现两次以上）
- [ ] 模块是否可以深化（小接口 + 深实现 = 高 leverage，见 §深化模块）
- [ ] 每次重构后运行所有测试（确认不改行为）
- [ ] 禁止在 RED 状态重构

## 深化模块 (Deepen Module)

### 概念

来自 improve-codebase-architecture 的接口深度概念：

```
小接口 + 深实现 = 高 leverage

小接口: 仅暴露必要的最小公共 API
深实现: 封装复杂的内部实现细节
高 leverage: 一个模块解决大量问题，但调用方只需理解少量接口
```

### REFACTOR 步骤中的深化

在 REFACTOR 阶段检查：

| 检查项 | 行动 |
|--------|------|
| 模块接口是否精简 | 移除不必要的 public 方法 |
| 内部实现是否封装充分 | 将实现细节限制在模块内部 |
| 是否存在抽象泄漏 | 修复模块向调用方暴露内部状态的接口 |
| 能否增加模块深度 | 合并相关功能，减少跨模块调用链 |

### 深化示例

```
浅模块（反模式）:
  class UserService:
    - createUser()       # 直接操作
    - validateEmail()    # 暴露给外部调用
    - saveToDb()         # 暴露给外部调用
    - sendEmail()        # 暴露给外部调用

深模块（目标）:
  class UserService:
    - register(UserData)  # 唯一公共接口
      ├── validateEmail()   # 内部实现
      ├── saveToDb()        # 内部实现
      └── sendEmail()       # 内部实现
```

## 命名对齐约束

### 领域词汇表对齐

编码阶段 SHALL 将代码命名对齐项目级 `CONTEXT.md` 领域词汇表：

| 约束 | 说明 |
|------|------|
| 模块命名 | 模块/包名 SHALL 使用 CONTEXT.md 中的领域术语 |

| CONTEXT.md 术语定义不精确 | SHALL 记录建议到 skill-suggestion.md，待 explore 修订 |

## 重复制（执行类阶段）

> ⚠ **子代理强制规则**（参见 skills/kflow-code/references/repetition.md §12）：本阶段（编码）主工作 MUST 通过 Agent 子代理执行，主 Agent 仅负责调度和验收，SHALL NOT 直接执行编码主工作（TDD编码/编译验证/数据库迁移等），无例外。子代理崩溃时轮次级重试（≤3 次），全部失败标记 ⚠️ 阻塞。

编码阶段属于执行类阶段，采用弹性重复制模式。目标轮次由弹性轮次决策确定（首次执行 10 轮，回退重执行按影响范围分数缩减）。子代理每轮遍历 tasks.md 中全部功能点，对每个 FP 执行完整 TDD 垂直切片（含 Tracer Bullet）。

> 通用规范（复杂度公式、轮次执行细节、prompt 规范、弹性轮次决策、验证门控）参见 `skills/kflow-code/references/repetition.md`

### 阶段特定参数

- **遍历项**：外层遍历全部未完成编码的子变更，内层遍历每个子变更 tasks.md 中全部功能点
- **每轮工作**：每个 FP 执行完整 TDD 垂直切片（RED → GREEN → REFACTOR → 验证），首轮 SHALL 是 Tracer Bullet
- **复杂度权重**：功能点数 × 1 + 接口数 × 1.5 + 场景数 × 2
- **产物要求**：traceability.md「编码实现」列覆盖率 = 100%，涉及数据模型变更时迁移脚本+回滚脚本齐全
