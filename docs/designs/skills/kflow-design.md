## 执行流程

```
详细设计阶段流程:

┌─────────────────────────────────────────────────────────────┐
│                   DESIGN WORKFLOW                            │
├─────────────────────────────────────────────────────────────┤
│  1. PRE_HOOK  → 引用 skills/kflow-design/references/hooks.md design PRE_HOOK   │
│  2. CHECK     → 门控检查                                      │
│  2.5 TRACE    → 创建 traceability.md                           │
│  │   └── 基于 functional-designs/index.md FP 清单初始化空白矩阵│
│  3. ARCH      → 编写系统架构章节（合并原 architecture.md）     │
│  3.5 ADR      → 评估是否创建架构决策记录（条件步骤）             │
│  │   ├── 评估：是否满足三条件（难以逆转+缺上下文会奇怪+真正权衡）│
│  │   ├── 满足 → 创建 ADR 文件，含过期条件                        │
│  │   ├── 不满足 → 跳过，决策记录在 detailed-design.md 即可        │
│  │   └── 超出上限 → 提示变更粒度过大，建议拆分                    │
│  4. DOMAIN    → 按设计域组织章节（认证域/业务域/基础设施域等）  │
│  5. DESIGN    → 为所有功能点编写详细设计 + 复杂度评估（低/中/高）│
│  5.5 COMPLEX  → 高复杂度 FP 逐项确认（AskUserQuestion）          │
│  6. NFR       → 编写 NFR 章节（性能/安全/可用性/可维护性）      │
│  7. APITESTS  → 编写统一 api-tests/                           │
│  8. E2ETESTS  → (前后端项目) 编写统一 e2e-tests/               │
│  9. DIVIDE    → 基于完整设计认知划分子变更 + FP 类型校验      │
│  │   子步骤:                                                    │
│  │   9a. 读取 functional-designs/index.md 获取每 FP 的类型列     │
│  │   9b. 按设计域聚合，每组 FP 类型一致性校验:                   │
│  │       - 全部「后端」→ 自动标记为「后端子变更」                │
│  │       - 全部「前端」→ 自动标记为「前端子变更」                │
│  │       - 混合（同时含后端+前端）→ ❌ 拒绝划分，提示拆分        │
│  │   9c. 类型由系统推断，SHALL NOT 人工手动填写                  │
│  │   9d. 旧版兼容：functional-designs/index.md 缺少类型列        │
│  │       → 🟡 警告，提示重新执行 kflow-explore REVISION 补充    │
│  10. SELFREV   → 10 轮自循环审查（子代理串行 + 重复制）             │
│  │   │   每轮子代理独立执行全部四个维度（一致性+完备性+可行性+可测性）│
│  │   │   子代理发现问题 → 直接修复 + 生成自审报告                  │
│  │   报告路径: self-reviews/design/{YYYYMMDD}-{HHMMSS}.md        │
│  │   必须完成全部 10 轮，不可提前终止                             │
│  11. REVIEW    → 执行四视角交叉审查（并行 Agent，变更级一次）     │
│  12. SYNTH    → 输出审查综合报告（含问题追踪矩阵和关闭条件）     │
│  13. FIX      → 根据审查问题更新设计文档                        │
│  14. APPROVAL → 展示设计给用户最终评审（AskUserQuestion）       │
│  15. COMPLETE → 记录用户评审结果，更新状态文件                   │
│  16. POST_HOOK → 引用 skills/kflow-design/references/hooks.md design POST_HOOK │
└─────────────────────────────────────────────────────────────┘
```

## 复杂度评估与确认

> **版本**: 2.2.0 新增

### 概述

详细设计阶段在每个设计域完成后，对域内每个功能点进行复杂度评估（低/中/高），生成复杂度分布表。所有设计域完成后，对标记为「高」复杂度的 FP 逐项通过 AskUserQuestion 与用户确认设计方案，确认通过后方可继续 NFR 设计。

### 复杂度判定标准

| 复杂度 | 判定标准 |
|--------|---------|
| **低** | 标准 CRUD 操作、简单数据校验、无外部集成、无性能敏感逻辑 |
| **中** | 多表关联查询、中等复杂业务规则、单系统集成、少量性能考量 |
| **高** | 系统集成（多外部服务）、分布式事务、非平凡算法、性能敏感（高并发/大数据量）、安全敏感（认证/授权/加密）、业务规则模糊需用户澄清 |

### 高复杂度 FP 确认

所有标记为「高」复杂度的 FP 必须逐项通过 AskUserQuestion 与用户确认，不可批量确认或跳过。确认记录写入复杂度分布表下方。

## ADR 架构决策记录

### 概述

架构决策记录（Architecture Decision Record, ADR）用于记录重要的架构决策及其背景。ADR 在详细设计流程的 ARCH 步骤后按三条件评估创建。

### 三条件过滤

仅当以下三项条件**全部满足**时，才创建 ADR：

| 条件 | 说明 | 不满足时的做法 |
|------|------|--------------|
| 难以逆转 | 改变决策的成本有意义，改主意代价高 | 记录在 detailed-design.md 即可 |
| 缺上下文会奇怪 | 未来读者会问"为什么这么做？" | 记录在 detailed-design.md 即可 |
| 真正做了权衡 | 存在真实的替代方案，选了其中一个有具体理由 | 记录在 detailed-design.md 即可 |

**不满足三条件**：决策记录在 `detailed-design.md` 中即可，不创建独立 ADR 文件。

### 每变更上限

| 变更类型 | ADR 上限 | 说明 |
|---------|---------|------|
| 功能需求级变更 | ≤ 2 个 | 常规功能变更 |
| 产品级变更 | ≤ 5 个 | 大型产品变更 |
| 功能缺陷级变更 | 0 个 | 通常不创建 ADR |

超出上限时 SHALL 提示用户变更粒度过大，建议拆分为多个变更。

### ADR 序号管理

ADR 文件按项目级递增序号管理：

1. 读取 `docs/adr/` 目录现有 ADR 文件
2. 取现有最大序号 +1 作为新序号
3. 文件名格式：`{4位递增序号}-{kebab-case标题}.md`（如 `0003-choose-redis-cluster.md`）

### ADR 文件格式

每个 ADR SHALL 遵循以下格式：

```markdown
# ADR-{序号}: {标题}

**日期**: YYYY-MM-DD
**状态**: 提议 | 接受 | 废弃 | 取代
**过期条件**: {何时应重新评估此决策}

## 背景

{当前场景描述和组织，以及需要决策的技术问题}

## 决策

{选定的方案和具体理由}

## 后果

{决策带来的正面、负面和中立影响}

## 替代方案

{考虑过但未采用的其他方案及原因}
```

### 过期条件

每个 ADR SHALL 标注何时重新审视此决策：

| 过期条件类型 | 示例 |
|-------------|------|
| 数据量阈值 | 数据量 < 10GB 持续 6 个月 → 重新评估 |

## E2E 测试用例设计（含元素覆盖树）

> **版本**: 3.0.0 新增

E2E 测试用例设计（执行流程步骤 8）SHALL 包含元素覆盖树相关的四个前置子步骤，确保原型中所有交互元素均有对应测试场景。设计完成后进入子变更划分。

### 8.0 CHECK — 确认元素覆盖树时效性

在进入 E2E 测试用例设计前，SHALL 确认元素覆盖树的时效性：

```
1. 检测 element-coverage-tree.md 是否存在:
   ├── 有 prototype → 检测 prototype/element-coverage-tree.md
   └── 无 prototype → 检测 e2e-tests/element-coverage-tree.md（本次变更首次进入时不存在）
2. 若文件存在:
   a. 读取树的生成时间戳
   b. 有 prototype → 比对 prototype/index.html 修改时间
   c. 有 prototype 且源头已更新 → AskUserQuestion:
      "元素覆盖树可能已过期。是否重新生成？"
      选项: [重新生成] [增量更新] [保持不变]
3. 若文件不存在:
   a. 有 prototype → 进入 8.1 路径 A（静态 HTML 解析生成）
   b. 无 prototype → 进入 8.1 路径 B（playwright-cli 逐页探索生成）
4. 进入 8.1 EXPLORE（如需新生成）或 8.2 DESIGN（直接使用现有树）
```

### 8.1 EXPLORE — 探索生成元素覆盖树

按 prototype 是否存在分两条生成路径。无论哪条路径，树落点规则为：

| 条件 | 落点 |
|------|------|
| prototype/index.html 存在 | `prototype/element-coverage-tree.md`（树在原型目录下） |
| 无 prototype + 前端项目 | `e2e-tests/element-coverage-tree.md`（树在 E2E 目录下） |
| 纯后端项目 | 无（不生成树） |

#### 路径 A — 有原型（静态 HTML 解析）

当 prototype/index.html 存在时，通过解析 DOM 树 + CSS/JS 扫描生成初始树：

```
1. 解析 prototype/*.html 的 DOM 树
2. 提取所有交互元素: <button>, <a>, <input>, <select>, <textarea>, <dialog>, [role="dialog"], <details>
3. 提取 CSS 伪类状态: :hover, :active, :focus, :disabled（扫描 stylesheet 和内联 style）
   - 扫描 class 命名推测状态: is-loading, is-error, is-empty, is-disabled, is-active
4. 提取 JS 事件绑定: onclick, addEventListener, @click（Vue）, onClick（React）
   - 分析事件处理中的 DOM 操作推断操作链（弹窗显示/隐藏、跳转）
5. 构建初始树（含页面导航 + 元素 + 状态 + 操作链，TC-ID 列为空）
6. 输出 prototype/element-coverage-tree.md
```

> 树结构定义（四层 + 操作链 + 符号约定 + 节点元数据格式）详见 [element-coverage-tree.md 产物格式规范](#element-coverage-tree-产物格式规范)。

#### 路径 B — 无原型（playwright-cli 动态探索）

当 prototype/index.html 不存在且为前后端项目时，通过 playwright-cli 逐页探索实际前端页面：

```
1. 读取路由配置获取全部页面路径列表
2. 逐页 playwright-cli 探索:
   a. open {url} → waitForLoadState → snapshot 获取静态可交互元素列表
   b. 逐个按钮 click → snapshot 观察变化（弹窗/下拉/跳转）
   c. 逐个 input focus + fill → 观察校验状态和建议列表
   d. 逐个 hover 触发元素 → 观察 tooltip/popover
   e. 表单: 空提交（观察校验错误元素）→ 合法数据提交（观察成功反馈 + 跳转）
   f. 网络监听: page.on('response') → 提取 API 调用路径
3. 汇总构建 e2e-tests/element-coverage-tree.md
4. 路径 B 下树文件含已填充的 TC-ID（探索与用例设计在同一 design 阶段连续完成）
```

> **注意**: 路径 B 的 playwright-cli 探索在 design 阶段的子代理中执行，不阻塞用户交互。

### 8.2 DESIGN — TC-ID 映射

元素覆盖树生成后，为每个状态节点（🎯）分配 E2E 测试场景 ID：

```
1. 遍历元素覆盖树中所有 🎯 状态节点
2. 为每个状态节点编写对应的 E2E 测试场景（写入 e2e-tests/part-NN.md）
3. 将场景 ID（TC-XXX）回填到树的状态节点上，格式: 🎯 <状态名> → TC-001
4. 每个元素节点的元数据块中填充 tc_ids 字段
```

**节点元数据格式**（每个元素/状态节点附带）：

```yaml
element:
  ref: "原型 ref 或 playwright snapshot ref"  # 元素定位引用
  type: "button|input|select|link|dialog|toast|..."
  tc_ids: ["TC-001", "TC-003"]                # design 阶段填充
  data_source: "后端API|前端静态|配置控制|浏览器存储"  # 可选，e2e-data-tracing 扩展
  tested: null                                  # e2e-test 阶段标记: null/✅/❌
```

**TC-ID 映射要求**：
- 每个 🎯 状态节点 MUST 有至少一个 TC-ID
- 一个 TC-ID 可覆盖多个状态节点（如同一测试场景覆盖多个元素的默认态）
- TC-ID 映射写入树文件（YAML 元数据块中的 `tc_ids` 列表）
- 同时 e2e-tests/part-NN.md 中每个 TC 的基本信息 SHALL 包含「覆盖元素」字段，列出对应的元素树路径

### 8.3 VERIFY — TC-ID 覆盖率 100% 门控检查

E2E 测试用例编写和 TC-ID 映射完成后，SHALL 执行覆盖率门控检查：

```
1. 遍历元素覆盖树中所有 🎯 状态节点
2. 统计:
   ├── 总状态节点数: {N}
   ├── 有 TC-ID 的状态节点数: {M}
   └── 覆盖率 = M / N × 100%
3. 覆盖率 = 100% → 门控通过 ✅
4. 覆盖率 < 100% → 门控阻塞:
   a. 输出未覆盖元素清单（完整树路径: 📄 页面 → 🏗️ 区域 → 🔘 元素 → 🎯 状态）
   b. 格式:
      ### 未覆盖元素清单
      | 页面 | 区域 | 元素 | 状态 | 建议 |
      |------|------|------|------|------|
      | /dashboard | 用户信息区 | [编辑按钮] | click → 编辑弹窗 | 补充编辑弹窗内表单测试场景 |
   c. agent SHALL 回到 8.2 为未覆盖元素补充测试用例
   d. 补充完成后重新执行 8.3 覆盖率检查
5. 覆盖率 100% 后方可释放门控，进入步骤 9（DIVIDE 子变更划分）
```

> **覆盖率语义**: 100% 是基线，不是目标。树中每个元素+状态都必须有对应的 E2E 测试场景，不允许存在"有意不测"的元素。

### element-coverage-tree 产物格式规范

#### 四层树结构 + 符号约定

元素覆盖树采用「页面 → 区域 → 元素 → 状态」四层结构，状态节点下挂载操作驱动的动态链路：

```
📄 <页面路径>
├── 🏗️ <区域名称>
│   ├── 🔘 <元素描述> [<元素类型>]
│   │   ├── 🎯 <状态名> → <TC-ID>
│   │   │   ├── 💬 <产生的浮窗名>
│   │   │   │   ├── 🔘 <浮窗内元素> → <TC-ID>
│   │   │   │   └── ...
│   │   │   └── 🔗 跳转 → 📄 <目标页面>
│   │   └── 🎯 <另一状态> → <TC-ID>
│   └── 📝 <表单名>
│       ├── <字段名> [input/select/...]
│       │   ├── 🎯 focus → <TC-ID>
│       │   ├── 🎯 空值校验 → <TC-ID>
│       │   └── 🎯 disabled → <TC-ID>
│       └── <提交 button>
│           ├── 🎯 hover → <TC-ID>
│           ├── 🎯 loading → <TC-ID>
│           └── 🎯 disabled → <TC-ID>
└── 💬 <全局浮窗/Toast>
    ├── 🎯 成功 → <TC-ID>
    └── 🎯 失败 → <TC-ID>
```

**符号约定**:

| 符号 | 含义 | 示例 |
|------|------|------|
| 📄 | 页面 | `📄 /dashboard` |
| 🏗️ | 区域（section） | `🏗️ 搜索区` |
| 🔘 | 按钮 | `🔘 [登录]` |
| 📝 | 输入/表单 | `📝 [用户名 input]` |
| 💬 | 弹窗/浮窗/抽屉/Toast | `💬 [确认对话框]` |
| 📊 | 数据展示区 | `📊 [用户列表表格]` |
| 🔗 | 页面跳转 | `🔗 跳转 → 📄 /detail/{id}` |
| 🎯 | 交互状态 | `🎯 hover 态`、`🎯 loading 态` |

## 子变更划分规则

### 划分时机

子变更划分在详细设计完成、审查通过后执行：

1. 所有功能点的数据模型、接口、流程已完整设计
2. 功能点间依赖关系已明确
3. 跨子变更接口契约已定义
4. 基于完整设计认知进行划分，避免"先划分后设计"的矛盾

### 划分前 FP 类型一致性校验

> **版本**: 当前变更新增

在划分子变更前，SHALL 读取 functional-designs/index.md 中每 FP 的「类型」列进行一致性校验：

```
对于划分候选组中的每一组功能点:
  1. 读取 functional-designs/index.md 获取各 FP 的类型（后端/前端）
  2. 比较: 全部一致 → ✅ 通过
  3. 存在混合（后端+前端）→ ❌ 拒绝此组划分，提示拆分
  4. 类型列缺失（旧版文档兼容）→ 🟡 警告，提示重新执行 kflow-explore REVISION 补充类型
  5. 一致性通过后，系统自动推断子变更类型（后端/前端），SHALL NOT 人工手动填写
```

**类型自动推断规则**：

| FP 类型组成 | 子变更类型 | 操作 |
|-----------|----------|------|
| 全部「后端」 | 后端子变更 | ✅ 通过 |
| 全部「前端」 | 前端子变更 | ✅ 通过 |
| 混合（后端 + 前端） | — | ❌ 拒绝，提示拆分为独立的后端子变更和前端子变更 |

### 划分原则

1. 按设计域聚合：同一设计域的功能点尽量在同一子变更
2. 按依赖最小化：无依赖的功能点可并行划入不同子变更
3. 每子变更功能点 ≤ 10
4. 每个变更 ≤ 20 个子变更
5. 每子变更内 FP 类型 SHALL 一致：后端 FP 和前端 FP SHALL NOT 混入同一子变更

### 子变更实现顺序

1. 基础子变更（无依赖或依赖最少）
2. 核心业务子变更
3. 扩展功能子变更
4. 辅助功能子变更

### HITL/AFK 分类

每个子变更在划分时 SHALL 标注类型：AFK（可自动执行）或 HITL（设计不完整标记）。

| 类型 | 含义 | 约束 |
|------|------|------|
| **AFK** (Away From Keyboard) | 设计完整，可全自动执行 | 可并行启动 Agent |
| **HITL** (Human In The Loop) | 设计不完整标记——存在未决决策 | **阻塞进入 plan 阶段**，MUST 在设计阶段解决后转为 AFK |

### AFK 判定标准

子变更满足以下**所有**条件时标记为 AFK：

| 条件 | 说明 |
|------|------|
| 无架构选择决策点 | 所有技术选型在设计文档中已明确 |
| 无设计方案确认需求 | 接口契约、数据模型已完整定义 |
| 所有实现路径已明确 | 详细设计文档覆盖了所有功能点的实现路径 |

> **注意**: 原「UI/UX 方向决策」条件已移除——原型设计阶段已有用户评审门控，原型确认后不存在未决 UI 决策。

### HITL 判定标准

子变更涉及以下**任一**类型时标记为 HITL：

| 决策点类型 | 示例 | 关闭方式 |
|-----------|------|---------|
| 架构选择未裁决 | "Redis vs Memcached 待定" | 创建 ADR 记录决策 |
| 设计方案确认未完成 | "接口契约需用户批准" | 用户评审确认 |

> **注意**: 原「UI/UX 方向」决策点类型已移除——UI/UX 决策 SHALL 在原型设计阶段通过用户评审关闭。

### HITL 关闭流程

1. 设计阶段完成未决决策（ADR 创建、用户评审确认、接口契约批准）
2. 将子变更从 HITL 转为 AFK
3. 更新 detailed-design.md 子变更划分表中的类型
4. plan 阶段入口门控检测到 AFK → 放行

### 执行策略影响

| 类型 | 并行策略 | plan 入口门控 |
|------|---------|-------------|
| AFK | 可并行（无依赖时） | 放行 |
| HITL | 不进入 plan（阻塞） | ❌ 阻塞 |

## 用户评审

### 交互时机

用户评审在 AI 四视角审查完成、所有高/中严重度问题已修复后触发。此时设计文档已经过 AI 审查和修订，用户看到的是最终版本。

### 交互方式

通过 `AskUserQuestion` 向用户展示评审请求：

- **问题**: "详细设计已完成并通过 AI 审查。关键决策摘要：{架构选型}、{子变更划分}、{NFR 要点}。审查中发现并修复 {n} 个高严重度问题、{m} 个中严重度问题。是否确认通过？"
- **选项 1**: 「确认通过」— 设计满足需求，进入计划阶段
- **选项 2**: 「需要修订」— 设计需修改，提出修订意见后重新审查和评审

### 评审前展示内容

向用户展示的摘要信息包括：

1. 系统架构概述（技术选型、模块划分）
2. 子变更划分结果（各子变更功能点数和依赖关系）
3. NFR 要点（性能目标、安全要求）
4. AI 审查综合报告关键发现（高/中严重度问题数及修复状态）

### 评审结果处理

| 用户选择 | 阶段状态 | 评审记录 | 后续动作 |
|---------|---------|---------|---------|
| 确认通过 | ✅ 完成 | ✅ 已确认 | 门控释放，可进入计划阶段 |
| 需要修订 | ⚠️ 需修订 | ⚠️ 需修订 | 阶段回退：重置详细设计为需修订，后续阶段重置为待开始 |

### 评审记录写入

评审结果写入变更级 `.status.md` 的「用户评审记录」表格：

```markdown
## 用户评审记录

| 阶段 | 评审状态 | 评审时间 | 备注 |
|------|---------|---------|------|
| 原型设计 | ⏳ 待评审 | - | - |
| 详细设计 | ✅ 已确认 | 2026-05-05 16:00 | 设计确认通过，子变更划分合理 |
```

## 阶段边界约束

> **版本**: 1.9.0 新增

### 域内内容（design 负责）

| 内容类别 | 说明 |
|---------|------|
| 系统架构 | 技术选型、模块划分、部署架构 |
| 数据模型 | 实体定义、表结构、索引、约束、关系 |
| 接口设计 | API 路径、方法、入参、出参、错误码 |
| NFR | 性能/安全/可用性/可维护性需求定义 |
| 子变更划分 | 基于完整设计认知划分子变更 |
| 测试用例 | api-tests/ 和 e2e-tests/ 测试用例文档 |
| ADR | 满足三条件的架构决策记录 |

### 域外内容（禁止）

| 内容类别 | 说明 | 应由哪个阶段处理 |
|---------|------|----------------|
| 功能定义修改 | 新增/删除/修改 functional-designs/ 中的功能点 | explore（回退） |
| UI 布局修改 | 修改 prototype/ 中的 UI 设计 | prototype（回退） |
| 业务规则修改 | 修改功能设计中的业务规则 | explore（回退） |
| 代码实现 | 任何代码文件 | code |

### 越界处理

当 design 阶段发现上游产物存在缺陷时：
1. 记录到 `docs/skill-suggestion.md`
2. 提示用户是否需要阶段回退
3. 禁止直接修改 functional-designs/ 或 prototype/ 内容

## 10 轮自审

> **版本**: 1.9.0 新增

### 概述

详细设计阶段在 DIVIDE 步骤之后、REVIEW 步骤之前，强制执行 10 轮自循环审查。自审由子代理（Agent subagent）串行执行，采用重复制——每轮子代理独立执行全部四个维度（一致性/完备性/可行性/可测性）。

> **子代理隔离规则**：自审子代理意外停止/报错/要求重做时，主代理 MUST 重新创建子代理，SHALL NOT 接管自审执行。详见 [core-mechanisms/07-agent-model.md §15.11](../core-mechanisms/07-agent-model.md#1511-子代理隔离规则)。

### 审查维度与检查规则

详见 `.claude/skills/kflow-design/references/self-review.md`——四维度检查规则（一致性/完备性/可行性/可测性）、子代理串行执行流程、边审边修说明、串行约束、强制执行规则。

### 自审流程

```
10 轮自审执行流程（子代理串行，每轮全维度）:

1. 主 Agent 启动第一轮子代理:
   Agent(
     subagent_type="claude",
     description="Design 自审 Round {N}",
     prompt="读取 detailed-design.md、traceability.md、api-tests/、e2e-tests/ 下的所有文件，按一致性、完备性、可行性、可测性全部四个维度独立检查。发现问题直接修复设计文档，生成审查报告到 self-reviews/design/{YYYYMMDD}-{HHMMSS}.md。仅修复确认的问题，不做重构或额外改进。"
   )
2. 子代理返回审查报告路径
3. 主 Agent 读取报告，确认修复内容
4. 修复不合理 → 主 Agent 补充修复
5. 启动下一轮子代理（Round N+1），步骤同 Round 1
6. SHALL NOT 并行启动多个子代理（串行执行）
7. 重复直至完成全部 10 轮
8. 全部完成后进入四视角交叉审查
```

### 自审记录存储

- 根目录：`self-reviews/design/`
- 文件名：`{YYYYMMDD}-{HHMMSS}.md`（审查开始时间）
- 每轮一个独立文件，自然排序即时间序

### 强制执行规则

- SHALL 完成全部 10 轮自审，不允许提前终止
- SHALL 每轮启动独立子代理（Agent subagent），不允许主 Agent 自身执行自审
- SHALL NOT 并行启动多个子代理（串行执行）
- 自审全部完成后进入四视角交叉审查
- 四视角审查结果保存到 `cross-reviews/{timestamp}/` 目录

## REVISION 模式

> **版本**: 当前变更新增

### 触发条件

当 kflow-guide 的 DESIGN_REVISION 路由检测到用户意图为详细设计修订（含"详细"/"接口"/"架构"关键词）时，唤醒 kflow-design 进入 REVISION 模式。

### REVISION 模式流程

```
REVISION 模式流程:

1. 加载 detailed-design.md（或 detailed-design/ 目录）现有内容
2. 加载 .status.md 获取当前阶段状态
3. AskUserQuestion 确认修订需求:
   - "详细设计修订模式。请描述需要调整的内容（接口设计/数据模型/架构/NFR/配置项等）。"
4. 根据用户反馈修订 detailed-design.md 中的对应章节
5. 更新 detailed-design.md 修订记录节:
   ├── 版本号递增（semver: Major 结构性变更 / Minor 功能性变更 / Patch 修正性变更）
   └── 追加修订记录行: 版本 | 日期 | 修订类型 | 修订内容 | 影响功能点 | 触发阶段
6. 如涉及接口变更 → 同步更新 api-tests/ 中的测试用例
7. 如涉及子变更划分变更 → 更新子变更划分章节
8. 返回 kflow-guide 进入设计修订后处理流程
```

### REVISION 模式约束

- REVISION 模式下不重新执行完整的 design 流程（跳过 ARCH/ADR/DOMAIN/NFR/DIVIDE/APITESTS/E2ETESTS/SELFREV/REVIEW）
- 仅修订用户指定的设计域，不重写整个 detailed-design.md
- 修订完成后 SHALL 更新修订记录节
- 如修订导致接口契约变更，SHALL 同步受影响的其他设计域章节
