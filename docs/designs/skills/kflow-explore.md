## 执行流程

```
设计探索阶段流程:

┌─────────────────────────────────────────────────────────────┐
│                    EXPLORE WORKFLOW                          │
├─────────────────────────────────────────────────────────────┤
│  1. INIT      → 创建变更目录和状态文件                       │
│  2. CONTEXT   → 探索项目上下文（文件、文档、提交记录）         │
│  3. TYPEDET   → 检测项目类型（前后端 vs 纯后端）              │
│  │   ├── 检测 package.json 前端框架依赖                      │
│  │   ├── 检测 .vue/.tsx/.jsx 等前端源文件                   │
│  │   ├── 检测 vite.config/webpack.config 构建配置           │
│  │   └── 判断结果 → 前后端项目 或 纯后端项目                 │
│  4. CONFIRM   → 用户确认项目类型（AskUserQuestion）          │
│  5. CLASSIFY  → 判断变更粒度（产品级/功能级/缺陷级）          │
│  6. CLARIFY   → 澄清需求，收集用户意图                        │
│  │   ├── 检测 CONTEXT.md 是否存在                              │
│  │   ├── 不存在 → 首次构建 CONTEXT.md                           │
│  │   │   └── 从用户需求描述中提炼初始领域术语                   │
│  │   │       └── 每个术语含: 定义、别名、边界                   │
│  │   ├── 已存在 → 增补检查                                     │
│  │   │   └── 检查是否有新术语需要添加                          │
│  │   │   └── 用户使用模糊/过载术语 → 提议精确术语并写入         │
│  │   │   └── 增补完成后更新修订记录                            │
│  │   └── 术语提炼完成后继续需求澄清                            │
│  7. SPLIT     → 拆分功能点到原子级 + 标记类型（后端/前端）    │
│  │   子步骤:                                                    │
│  │   7a. 原子级拆分 — 按单一明确行为标准拆分                     │
│  │   7b. 类型判定 — 对每个 FP 按后端/前端标准判定               │
│  │   7c. 混合检测 — 无法归类(跨层)的 FP 强制继续拆分            │
│  │   7d. 关联标注 — 拆分产生的跨层 FP 对相互引用                │
│  8. RELATE    → 分析功能点关联关系                            │
│  9. OUTPUT    → 输出功能设计文档（含版本号+修订记录区+功能结构树）│
│  10. SELFREV  → 10 轮自循环审查（重复制:每轮全4维度）         │
│  │   每轮: 完整性+闭环性+必要性+清晰性 全部四个维度独立检查     │
│  │   发现问题修复 → 保存自审报告                               │
│  │   报告路径: self-reviews/explore/{YYYYMMDD}-{HHMMSS}.md     │
│  │   必须完成全部 10 轮，不可提前终止                           │
│  11. PROTO_GATE → 原型设计决策门控（仅前后端项目）           │
│  │   ├── 检查 .status.md 中是否存在 prototype_decision 标记   │
│  │   ├── 不存在 → AskUserQuestion: "检测到 {n} 个 UI 功能点，│
│  │   │   是否进入原型设计阶段？"                              │
│  │   │   ├── 确认创建原型 → prototype_decision=已选择原型设计 │
│  │   │   └── 跳过原型设计 → prototype_decision=已跳过         │
│  │   └── 已存在 → 跳过询问，直接进入下一阶段                  │
│  12. COMPLETE → 更新状态文件，标记阶段完成                    │
└─────────────────────────────────────────────────────────────┘
```

**注意**：设计探索阶段不再划分子变更。子变更划分后置到详细设计阶段完成后执行。

## 两动作规则

探索阶段 SHALL 遵循「两动作规则」行为准则，防止信息收集操作的结果在上下文轮转后不可恢复。

### 行为准则

每执行 **2 次**信息收集操作后，SHALL 将收集到的关键发现写入当前阶段的产出文件（`functional-designs/`）。

### 适用范围（计数触发）

以下操作计入两动作计数：

| 操作类型 | 说明 |
|---------|------|
| WebFetch | 获取外部网页内容 |
| WebSearch | 搜索外部信息 |
| Read（外部资源） | 读取项目外的文档、配置文件等 |
| Read（图片） | 查看图片/截图内容 |
| Read（PDF） | 阅读 PDF 文件内容 |

### 不适用范围（不计数）

以下操作不计入两动作计数：

| 操作类型 | 说明 | 原因 |
|---------|------|------|
| Read（项目内源代码） | 读取项目内的源代码文件 | 代码内容可通过文件系统随时重新获取 |
| Glob | 搜索项目文件 | 文件系统操作，非信息收集 |
| Grep | 搜索项目内容 | 文件系统操作，非信息收集 |
| Bash | 查看项目状态 | 本地操作，非外部信息收集 |

### 保存内容规范

写入产出文件的内容 SHALL 包含：

| 字段 | 说明 |
|------|------|
| 信息来源 | 操作类型 + URL/路径（便于追溯） |
| 关键要点 | 从信息收集结果中提炼的核心发现 |
| 影响分析 | 该发现对功能设计的影响或建议 |

### 执行时机

| 场景 | 行为 |
|------|------|
| 正常探索 | 每 2 次信息收集操作后立即保存 |
| 连续信息收集 | 每批相关信息收集完成后保存（不严格按计数） |
| 阶段结束前 | 强制检查并保存所有未写入的发现 |
| 会话中断前 | 立即保存所有中间发现 |

> **原则**："2 次"为启发式指导值，核心原则是"频繁保存中间发现"，防止非文本操作结果在上下文轮转后丢失。

## 项目类型检测规则

### 检测标识

| 标识类型 | 具体内容 | 存在时判断 |
|---------|---------|-----------|
| package.json 前端框架依赖 | react, vue, angular, next, nuxt, svelte 等 | 前后端项目 |
| 前端源文件 | .vue, .tsx, .jsx, .svelte, .angular 等 | 前后端项目 |
| 前端构建配置 | vite.config.*, webpack.config.*, tsconfig.json 等 | 前后端项目 |
| 以上均不存在 | - | 纯后端项目 |

### 用户确认流程

检测完成后，使用 AskUserQuestion 展示检测结果：
- 显示检测到的项目类型（前后端项目 / 纯后端项目）
- 用户可确认或手动修正项目类型
- 修正结果记录到 functional-designs/index.md

### 项目类型影响

| 项目类型 | 后续阶段影响 |
|---------|-------------|
| 前后端项目 | 原型设计阶段可选，E2E测试阶段必须，集成测试必须 |
| 纯后端项目 | 跳过原型设计阶段，跳过E2E测试阶段，集成测试必须 |

## 功能点拆分规则

### 原子级功能点定义

| 粒度 | 判断标准 | 示例 |
|------|---------|------|
| 过粗 | 包含多个独立行为 | "用户管理功能" ❌ |
| 合适 | 单一明确行为 | "用户注册"、"用户登录" ✅ |
| 过细 | 实现细节级别 | "添加注册按钮" ❌ |
| 无法归类（跨层） | 包含跨层行为，同时涉及后端和前端判定标准 | "实现注册功能（含 API + 页面）" ❌ — MUST 拆分为独立后端 FP 和前端 FP |

### FP 类型判定规则

> **版本**: 当前变更新增

每个功能点在拆分 SHALL 标记为「后端」或「前端」，类型作为后续子变更划分类型校验、任务模板选择、输入源检查的依据。

#### 后端 FP 判定标准

功能点涉及以下任一内容时，SHALL 标记为「后端」：

| 判定标准 | 说明 |
|---------|------|
| API 实现 | REST 接口 / GraphQL / gRPC |
| 服务逻辑 | 业务规则处理 / 工作流编排 |
| 数据模型定义 | ORM 实体 / 数据库表结构 |
| 数据库操作 | CRUD / 查询 / 事务 |
| 业务规则引擎 | 规则校验 / 计算逻辑 |
| 外部服务集成 | 第三方 API 调用 / 消息队列 |

#### 前端 FP 判定标准

功能点涉及以下任一内容时，SHALL 标记为「前端」：

| 判定标准 | 说明 |
|---------|------|
| 页面组件 | 页面布局 / 模板结构 |
| UI 组件 | 按钮 / 表单 / 表格 / 弹窗等可复用组件 |
| 路由配置 | 页面路径 / 导航守卫 |
| 状态管理 | store / context / reactive state |
| 样式实现 | CSS / SCSS / CSS-in-JS / Tailwind |
| CSS 变量注入 | design-tokens 应用 |

#### 无法归类则强制拆分

| 场景 | 处理方式 |
|------|---------|
| 一个 FP 同时涉及后端和前端判定标准 | 判定为粒度过粗，MUST 继续拆分 |
| 拆分结果 | 一个后端 FP + 一个前端 FP |
| 拆分后关联标注 | 两个新 FP SHALL 在「关联功能点」列相互引用，前端 FP SHALL 描述中标注「关联后端 FP: {FP-ID}」 |
| 禁止做法 | SHALL NOT 以混合形式写入 functional-designs/ |

**示例**：
- 「实现注册 API」→ 后端
- 「实现注册表单页面」→ 前端
- 「实现用户注册功能」→ ❌ 过粗，MUST 拆分为上述两个独立 FP
- 拆分后 FP-001(后端)「注册 API」+ FP-002(前端)「注册页面」→ 在「关联功能点」列相互引用 #FP-002 / #FP-001

### 功能点数量与子变更划分

> **重要变更**：子变更划分不再在设计探索阶段执行，而是在详细设计阶段完成后基于完整设计认知划分。

| 条件 | 处理方式 |
|------|---------|
| 功能点 ≤ 10 | 详细设计阶段确认后，创建单一子变更 |
| 功能点 > 10 且 ≤ 20 | 详细设计阶段完成后拆分为多个子变更 |
| 功能点总数 > 20 | 提示拆分为多个变更 |

### 功能点关联关系类型

| 关系类型 | 定义 | 标注格式 |
|----------|------|---------|
| 前置依赖 | A 必须在 B 前完成 | `A -> B` |
| 后置依赖 | B 完成后才可执行 A | `B <- A` |
| 并行关系 | A 和 B 可同时实现 | `A \|\| B` |
| 包含关系 | A 是 B 的子功能 | `B ⊃ A` |

## 领域词汇表管理 (CONTEXT.md)

### 概述

`CONTEXT.md` 是项目级领域词汇表，作为所有后续阶段术语引用的唯一真实来源。kflow-explore 在 CLARIFY 步骤中负责其首次构建和持续增补。

### 文件格式

```markdown
# CONTEXT — {项目名} 领域词汇表

> **版本**: 1.0.0
> **创建时间**: {YYYY-MM-DD}
> **修订记录**:
> | 版本 | 日期 | 修订内容 | 触发阶段 |
> |------|------|---------|---------|
> | 1.0.0 | {YYYY-MM-DD} | 初始构建 | 设计探索 |

---

## {领域术语1}

- **定义**: 该概念在项目中的精确定义
- **别名**: （可选，代码中出现的变体名称或简称）
- **边界**: 明确什么不是这个概念（显式排除）

## {领域术语2}
...
```

### 术语提炼规则

| 规则 | 说明 |
|------|------|
| 定义精确性 | 定义 SHALL 使用领域专家能理解的描述，不耦合实现细节 |
| 别名必查 | 如果代码中已有变体名，SHALL 记录为别名以便对齐 |
| 边界明确 | 每个术语 SHALL 明确排除相近但不同的概念 |
| 过载术语处理 | 用户使用模糊/过载术语时，SHALL 提议精确术语并写入 |
| 禁止实现耦合 | 禁止将术语与代码实现细节绑定（如类名、表名） |

### 首次构建流程

```
1. 检查 CONTEXT.md 是否存在
2. 不存在时:
   ├── 从用户需求描述中提取关键领域概念
   ├── 为每个概念编写: 定义、别名、边界
   ├── 按字母顺序排列术语
   └── 写入 CONTEXT.md（含版本号和修订记录）
3. 完成构建后继续需求澄清
```

### 增补流程

```
1. CONTEXT.md 已存在时
2. 在需求澄清过程中:
   ├── 检查新出现的术语是否已在词汇表中
   ├── 新术语 → 按首次构建规则添加到 CONTEXT.md
   ├── 模糊术语 → 与用户确认精确含义后写入
   └── 更新版本号（次版本 +1）和修订记录
```

### 消费链

CONTEXT.md 构建后贯穿整个变更流程：

```
explore 构建 → design 引用审查 → plan 用于任务描述 → code 用于命名 → review 用于审查报告
```

### 对齐检查

增补完成后 SHALL 执行对齐检查：
- 检查功能点描述中使用的术语是否与 CONTEXT.md 定义一致
- 功能设计文档中的实体名称 SHALL 使用 CONTEXT.md 中的术语
- 发现不一致时：更新功能设计文档或更新 CONTEXT.md 定义

## 渐进式流程调整

根据变更粒度自动调整流程深度：

| 粒度 | 流程深度 | 产出要求 |
|------|---------|---------|
| **产品级** | 完整流程 | 需求描述 + 项目类型 + 功能点清单 + 技术方案建议 |
| **功能需求级** | 标准流程 | 需求描述 + 项目类型 + 功能点清单 + 关联关系 |
| **功能缺陷级** | 简化流程 | 缺陷描述 + 项目类型 + 影响分析 + 修复范围 |

## 阶段边界约束

> **版本**: 1.8.0 新增

### 域内内容（explore 负责）

| 内容类别 | 说明 |
|---------|------|
| 功能结构树 | 以树状图展示模块→功能点层级，节点标注 FP-ID + 优先级 + 简述 |
| 可执行操作 | 用户在每个页面可执行的具体操作 |
| 表单项定义 | 字段名、类型、校验规则、默认值 |
| 业务规则 | 前置条件、校验规则、触发条件、后置结果 |
| 业务流程 | 用户视角的完整业务流程闭环 |
| 功能点拆分 | 原子级功能点、依赖关系、优先级 |

### 域外内容（禁止）

| 内容类别 | 说明 | 应由哪个阶段处理 |
|---------|------|----------------|
| 技术架构选型 | 框架、组件库、中间件选择 | design |
| 数据模型设计 | 实体定义、表结构、索引设计 | design |
| 接口定义 | API 路径、方法、入参出参 | design |
| 代码实现 | 任何代码文件 | code |
| 子变更划分 | 执行单元划分 | design |
| UI 原型 | 视觉设计和交互原型 | prototype |

### 越界处理

当 explore 阶段发现信息不足以完成功能设计时：
1. 在当前阶段文档中标记"待补充"并说明缺少什么
2. 记录到 `docs/skill-suggestion.md`
3. 禁止自创新文档来填补信息空缺
4. 禁止越界输出技术方案或接口定义

## 10 轮自审

> **版本**: 1.8.0 新增

### 概述

设计探索阶段在 OUTPUT 步骤之后、COMPLETE 步骤之前，强制执行 10 轮自循环审查。自审由子代理（Agent subagent）串行执行，采用重复制——每轮子代理独立执行全部四个维度（完整性/闭环性/必要性/清晰性）。

### 审查维度与检查规则

#### 完整性

| 检查项 | 规则 |
|--------|------|
| 页面覆盖 | 是否所有用户可见页面均已覆盖 |
| 菜单覆盖 | 是否所有菜单路径均有对应功能点 |
| 操作覆盖 | 是否所有用户可执行操作均有对应功能点 |
| 表单项覆盖 | 是否所有涉及表单输入的功能点均有完整表单项定义 |

#### 闭环性

| 检查项 | 规则 |
|--------|------|
| 流程完整性 | 每个业务流程是否从触发点到最终结果形成完整路径 |
| 断点检测 | 检查是否存在"无处可去"的中间状态 |
| 异常路径 | 业务流程中的分支条件和异常路径是否已标注 |

#### 必要性

| 检查项 | 规则 |
|--------|------|
| 多余功能点 | 检查是否存在无明确用户价值的功能点 |
| 冗余功能点 | 检查是否存在功能重叠的功能点 |
| 价值验证 | 每个功能点是否有明确的用户价值描述 |

#### 清晰性

| 检查项 | 规则 |
|--------|------|
| 描述无歧义 | 每个功能点描述是否无歧义 |
| 边界明确 | 每个功能点的输入/输出边界是否清晰定义 |
| 术语一致 | 功能点描述中使用的术语是否与 CONTEXT.md 定义一致 |

### 自审模式：重复制

> **v2.1.0 变更**: 从"分工制"（每轮只审查部分维度）改为"重复制"——每轮独立执行全部四个维度（完整性+闭环性+必要性+清晰性），10 轮形成自然收敛。

| 对比维度 | 分工制（旧） | 重复制（新） |
|---------|------------|------------|
| 每轮范围 | 部分维度 | 全部维度 |
| 发现问题 | 后期才暴露其他维度问题 | 早期就暴露各类问题 |
| 收敛性 | 不明显 | 自然收敛（后期问题越来越少） |
| 独立性 | 维度间串行 | 每轮独立完整审查 |

### 自审流程

```
10 轮自审执行流程（重复制）:

1. 记录开始时间（作为时间戳）
2. 按完整性、闭环性、必要性、清晰性四个维度逐项检查
3. 发现问题 → 立即修复
4. 生成自审报告 → 保存到 self-reviews/explore/{YYYYMMDD}-{HHMMSS}.md
5. 进入下一轮
6. 每轮均执行全部四个维度的完整检查
7. 重复直至完成全部 10 轮
8. 全部完成后标记阶段完成
```

### 自审记录存储

- 根目录：`self-reviews/explore/`
- 文件名：`{YYYYMMDD}-{HHMMSS}.md`（审查开始时间）
- 每轮一个独立文件，自然排序即时间序
- 报告模板参考：`docs/designs/templates/changes/{change}/self-reviews/review-round.md`

### 自审报告内容

每轮报告包含：
- 审查维度得分表（维度名、本轮得分、上轮得分、变化值）
- 新发现问题清单（序号、问题描述、严重度、状态）
- 上轮问题修复验证（序号、上轮问题、修复结果）
- 本轮改进内容描述
- 仍存在问题（进入下一轮继续跟踪）

### 强制执行规则

- SHALL 完成全部 10 轮自审，不允许提前终止
- 即使连续多轮无新问题也必须完成全部 10 轮
- 自审全部完成后标记阶段完成，释放 prototype 或 design 阶段门控

## REVISION 模式

> **版本**: 当前变更新增

### 触发条件

当 kflow-guide 的 DESIGN_REVISION 路由检测到用户意图为功能设计修订（含"功能"/"需求"关键词）时，唤醒 kflow-explore 进入 REVISION 模式。

### REVISION 模式流程

```
REVISION 模式流程:

1. 加载 functional-designs/index.md 和 functional-designs/part-NN.md 现有内容
2. 加载 .status.md 获取当前阶段状态
3. AskUserQuestion 确认修订需求:
   - "功能设计修订模式。请描述需要调整的内容。"
4. 根据用户反馈修订 functional-designs/ 中的对应功能点
5. 更新 functional-designs/index.md 修订记录:
   ├── 版本号递增（semver）
   └── 追加修订记录行: 版本 | 日期 | 修订类型 | 修订内容 | 影响功能点 | 触发阶段
6. 返回 kflow-guide 进入设计修订后处理流程
```

### REVISION 模式约束

- REVISION 模式下不重新执行完整的 explore 流程（跳过 INIT/CONTEXT/TYPEDET/CLASSIFY）
- 仅修订用户指定的功能点，不重写整个 functional-designs/
- 修订完成后 SHALL 更新 index.md 的修订记录表
- 如修订涉及功能点新增/删除，同步更新功能结构树和分册
