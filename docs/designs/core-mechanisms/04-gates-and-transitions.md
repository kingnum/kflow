# KFlow Skills 核心运行机制

> **版本**: 参见仓库根目录 `VERSION` 文件
> **创建时间**: 2026-04-29
> **更新时间**: 2026-05-17

> **加载层级**: 基础层
> **适用阶段**: 全部

本文档定义 KFlow Skills 体系的核心运行机制，包括目录结构、状态文件、任务清单、阶段流转规则、回退机制和条件产物引用规范。

## 五、条件产物引用规范

### 5.1 图例定义

所有 Skill 文档的输入/输出产物使用标准化图例：

| 图例 | 含义 | 使用场景 |
|------|------|---------|
| ✅ 必须 | 文件必须存在，门控强制检查 | 所有项目类型都需要 |
| 🔶 条件 | 取决于前置阶段是否执行或项目类型 | 前后端项目需要，纯后端不需要；或可选阶段产物 |
| ⏭️ 不适用 | 该项目类型不需要 | 纯后端跳过E2E相关产物 |

### 5.2 使用规则

- 每个产物按项目类型（前后端/纯后端）分别标注图例
- 门控检查中显式处理三种情况：跳过（通过）、完成有文件（通过）、待开始（阻塞）
- 不因条件产物不存在而误报阻塞

### 5.3 子变更类型严格二分

子变更类型 SHALL 限定为「后端子变更」或「前端子变更」，禁止创建前后端混合子变更。

| 类型 | 判定标准 | 输入源 | 任务模板 |
|------|---------|--------|---------|
| **后端子变更** | 全部功能点均为后端 FP（API 实现/服务逻辑/数据模型/数据库操作/业务规则） | functional-designs/、detailed-design.md、api-tests/、CONTEXT.md | TDD 模板（Red → Green → Refactor） |
| **前端子变更** | 全部功能点均为前端 FP（页面/组件/路由/状态管理/样式/CSS 变量注入） | prototype/* 核心产物 + detailed-design.md（API 契约章节）+ functional-designs/ | 原型转译模板（5 步：骨架→令牌→状态→API→验证） |

**混合子变更拒绝规则**：
- 子变更同时包含后端 FP 和前端 FP → 拒绝此划分，提示拆分
- 共享类型定义（如公共 interface/type）→ 由变更级 agent 在「集成对接」步骤统一管理，不属于子变更域

**前端 FP > 10 拆分规则**：
- 前端功能点数 > 10 → 拆分为「骨架子变更」（先行，搭建脚手架/路由/布局/组件库/设计令牌）+「页面组子变更」（串行追加，各自操作独立页面组件文件）
- 详见 `frontend-implementation-subchange` spec。

#### 5.3.1 共享关切归属规则

当一个概念同时涉及后端和前端时，按以下规则归属到对应的子变更类型（不创建混合子变更）：

| 共享关切 | 归属子变更类型 | 说明 |
|---------|-------------|------|
| 错误码定义（code/枚举） | 后端子变更 | 错误码在后端定义，前端仅消费 |
| 错误码提示文案 | 前端子变更 | 用户可见的提示文字由前端控制 |
| 配置项（开关/阈值/参数） | 后端子变更 | 配置项在服务端管理，前端通过 API 获取 |
| 共享 DTO / interface | 变更级 `shared-types/` | 前后端共同消费的类型定义由变更级统一管理，非子变更域 |
| Schema 变更（表结构/字段） | 后端子变更 | 数据库 Schema 变更由后端负责 |
| Schema → UI 影响链 | 关联标注 | 后端 Schema 变更 → 标注「关联前端 FP: {FP-ID}」给前端子变更作为参考 |

#### 5.3.2 共享类型目录规则

前后端子变更共同消费的 DTO/interface 定义 SHALL 存放在 `changes/{change}/shared-types/` 目录：

- **创建条件**：存在前后端子变更共同消费的 DTO/interface
- **产出内容**：同时产出后端语言和前端语言的类型定义文件
- **输入引用**：后端子变更 plan 阶段将 `shared-types/` 作为 🔶 条件输入
- **前端 mock**：前端子变更基于 `shared-types/` 中的类型定义编写 mock 数据
- **跳过条件**：变更不存在跨子变更共享的类型定义时，不创建此目录

                                                                                          ▼
                                                                                    划分子变更
                                                                                          │
                                                                                          ▼

                                                                                          ▼

### 6.2 缺陷修复循环

#### 子变更级缺陷修复循环（二分法）

```
子变更测试阶段发现失败 ───────────────────────────────────────────────────▶
    │
    ▼
根因分类 ──────────────────────────────────────────────────────────────▶
    │
    ├── 实现错误 (80%) → 修复代码 → 本地验证 → 返回测试阶段
    │
    ├── 测试错误 (15%) → 修正测试用例/测试代码 → 验证 → 返回测试阶段
    │
    └── 设计错误 (5%) → 触发阶段回退
           │
           ├── 标记当前阶段 ❌ 阻塞
           ├── 标记设计阶段 ⚠️ 需修订
           ├── 重置后续阶段 ⏳ 待开始
           └── 执行设计修订 → 重新审查 → 重新计划 → 重新编码 → 重新测试
```

#### 子变更级修复尝试上限

为防止无限自动循环，子变更级缺陷修复 SHALL 遵循以下尝试上限规则：

```
同一测试用例修复尝试上限:

尝试 1: 直接修复（按二分法根因分类采用针对性方案）
    │
    ├── 成功 → 返回测试阶段
    │
    └── 失败 → 尝试 2
                  │
                  ├── 采用替代方案（不同工具/方法，不得重复尝试 1 方案）
                  │
                  ├── 成功 → 返回测试阶段
                  │
                  └── 失败 → 尝试 3
                                │
                                ├── 重新审视问题假设，考虑更广泛的替代方案
                                │   （不得重复前两次方案）
                                │
                                ├── 成功 → 返回测试阶段
                                │
                                └── 失败 → 停止自动修复循环
                                              │
                                              └── 使用 AskUserQuestion 升级用户
                                                  提供三个选项:
                                                  1. 保留当前状态（人工介入）
                                                  2. 跳过该功能点（标记为已知问题）
                                                  3. 触发设计阶段回退
```

**尝试计数规则**：

| 规则 | 说明 |
|------|------|
| 不同测试用例独立计数 | 每个测试用例的尝试次数独立计算，一个用例达到上限不影响其他用例 |
| 根因变更重置计数 | 修复过程中根因分类发生变更时（如从实现错误变更为设计错误），重置尝试计数 |
| 跨轮次累加 | 同一测试用例在不同轮次中的尝试次数累加计算 |
| 上限兜底 | 无论根因类型，同一用例累计 3 次失败后 SHALL 升级用户 |

#### 3次修复上限与10轮迭代的层级关系

3次修复上限与10轮迭代为独立共存的两个维度，约束不同层级：

```
层级关系:
  外层循环（轮次迭代，下限）: 执行轮次 N/10（N ∈ [1, 10]）
      │
      ├── 第 N 轮内 ──────────────────────────────
      │   ├── 内层循环（修复尝试，上限）:          │
      │   │   每测试用例修复尝试次数 ≤ 3             │
      │   │   ├── 失败 1 次 → 采用替代方案重试       │
      │   │   ├── 失败 2 次 → 重新审视假设重试       │
      │   │   └── 失败 3 次 → 升级用户决策           │
      │   │                                         │
      │   └── 3次上限触发后:                         │
      │       ├── 该用例标记为「用户决策中」         │
      │       └── 继续处理当前轮次其他可修复问题      │
      │                                             │
      ├── 第 N 轮完成 → 递增计数器 → 进入第 N+1 轮   │
      │                                             │
      └── 第 10 轮完成 → 计数器 = 10/10 → 返回验收报告 │
```

| 维度 | 3次修复上限 | 10轮迭代 |
|------|-----------|---------|
| 约束对象 | 单测试用例修复尝试次数 | Agent 迭代子代理整体迭代轮次 |
| 方向 | 上限（ceiling）——阻止无限重试 | 下限（floor）——确保充分迭代 |
| 计数范围 | 每用例独立计数，跨轮次累加 | 每阶段统一计数，跨用例不孤立 |
| 触发后果 | 达到 3 次 → 升级用户决策 | 未达 10/10 → 拒收，继续执行 |
| 互动关系 | 上限触发不终止轮次（仍须递增计数器至 10/10） | 下限保障整体迭代深度，内嵌上限防护 |

**场景示例**：
- 第 5 轮某用例 3 次修复均失败 → 升级用户决策（保留/跳过/回退），Agent 继续完成第 5 轮其他工作 → 递增至 6/10 → 继续第 6-10 轮
- 极端情况所有用例第 1 轮全部触发 3 次上限 → 升级用户，Agent 子代理后续第 2-10 轮递增计数器并标注「所有问题已升级用户决策，无可修复项」，仍须递增至 10/10 后方可返回

#### 变更级集成测试缺陷修复循环（四分法）

```
集成测试阶段发现失败 ───────────────────────────────────────────────────▶
    │
    ▼
根因分类（四分法）──────────────────────────────────────────────────────▶
    │
    ├── 接口实现错误 (60%) → 修复对应子变更代码 → 本地验证 → 返回集成测试
    │
    ├── 接口契约错误 (20%) → 更新接口契约文档 → 联动修复调用方 → 返回集成测试
    │
    ├── 集成测试用例错误 (15%) → 修正集成测试用例 → 验证 → 返回集成测试
    │
    └── 架构设计错误 (5%) → 触发阶段回退
           │
           ├── 标记当前阶段 ❌ 阻塞
           ├── 标记设计阶段 ⚠️ 需修订
           ├── 重置后续阶段 ⏳ 待开始
           └── 执行设计修订 → 重新审查 → 重新计划 → 重新编码 → 重新测试

多轮循环: 集成测试失败 → kflow-integration-test (内聚修复) → 修复后重新集成测试 → 循环直到通过
修复报告路径: test-reports/integration/fix-reports/fix-{timestamp}.md
架构评估触发: 连续3轮同一用例失败 → 自动触发架构评估
```

### 6.3 归档条件

> 完整规范参见 `kflow-archive 的 references/archive-rules.md` §1-§2

### 6.5 阶段回退触发来源与目标

> 完整规范参见 `各 skill 的 references/gates.md` §3

系统支持六种回退触发来源（缺陷修复设计错误、代码审查结构问题、需求变更、审计发现、编码发现原型问题、编码发现功能点问题），回退目标包含 explore、design、prototype-design 三种阶段。

### 6.6 编码阶段发现上游问题的标准化决策流程

当编码阶段发现上游产物存在问题时，SHALL 执行标准化决策流程而非自动回退：

```
编码阶段发现上游问题:
    │
    ├── 步骤 1: 判定根因归属
    │   ├── 原型交互/视觉/状态问题 → 回退目标: prototype-design（REVISION 模式）
    │   ├── 功能点/业务规则问题 → 回退目标: explore（功能性）或 design（技术性）
    │   └── 纯实现层面可修复 → 不触发回退（继续编码）
    │
    ├── 步骤 2: 记录到 skill-suggestion.md
    │   └── 记录根因判定结果和影响范围
    │
    ├── 步骤 3: AskUserQuestion 确认决策
    │   ├── "确认回退到 [目标阶段]" → 执行回退流程
    │   ├── "暂缓此功能点（标记 ⏸️）" → 标记该 FP 为 ⏸️ 暂缓，继续其他 FP
    │   └── "记录为已知问题继续" → 记录不阻止继续
    │
    └── 步骤 4: 回退执行（如用户确认）
        ├── 原型回退 → 进入 prototype-design REVISION 模式（加载已有原型 + 用户修订需求 → 修订 → 验证 → 用户确认 → 回退到 design 继续）
        ├── 功能设计回退 → 回退到 explore 修订功能点设计
        └── 技术设计回退 → 回退到 design 修订技术方案
```

### 6.7 Archive 禁止自动流转

> 完整规范参见 `kflow-archive 的 references/archive-rules.md` §4

归档阶段是 KFlow 体系中唯一禁止自动流转的阶段，审计通过后 MUST 通过 AskUserQuestion 获取用户显式确认后方可进入。

### 6.8 HITL 重新定义为设计阶段标记

HITL（Human In The Loop）不再作为执行类型，改为「设计不完整标记」。设计阶段划分子变更时，若存在未决决策，标记为 HITL 并阻塞进入 plan 阶段。所有进入 plan 的子变更 MUST 是 AFK。

#### HITL 新语义

| 维度 | 旧语义（v0.X） | 新语义（v0.7+） |
|------|--------------|----------------|
| 定义 | 执行类型——含人工决策点，编码时触发 | 设计阶段标记——设计不完整，阻塞执行 |
| 阻塞位置 | 编码阶段决策点暂停 | **plan 阶段入口** |
| 触发阶段 | code | design / prototype-design |
| 关闭方式 | 用户确认决策后继续 | 设计阶段完成未决决策后转为 AFK |
| 编码阶段行为 | AskUserQuestion 暂停 | 不出现（所有决策在设计阶段已关闭） |

#### HITL 阻塞 plan 入口规则

```
HITL 阻塞 plan 入口:

  子变更标记为 HITL:
    │
    └── plan 入口门控 SHALL 检测到 HITL 标记 → ❌ 阻塞
         └── 提示: "子变更 {name} 存在未决设计决策，请先在 design 或 prototype-design 阶段完成决策"
  
  解决方案:
    1. 在设计阶段完成未决决策（创建 ADR、用户评审确认、接口契约批准）
    2. 将子变更标记从 HITL 转为 AFK
    3. 更新 detailed-design.md 子变更划分表中的执行类型
    4. plan 入口门控释放
```

#### AFK 新判定标准（移除 UI/UX 方向决策条件）

| 条件 | 说明 |
|------|------|
| 无架构选择决策点 | 所有技术选型在设计文档中已明确 |
| 无设计方案确认需求 | 接口契约、数据模型已完整定义 |
| 所有实现路径已明确 | 详细设计文档覆盖了所有功能点的实现路径 |

原「UI/UX 方向决策」条件移除——原型设计阶段已有用户评审门控，原型确认后不存在未决 UI 决策。

#### 常见 HITL 场景与关闭方式

| 场景 | 未决决策 | 关闭方式 | 关闭阶段 |
|------|---------|---------|---------|
| 技术选型未定 | Redis vs Memcached | 创建 ADR 记录决策 | design |
| 接口契约未确认 | API 入参/出参待批准 | 用户评审确认接口设计 | design |
| 原型交互未确认 | 布局方案 A vs B | 用户评审确认原型 | prototype-design |
| 数据模型方案有分歧 | 范式化 vs 反范式化 | ADR + 用户确认 | design |
